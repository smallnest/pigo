Implementation Notes

Issue #155 — 解析并校验 npm:<name> 包引用 — 2026-07-20

Design Decisions

Decision 保留显式 Registry 字段而非硬编码 npm

Ambiguity: 本期只支持 npm，是否需要抽象出 registry 概念不明确。

Choice: PackageRef 带 Registry 字段（当前恒为 RegistryNPM），解析时严格校验前缀。

Rationale: 未来若加 github:/file: 源，调用方无需改动结构，符合开闭原则；成本几乎为零。

Decision scoped 名的版本分隔用「最后一个 @」且跳过 scope 的 @

Ambiguity: @scope/name@1.2.3 里有两个 @，如何区分 scope 标记与版本分隔？

Choice: 从下标 1 起找最后一个 @ 作为版本分隔，从而 @scope/name 不误拆、@scope/name@1.2.3 正确拆为 (@scope/name, 1.2.3)。

Rationale: 覆盖 scoped/plain × 有无版本四种组合，测试已固化。

Deviations

Deviation 未完整复刻 npm 官方 validate-npm-package-name

Spec: 验收要求「非法 npm 包名被拒绝」。

Implemented: 实现了一个实用子集：非空、≤214 字符、小写、无空白/控制字符、无 shell 敌意字符、scoped 形状校验，而非逐条复刻官方库。

Why: 目的是「安全地把名字交给 npm CLI」而非做 npm 的注册校验。拒绝的是不安全/明显错误的类别，足够满足 install 场景且避免过度工程。

Tradeoffs

Tradeoff 字符黑名单 vs 白名单

Alternatives: (A) 只允许 [a-z0-9-._/@] 白名单；(B) 拒绝一组 shell 敌意/非法字符的黑名单。

Pros/Cons: 白名单最严格最安全，但可能误伤 npm 实际允许的少见合法字符；黑名单明确挡住注入风险字符，兼容性更好。

Winner: 黑名单——安全目标（防命令注入）已达成，同时不误伤合法包名；配合小写与长度校验形成多层防护。
