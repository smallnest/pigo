Implementation Notes

Issue #39 — [impl] headless/stdio 运行模式（US-020）— 2026-07-10

Design Decisions

Decision 退出码经「返回 error」而非在库内 os.Exit

Ambiguity: 验收要求"退出码正确反映成功/失败"，但没说这个映射在哪一层做。

Choice: RunHeadless 返回 error（成功 nil，失败 *ErrRunFailed），由 cmd/pigo/main.go 负责 os.Exit(1)；库层从不 os.Exit。

Rationale: 库函数直接退出进程不可测、不可组合。返回 error 让 e2e 测试能在同进程内断言成功/失败语义，退出码映射只在 main 这一薄壳里发生。

Decision stream-json 事件封装为白名单字段的 envelope

Ambiguity: "事件以行分隔 JSON 输出"没规定每行的 schema。

Choice: eventEnvelope 对每种 AgentEvent 显式挑选安全字段（type 判别式 + text/toolName/isError/stopReason/messageCount），逐行 json.Marshal + \n。

Rationale: 白名单而非整体序列化，既保证消费者能按 type 分派，又杜绝把 API key/内部结构等敏感或易变字段泄漏到线上（US-012 不记录敏感值）。

Decision Provider 从 --model 前缀解析，key 取自环境经 CredentialStore

Ambiguity: 验收只给了 pigo -p "..."，未规定 provider/密钥装配方式。

Choice: ollama/ 前缀或含 11434 的 base-url → 本地 Ollama（无鉴权）；否则默认 OpenRouter。密钥经 NewCredentialStore(nil).GetAPIKey 从环境按 provider 名解析，绝不落日志。

Rationale: 复用既有 provider 与 auth 组件（US-011/US-012），CLI 只做装配；密钥按 key 名引用符合安全约束。

Deviations

Deviation 额外交付 cmd/pigo/main.go，超出"仅运行模式"的字面范围

Spec: #39 标题聚焦"headless/stdio 运行模式"，验收含库层能力。

Implemented: 除 internal/agent/headless.go 的 RunHeadless 外，新建了 cmd/pigo/main.go 可执行入口。

Why: 验收明列"真实终端验证：pigo -p '读取 README 并总结' 出结果"——没有 CLI 二进制无法满足。仓库此前无 cmd/ 目录，故一并建立。

Tradeoffs

Tradeoff stream-json 写失败后继续 drain，而非立即 return

Alternatives: (A) 首个写错误即记录到 writeErr、停止再写但继续消费 Events() 到 channel 关闭，循环后才 return；(B) 写错误立即 return。

Pros/Cons: B 更短，但 loop 的 producer goroutine 在 buffer=0 的同步 Emit 上会因消费者提前退出而永久阻塞——goroutine 泄漏。A 多几行，却兑现了 headless.go 顶部"never leak a goroutine"的契约。

Winner: A——正确性优先于简洁；broken pipe 下仍不泄漏。

Tradeoff 测试用本地 as() shim 而非 errors.As

Alternatives: (A) 测试内写一个只解一层的 as(err, **ErrRunFailed)；(B) import errors 用 errors.As。

Pros/Cons: B 更标准；A 免去测试文件为单次一层解包引入额外 import。

Winner: A——轻量且意图清晰；若将来出现错误包裹链再切回 errors.As。
