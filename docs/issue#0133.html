Implementation Notes

Issue #133 — 插件系统：生命周期事件订阅 (US-017) — 2026-07-17

Design Decisions

Decision Subscription declared in the manifest, not per-event registration

Ambiguity: The spec says "插件握手时声明订阅的事件类型" but not the exact shape.

Choice: Added Events []string to Manifest; the plugin lists the agentcore.Event* discriminants it wants at initialize time. pigo delivers only those via one-way event notifications.

Rationale: Reuses the existing handshake — no new subscribe/unsubscribe RPC. pigo can gate payload construction on Manager.Subscribers(type), so when nobody listens it does zero work.

Decision Reuse the existing agentcore event stream as the source

Ambiguity: "复用现有 agentcore 事件流作事件源" — but the stream is consumed in three drivers (REPL, headless, sub-agent).

Choice: A new plugin.EventNotifier maps each AgentEvent to a wire-safe payload and dispatches it. It is wired as the DrainStream/RunHeadless OnEvent callback — the single event-observation seam that already existed for stream-json.

Rationale: No new event plumbing in the loop; the notifier is just another observer on the one drain path. Headless composes it with the stream-json serialiser (both observe every event).

Decision Payload mirrors the stream-json envelope's field selection

Ambiguity: The spec doesn't define the event payload.

Choice: eventPayload emits only observable, non-secret fields (ids, tool names, counts, stop reasons, streamed text) — the same subset headless.eventEnvelope already sends over stream-json.

Rationale: One consistent shape for plugin authors and stream consumers; no risk of leaking provider keys or raw internal structs to a subprocess.

Deviations

Deviation Also bounded the shutdown notify, not just event delivery

Spec: requires the event notification to be timeout-isolated.

Implemented: the same eventTimeout bound was added to Plugin.Close's best-effort shutdown notify.

Why: a plugin that stops reading stdin fills its write pipe; without the bound, a blocked shutdown write holds the transport write-mutex and made Close wait the full closeGrace (measured 58s → 7s after the fix). Same failure mode the spec flagged for events, so it got the same guard.

Tradeoffs

Tradeoff Timeout-per-event on the drain goroutine vs. a per-plugin outbound queue

Alternatives: (a) deliver inline with a short per-plugin timeout; (b) give each plugin a buffered queue + dedicated writer goroutine that drops on overflow.

Chosen (a): SendEvent runs the write on a goroutine and selects against a 2s timer; a hung plugin costs at most 2s per event and is logged.

Why: far simpler, no per-plugin lifecycle/backpressure state, and events are fire-and-forget by contract. Cost: a chronically slow (but not hung) plugin adds up to 2s of latency per delivered event. Acceptable for the low-priority audit use case; a queue can be added later without changing the manifest.

Tradeoff Real-subprocess event test vs. mock, and the 60s→9s timeout test

Alternatives: assert delivery against a mock client, or run a real plugin that logs received events to a file.

Chosen: a real event-logging plugin (eventPluginSrc) proves subscribed-only delivery end-to-end; a real hung plugin proves the timeout bound. The hung-plugin test dropped from ~60s to ~9s once Close was also bounded — a bonus signal that the deviation above was correct.

Why: the whole risk is the process boundary and the blocking-write failure mode; a mock would exercise neither.

Open Questions

Question Sub-agent driver does not deliver events to plugins

Only the REPL and headless drivers wire the notifier. The sub-agent tool (runtime/subagent.go) consumes its own stream and currently has no plugin manager in scope. Confirm whether plugin events should fire for nested sub-agent runs, or only for the top-level run (current behavior).

Question No per-event ordering/delivery guarantee across a burst

Events are delivered in stream order, but each is fire-and-forget; a plugin that is briefly slow may have a later event's write complete before... it won't, since delivery is synchronous per plugin in DispatchEvent. Ordering holds per plugin. Flagging only to confirm the synchronous-per-plugin model is the intended guarantee.
