Issue #91 — TUI: 宽度感知的渲染与对齐 — 2026-07-10

Design Decisions

Decision 用 go-runewidth 直接算宽度，而非 lipgloss.Width

Ambiguity: AC 写「lipgloss.Width（或 go-runewidth）」，二选一。

Choice: 新建 rendering.go，用 runewidth.StringWidth/RuneWidth 实现 displayWidth/truncateWidth/wrapWidth，并把 go-runewidth 提升为 direct 依赖。

Rationale: 折行/截断需要逐 rune 累加列宽并在边界停下，RuneWidth(r) 正好提供单 rune 列宽；lipgloss.Width 只给整串宽度，做不了「逐 rune 判断是否越界」。go-runewidth 本就是 lipgloss 的底层，直接用它最贴合需求且无新增依赖树。

Decision 采用硬列折行（不按词边界）

Ambiguity: AC 只要求「按显示宽度在 rune 边界折行，不切半个宽字符」，未规定是否按单词折。

Choice: wrapWidth 纯按列宽在 rune 边界折，不做 word-wrap。

Rationale: transcript 内容是代码、CJK 散文这类字符导向文本，中文本无空格分词，按词折反而在 CJK 上无意义且易错。硬列折是安全默认；真正的 Markdown 排版留给 #94。

Decision renderEntry 增加 width 参数，仅对 tool-result 摘要做宽度截断

Ambiguity: AC 要求「firstLine/tool-result 摘要按显示宽度截断」，但普通文本行由 wrapWidth 折行即可。

Choice: renderEntry(e, width) 只在 entryToolResult 分支把首行 gist 截到 width - 前缀宽；其余种类交给外层 wrapWidth 折行。

Rationale: tool-result 是「一行摘要」语义（截断），普通文本是「完整可见」语义（折行），两者处理不同。给前缀 " ↳ " 预留列宽，保证整行含前缀不超屏。

Deviations

None 实现遵循 spec

AC 四项（显示宽度替换 len()、rune 边界折行不切宽字符、tool-result 按宽截断、宽字符折行/截断单测全绿）均按字面满足。

Tradeoffs

Tradeoff width <= 0 的降级行为

Alternatives: (a) width 未知（如首帧 WindowSizeMsg 未到）时不折行、不截断，原样输出；(b) 用一个默认宽度（如 80）兜底。

Chosen: (a)。

Why it won: 猜一个默认宽度可能与真实终端不符，反造成错位；原样输出至少不主动破坏，等首个 WindowSizeMsg 到达后自然进入宽度感知。wrapWidth 在 width<=0 时也避免了除零/死循环。
