Implementation Notes

Issue #132 — 插件系统：子进程 JSON-RPC 协议 (US-016) — 2026-07-17

Design Decisions

Decision Reuse internal/jsonrpc as the plugin transport

Ambiguity: US-016 specifies "external plugins as subprocesses" but not the wire protocol.

Choice: Plugins speak line-delimited JSON-RPC 2.0 over stdio via the existing internal/jsonrpc.Client, which was already built for MCP/sub-agent reuse.

Rationale: One transport, one crash-handling path, one place to test. No new framing/parsing code, and the plugin protocol stays language-agnostic (any executable that can read/write JSON lines).

Decision Three-method protocol: initialize / tools/call / shutdown

Ambiguity: The spec lists tools and (future) commands but not the handshake shape.

Choice: A single initialize returns the full Manifest (name, version, tools, commands) up front; tools/call forwards one invocation; shutdown is a best-effort notification on Close.

Rationale: Declaring everything at handshake time means pigo registers all plugin tools before the agent loop starts — no lazy discovery races. Commands is carried in the manifest now so command support (#133-area) needs no protocol change.

Decision Plugin tools run ToolExecutionSequential

Ambiguity: The AgentTool interface allows parallel execution; the spec is silent on plugin concurrency.

Choice: Every plugin-backed tool reports Sequential.

Rationale: A plugin call crosses a process boundary with unknown side effects and a single stdio channel per process. Sequential is the safe default; a plugin author cannot accidentally get concurrent calls multiplexed onto one pipe.

Deviations

Deviation A crashed/misbehaving plugin degrades to an error result, never a Go error

Spec: "plugins must be isolated" (general requirement).

Implemented: pluginTool.Execute always returns (AgentToolResult, nil) — a transport failure becomes a text result "<name>: plugin call failed: <err>". Discovery failures are logged to a warn writer and the plugin is skipped, so one bad plugin never aborts loading the rest.

Why: The agent loop treats a non-nil tool error as fatal; isolating at the tool boundary keeps a dead plugin from taking down the whole session.

Tradeoffs

Tradeoff Executable-bit discovery vs. a manifest/registry file

Alternatives: (a) scan $PIGO_HOME/plugins for executable regular files; (b) require a plugins.json registry listing each plugin.

Chosen (a): Any executable dropped into the dir is a plugin; non-executables and subdirs are ignored; load order is name-sorted for determinism.

Why: Zero-config "drop it in and it works" matches the config-dir convention already used for sessions/commands. A registry file adds a second source of truth to keep in sync. Cost: no per-plugin enable/disable short of chmod — acceptable for now.

Tradeoff Real-subprocess tests vs. mocked transport

Alternatives: mock jsonrpc.Client, or compile tiny real plugins with go build at test time.

Chosen: tests compile echoPluginSrc / crashPluginSrc into real executables and exercise the full launch→handshake→call path (including a genuine mid-call os.Exit(1) crash).

Why: The whole point of the feature is the process boundary; a mock would test the adapter but not the transport, framing, or crash isolation that actually carry the risk. Cost: ~4s test time for the builds — worth it for real coverage.

Open Questions

Question Command registration is declared but not yet wired

Manifest.Commands is parsed and carried, but pigo does not yet register plugin slash commands with the REPL command table. Confirm this lands in a follow-up (the manifest already supports it, so no protocol change is needed).

Question Tool name collisions across plugins / builtins

Plugin tools are appended to the builtin set in load order with no namespacing. If two plugins (or a plugin and a builtin) declare the same tool name, the registry behavior on duplicate is the arbiter. Should plugin tools be namespaced by Manifest.Name? Deferred pending a real collision.
