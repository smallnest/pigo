Implementation Notes

Issue #134 — 项目信任 Project Trust (US-018) — 2026-07-19

Design Decisions

Decision Tri-state trust via nullable *bool (true / false / null)

Ambiguity: The spec said trust is stored as path -> bool|null but did not define what null means versus an absent entry, nor how "just once" is represented.

Choice: internal/trust uses a Decision enum (Trusted / Untrusted / Undecided) backed by *bool: true=trusted, false=untrusted, nil=undecided. "Just once" is NOT persisted — it is an in-memory session grant (SetSessionTrust).

Rationale: A nullable boolean preserves the literal bool|null schema with full round-trip fidelity (tested by TestNullEntryRoundTrip), while a separate session map keeps "just once" genuinely per-process — persisting it would contradict "just this once". NearestTrustDecision returns a Found flag so the REPL can distinguish "no entry anywhere" (prompt) from "entry exists, even null" (don't re-prompt).

Decision Side-effect tool set = {bash, write, edit}; read-only tools never gated

Ambiguity: The spec said "具有副作用的工具（bash/write/edit）" but did not say how to classify the rest.

Choice: Only bash, write, edit are gated. read, grep, find, webfetch (read-only) and todo (in-memory) always run.

Rationale: Trust protects against unwanted filesystem/process mutation. Read-only inspection does not need a trust gate, and gating it would make the agent useless in untrusted dirs. Notably all three gated tools are ToolExecutionSequential, so the BeforeToolCall hook fires on the run-loop producer goroutine, never concurrently with itself.

Decision Trust gating is REPL-only; headless mode unchanged

Ambiguity: AC 5 ("未信任目录下 bash/write/edit 默认需确认") did not specify whether it applies to the headless (-p) path.

Choice: The BeforeToolCall hook is wired only into streamRun (the REPL). Headless (runtime.RunHeadless) keeps no hook and runs tools as before.

Rationale: Headless is non-interactive (CI/scripts); it cannot prompt. AC 3 ("REPL 询问") and the confirmation flow are inherently interactive. Applying trust to headless would either block tools silently (breaking scripts) or require a non-interactive policy the spec does not define.

Decision Confirmation prompt offers y / N / a (a = "always this session")

Ambiguity: The spec said "默认需确认" without specifying the prompt's choices.

Choice: The per-tool prompt is [y]es / [n]o / [a]lways (trust for this session) [y/N/a], default N (deny). a grants session trust via SetSessionTrust so subsequent side-effect calls skip the prompt.

Rationale: Default-deny keeps the safety property; a avoids prompt fatigue when the user has already decided to trust for the session, without persisting a grant they did not explicitly confirm via /trust or the startup dialog.

Deviations

Deviation /trust off clears session trust before persisting Untrusted

Spec: AC 4 said "/trust 命令保存当前项目信任决策" with no mention of session trust.

Implemented: /trust off calls Manager.ClearSessionTrust(cwd) before SetDecision(cwd, Untrusted).

Why: IsTrusted checks the in-memory session map before the persisted decision, so an "always" granted earlier in the session would keep the dir trusted until restart — making /trust off ineffective and its "side-effect tools will ask" message false. Clearing first makes off take effect immediately. Caught in code review (M2).

Deviation REPL main loop switched from bufio.Scanner to a shared *bufio.Reader.ReadString

Spec: Not specified; the REPL previously used bufio.Scanner.

Implemented: runREPL now reads lines via deps.in.ReadString('\n') on a single shared *bufio.Reader that both the main loop and the tool-call confirmation prompt read from.

Why: A Scanner has its own private buffer and can read past the current line, trapping bytes the confirmation prompt would then never see (it reads from the same underlying reader). Sharing one *bufio.Reader guarantees typed-ahead input is never split between the loop and a confirmation. This was required for correct trust-prompt behavior.

Deviation Trust file honors $PIGO_HOME in addition to ~/.pigo

Spec: AC 1 said the store is at ~/.pigo/trust.json.

Implemented: trust.DefaultPath() returns $PIGO_HOME/trust.json when PIGO_HOME is set, else ~/.pigo/trust.json.

Why: Every other pigo home-dir path (sessions, plugins, commands) already honors PIGO_HOME. Matching that keeps the trust store in the same place as the rest of the user's pigo state when the override is used.

Tradeoffs

Tradeoff SIGINT during a trust confirmation does not instantly interrupt

Alternatives: (a) ctx-aware prompt with a reader goroutine + select on ctx.Done(); (b) have the SIGINT handler inject a line into stdin.

Pros/cons: (a) leaks a goroutine blocked on the shared reader, which would then race with the next prompt's read — unsafe. (b) is platform-specific and hacky.

Chose: Document the limitation. It is safe: a post-Ctrl+C "yes" still aborts the tool because executeToolCall's emit(ToolExecutionStartEvent) returns ctx.Err() and the tool never runs. The cost is purely that the user must answer the prompt before the interrupt takes effect — a UX wart in a narrow scenario, not a safety issue.

Tradeoff Unbounded ReadString replaces the old 4 MiB Scanner cap

Alternatives: Keep Scanner with its 4 MiB cap; or write a bounded line reader on top of ReadSlice.

Pros/cons: Scanner reintroduces the buffer-trapping problem. A bounded reader adds ~15 lines of fiddly code for a pathological-paste OOM that does not occur on a terminal.

Chose: ReadString (unbounded) for correct buffer sharing, documented as an acceptable tradeoff for a terminal REPL where OS line-buffering bounds normal input. The cap loss is acknowledged in the replScanBufInit comment.

Tradeoff Confirmation writes to stdout from the run-loop goroutine

Alternatives: Route the confirmation through a custom event that the main goroutine renders in DrainStream.

Pros/cons: The event approach is cleaner (single writer) but a much larger change touching the event stream and StreamHandler.

Chose: Direct write from the hook. It is safe because streamRun uses an unbuffered event stream (EventBuffer=0, the default): by the time the hook runs, the main goroutine has finished draining the assistant's streamed text and is idle in DrainStream. The invariant is documented in trustBeforeToolCall; raising EventBuffer would require the event-routing approach.

Tradeoff confirmMu kept despite side-effect tools being Sequential today

Alternatives: Remove the mutex entirely, since bash/write/edit are ToolExecutionSequential and the hook never runs in parallel today.

Chose: Keep the mutex (nil-safe) as cheap future-proofing: if a side-effect tool ever becomes Parallel, concurrent confirmations would interleave on the shared stdin/stdout without it. The cost is one lock per gated call; the benefit is correctness under a plausible future change.
