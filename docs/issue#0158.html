Implementation Notes

Issue #158 — 分发 extension/MCP 适配器到 plugins 目录 — 2026-07-20

Design Decisions

Decision <name>.pkg/ 全量包 + <name> 启动器 的双件布局

Ambiguity: plugin.Discover 只运行 plugins/ 下的可执行普通文件（忽略子目录），而 npm extension 是一棵带 bin 入口、依赖同级文件才能运行的包目录。二者不匹配。

Choice: 把整棵包解到 plugins/<name>.pkg/（目录，Discover 会跳过），另写一个可执行 shell 启动器 plugins/<name>（文件，Discover 会运行），启动器 exec 包内真正的 bin 入口并转发 argv。

Rationale: 无需改动 plugin.Discover 的既有约定即可让 extension 被发现；bin 保留自身 shebang（与 npm 安装 bin 的方式一致），Node 脚本和原生二进制都能跑。

Decision 记录所有落盘文件供卸载精确清理

Ambiguity: Issue 只要求"分发到 plugins"。

Choice: DistributeExtension 返回所有创建的文件路径（含 payload 目录与启动器），交给 lockfile 记录。

Rationale: #163 的 uninstall / #164 的 update 需要精确知道装了哪些文件才能干净移除，避免残留。

Deviations

Deviation Windows 暂不支持 extension 安装

Spec: 未限定平台。

Implemented: runtime.GOOS == "windows" 时 DistributeExtension 直接返回明确错误，测试在 Windows 上 skip。

Why: 启动器是 /bin/sh 脚本，Windows 需要 .cmd 垫片且路径/可执行语义不同；本期聚焦 macOS/Linux，Windows 留作后续。skill/prompt/theme 分发（#159-#161）不受此限。

Tradeoffs

Tradeoff 启动器 shell 脚本 vs 直接软链/复制 bin

Alternatives: (A) 写 shell 启动器 exec 包内 bin；(B) 把 bin 文件本身复制/软链到 plugins/<name>。

Pros/Cons: B 更简单，但 bin 通常用相对路径 require 同级模块，脱离包目录会找不到依赖；软链又在跨卷/Windows 上不可靠。A 保证 bin 在其原始包目录内以正确 CWD 语义运行。

Winner: A。多一个小文件换来"bin 能真正跑起来"的正确性。

Open Questions

Question extension 运行时的依赖（node_modules）

Assumption: npm pack 只含包自身文件，不含 node_modules。假定 pi extension 要么零运行时依赖，要么自带打包产物。

Verify: 若某些 extension 需要安装依赖才能运行，需在 install 阶段补 npm install --omit=dev 或要求包预打包；确认本期 MCP 适配器是否零依赖。
