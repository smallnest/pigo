Issue #95 — TUI: 角色样式化的 transcript 渲染 — 2026-07-10

Design Decisions

Decision 抽出 rolePrefix(kind) 单一真源，renderEntry 与 assistant 路径共用

Ambiguity: AC 要求「角色前缀保留或对标 zero 优化，NO_COLOR 下靠前缀/glyph 区分」，未规定前缀集中管理方式。

Choice: 新增 rolePrefix(kind entryKind) string，返回每类的 glyph+空格；renderEntry 原本散落的字面量（"you> "、"⚙ "、" ↳ "、"· "）改为调用它，assistant 也从同一函数取 glyph。

Rationale: 前缀是 NO_COLOR 下唯一的来源区分手段，集中到一处避免各 entry 分支与测试各写一份、易漂移；测试 TestRolePrefixesAreDistinct 直接校验各类前缀非空且互不相同。

Decision assistant 新增 ● 角色 glyph，且置于渲染体上方独立一行

Ambiguity: #94 的 assistant 走 glamour Markdown 渲染，原本没有角色前缀；AC 要求各 kind（含 assistant）都能靠前缀区分。

Choice: assistant 前缀取 ●，套主题 accent 色；把 glyph 放在 glamour 输出「上方单独一行」（glyph + "\n" + body），而非行内前缀。

Rationale: glamour 的输出宽度已绑定 viewport，若在首行行内加前缀会把该行推过折行宽度（实测 ● 前缀使 20 列限制下首行变 22 列，测试 fail）；独立 glyph 行既满足"可区分"又不破坏 #94 的宽度对齐。

Deviations

None 实现遵循 spec

AC 四项（各 entryKind 主题上色、角色前缀在 NO_COLOR 下仍可区分、带前缀行按显示宽度对齐中文不错位、build/vet/gofmt/test 全绿）均满足。着色能力其实在 #93 的 styleEntry 已就位，本 issue 补齐前缀集中化 + assistant 角色 glyph，闭合 AC。

Tradeoffs

Tradeoff assistant 角色标记用独立 glyph 行，而非行内前缀

Alternatives: (a) 像 user/tool 那样行内前缀 ● 正文…；(b) glyph 独占一行，正文在下。

Chosen: (b)。

Why it won: (a) 会把 glamour 已按 width 折好的首行再加宽 2 列导致溢出，且 Markdown 正文首行可能是标题/代码块，行内插前缀会破坏其块级排版。(b) 让 glamour 输出保持原样，代价是每条 assistant 多占一行——在对话流里可接受，反而让来源更醒目。

Open Questions

Question assistant ● glyph 的选型是否对标 zero

选用 ●（实心圆）作 assistant 标记，与 user you>、tool ⚙/↳、system · 区分。zero 具体用哪个字符未逐一核对；若需严格对标 zero 的角色符号体系，可在此调整（单宽字符即可，避免影响宽度对齐）。

Question 是否给角色前缀本身也做差异化配色

当前非 assistant 行整行套该 kind 的主题色（前缀与正文同色）。zero 有时让前缀/标签用更强调的色、正文用常规色。是否需要「前缀强调色 + 正文常规色」的两段式着色，留待样式体系收敛后确认；本 issue 先整行同色，简单且已满足 AC。
