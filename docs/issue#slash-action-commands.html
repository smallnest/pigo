Implementation Notes

Feature: TUI 动作型斜杠命令（/model、/help）— 无对应 GitHub issue（源于「为啥 tui 下没有斜杠命令」的追问）— 2026-07-10

Design Decisions

Decision 给 SlashCommand 增加 Action 通道，而非把副作用塞进 Expand

Ambiguity: 原设计 Expand func(args string) string 只产出 prompt 文本，结构上无法表达「切换模型」这类控制动作。需求（支持 /model）没规定如何扩展。

Choice: 新增 Action func(args string) string 字段与 ResolveOutcome，返回结构化 SlashOutcome{Handled, Kind, Prompt, Message}。Action 命令执行副作用并返回状态行，不产出 prompt；Expand 命令保持原样。

Rationale: 两类命令语义根本不同（一个生成 prompt 交给 agent 跑，一个改运行时状态后直接展示结果）。用独立字段区分比让 Expand 兼职副作用清晰得多，且 Action 闭包可捕获并修改 live 状态，这是纯函数 Expand 做不到的。

Decision 新增实例级 AddBuiltin，而非只用全局 RegisterBuiltin

Ambiguity: 现有内置命令走 init() 期的全局 RegisterBuiltin；但 /model 的闭包必须捕获 main 里创建的 live 运行配置，init() 时该状态还不存在。

Choice: 给 *SlashRegistry 加 AddBuiltin，把内置命令直接装到 registry 实例上，同样标 SourceBuiltin（对 user 命令仍胜出）。

Rationale: 保留 init()-only 的全局路径（并发契约不变），又让需要 per-run 状态的动作命令有合法注册点，无需引入全局可变状态或锁。

Decision live 运行配置无锁，靠 Update goroutine 单线程约束

Ambiguity: /model 会 mid-session 改 model/provider，run 闭包每轮读它——是否需要同步？

Choice: liveRunConfig 不加锁，注释声明「只在 bubbletea 单一 Update goroutine 上读写」（slash action 在该 goroutine 执行，startRun 也同步在其中调用 run 闭包）。

Rationale: bubbletea 契约保证所有状态变更都在 Update goroutine，符合项目既有 uiState 的无锁设计，避免过度工程。

Deviations

None 无 spec 可偏离

本功能无正式 PRD/issue，是对用户「为啥没有 /model」追问的直接响应。实现目标（可注册内置命令 + 支持动作型命令 + /model 运行时切换 + /help 列表）均已达成，build/vet/gofmt/test 全绿。

Tradeoffs

Tradeoff 保留旧 Resolve 字符串 API 而非直接替换

Alternatives: (a) 破坏性地把 Resolve 改成返回 outcome；(b) 新增 ResolveOutcome，Resolve 委托它并保持旧签名。

Chosen: (b)。

Why it won: 现有测试与潜在只处理 prompt 命令的调用方仍可用旧 API；旧 Resolve 对 action 命令报「handled 但 prompt 空、动作不执行」，语义安全。代价是两个入口并存，但注释已标明 action 命令必须走 ResolveOutcome。

Tradeoff /model 切换失败时返回状态行而非报错中断

Alternatives: (a) resolveProvider 失败即 abort/报致命错；(b) 返回「cannot switch」状态行，保持原模型不变。

Chosen: (b)。

Why it won: 交互式会话里误输入模型 id 很常见，静默保持当前模型并提示，比打断会话体验好。

Open Questions

Question /model 是否应校验模型 id 真实可用

当前 resolveProvider 只按前缀映射 provider（ollama/ 走 Ollama，其余走 OpenRouter），不校验模型是否真实存在——非法 id 要到下一轮真正请求时才失败。是否需要在切换时做一次轻量校验（如查模型列表）？留待确认。

Question 是否需要 /model 的补全或候选列表

现在 /model 无参数只显示当前模型。是否要让它列出内置 provider 支持的常用模型 id 作为候选，方便用户切换？本次未做。

Question 动作命令的状态行样式

动作结果与错误都走 entrySystem（· 前缀，system 主题色）。是否需要为「成功切换」与「错误」区分样式（如成功用 accent、失败用警告色）？当前统一 system 色，简单且够用。
