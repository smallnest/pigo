Implementation Notes

Issue #44 — sandbox 进程内门 + secret redaction（US-026）— 2026-07-10

Design Decisions

Decision 纯函数 Evaluate 与运行时 SandboxGate 分离

Ambiguity: 验收要求 "Evaluate(ctx, Request) Decision 纯函数门经 beforeToolCall 接入"，但未规定策略引擎与 hook 适配层如何切分。

Choice: Policy.Evaluate 是无副作用的纯函数（同一 Request 恒定返回同一 Decision）；SandboxGate 单独承担 hook 适配——BeforeHook 把 Decision 翻译成 Block、AfterHook 做 secret 擦除。二者放在 sandbox.go 与 sandbox_secrets.go。

Rationale: 纯函数便于穷举单测（deny path / deny command / redaction 各自独立），运行时接入与策略判定解耦，符合 pi→pigo 的 "callback 变 struct func 字段" 映射。

Decision Decision 零值 = DecisionDeny，全程 fail-closed

Ambiguity: 验收强调 "进程内门 + 逐命令审批绝不 fail-open"，但未规定各歧义分支的默认走向。

Choice: 枚举顺序 Deny=0, Prompt, Allow，使遗漏赋值的路径天然 deny；路径解析失败、未知 AccessKind、prompt 无 approver、命令不可解析——全部落到 deny/prompt，绝不 allow。

Rationale: 用类型系统的零值语义兜底安全默认，比依赖每个分支显式写对更可靠。

Decision secret 擦除接入 AfterToolCall 而非 registry defer

Ambiguity: 验收原文 "registry 单 choke-point defer scrubResultSecrets"，但 result 的最终裁定实际发生在 tool_executor.go 的 finalize 阶段，registry 只负责 schema 编译/校验。

Choice: 把 scrubContentList 挂到 AfterHook（AfterToolCallFunc），它按字段覆盖 Content、保留 Details/Terminate/IsError，是所有 result 流向模型前的真正单一 choke-point。

Rationale: AfterToolCall 已是执行器唯一的 result 收口，无需再在 registry 侧加 defer；擦除逻辑与既有 hook 契约天然契合。

Decision 精度优先正则 + 已知 key 注册表 + longest-match-first

Ambiguity: "精度优先正则" 未给出具体 key 形态清单。

Choice: 内置 8 类形态正则（openai/anthropic/aws/github/google/slack/bearer/PEM 私钥块），均以显著前缀+受限字符集锚定；外加 SecretRegistry 存环境/配置注册的明文值。所有命中按长度降序稳定排序后逐一替换，marker 仅含 label 不含值。

Rationale: longest-first 确保宽泛 pattern 不会截断更长的具体命中；anthropic-key 在注册表中排在 openai-key 前，等长命中时稳定序保证更具体标签胜出。全程只输出 [REDACTED:<label>]，恪守 "secret 值绝不写日志" 约束。

Deviations

Deviation OS 级隔离不实现，仅进程内门

Spec: US-026 "降级带底线（OS 隔离不做，进程内门 + 逐命令审批绝不 fail-open）"。

Implemented: 无 namespace/seccomp/容器隔离；安全边界完全由进程内 Policy.Evaluate + 逐命令 prompt 审批构成，且所有歧义分支 fail-closed。

Why: 验收本身即把 OS 隔离列为不做项，明确以 "进程内门不 fail-open" 为底线；本实现严格落在该底线上。

Tradeoffs

Tradeoff 真 AST（mvdan.cc/sh/v3）vs 正则匹配命令

Alternatives: (A) 用 mvdan.cc/sh/v3 解析 shell AST 提取每个 CallExpr 的 argv[0]；(B) 正则切分命令行按关键字匹配。

Pros/Cons: A 能结构化识别管道/子 shell/命令替换/重定向，且解析失败可 fail-toward-risky；B 零依赖但对 a | rm b、$(...)、引号嵌套等易漏判或误判。

Winner: A——验收明确 "shell 命令风险分类用真 AST"；不可解析命令回退 riskRisky（绝不 safe），安全性优先于零依赖。

Tradeoff 符号链接解析 evalSymlinksLenient 处理不存在的写目标

Alternatives: (A) 直接 filepath.EvalSymlinks（对不存在路径报错）；(B) 宽松解析：解析最深已存在祖先再拼回剩余段。

Pros/Cons: A 对尚未创建的写目标会失败、迫使 fail-closed 但过度拒绝；B 允许合法的新建写，同时仍能捕获 "jail 内软链指向 jail 外" 的越狱。

Winner: B——写目标常不存在，需在保持越狱防护的前提下允许合法创建。
