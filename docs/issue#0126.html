Implementation Notes

Issue #126 — 图像输入：Content 块 + 两家 provider 编码 (US-010) — 2026-07-17

Design Decisions

Decision Reused existing ImageContent — no new content type added

Ambiguity: The first acceptance criterion asks to "add an image block type (base64 + media type)".

Choice: agentcore.ImageContent{Type, Data, MimeType} and NewImageContent already existed (with the ContentTypeImage discriminant and a decode case), so the criterion was already satisfied. The work was provider encoding + input syntax + a capability guard.

Rationale: Re-adding the type would duplicate a working discriminated-union member. Verified the existing type covers base64 data + mime type exactly as specified.

Decision User content collapses to a plain string when no image is present

Ambiguity: The spec mandates the array/block form for images but is silent on text-only messages.

Choice: openAIUserContent / anthropicUserContent return a plain string for text-only messages and switch to the array/block form only when an ImageContent is present.

Rationale: Preserves the existing text-only wire shape byte-for-byte (many OpenAI-compatible gateways are picky), so this change is a no-op for the overwhelmingly common non-image turn. Mirrors pi's hasImages branch in anthropic-messages.ts / openai-completions.ts.

Decision Two input syntaxes: @image:<path> and Markdown ![alt](path)

Ambiguity: The spec offers both syntaxes as examples ("如 @image:./a.png 或 ![](path)") without mandating one.

Choice: A single regexp matches both; @image: paths run to the next whitespace (bare token), Markdown paths are delimited by parentheses (may contain spaces). Non-reference text becomes interleaved TextContent blocks.

Rationale: Supporting both costs one alternation in the pattern and matches the two conventions users would reach for. Delimiting rules follow each syntax's natural shape.

Decision Capability guard via new Model.SupportsImages flag, checked at StreamCompletion

Ambiguity: "非多模态模型收到图片输入时清晰报错" — where and how to detect capability.

Choice: Added SupportsImages bool to Model. checkImageSupport runs at the top of both drivers' StreamCompletion (right after the API-key check) and returns a "cannot build the stream" error — the same failure class as a missing key.

Rationale: The error surfaces before any network round-trip, consistent with the provider interface's dual failure model (build-time error vs. mid-stream terminal event).

Deviations

None — implementation followed the spec. All four functional criteria (image block type, two-provider encoding, input syntax, clear error on non-multimodal model) plus both test criteria are covered.

Tradeoffs

Tradeoff Unknown models (absent from catalog) are treated permissively

Alternatives: (a) fail closed — reject image input for any model not in the catalog; (b) fail open — allow it and let the provider validate.

Chosen: Fail open. checkImageSupport only errors when the model is in the catalog and declares SupportsImages:false. A model resolved via --protocol/prefix (built with SupportsImages:true in resolveProvider) or genuinely unknown defers to the upstream.

Why it won: The catalog is not exhaustive (arbitrary base-url + model id). Failing closed would block legitimate multimodal models the catalog simply hasn't enumerated. The guard's job is catching the known text-only case, not policing every id.

Tradeoff Mime type from extension, with content-sniff fallback

Alternatives: (a) always content-sniff via http.DetectContentType; (b) extension-only; (c) extension first, sniff fallback.

Chosen: (c) — mimeFromPath maps common extensions (png/jpg/jpeg/gif/webp); unknown extensions fall back to http.DetectContentType. Non-image/ results are rejected.

Why it won: Extensions are cheap and unambiguous for the common cases; the sniff fallback handles odd extensions without a hard dependency on file contents, and the image/ prefix check prevents attaching a text file as an image.

Open Questions

Question Should REPL-side image input be persisted/rendered specially in the session tree?

Image blocks are stored in the message like any other content and round-trip through the session store, but the REPL echoes only text and the HTML export (from #124) has no image rendering. Confirm whether inline image thumbnails in /tree or the HTML export are wanted as a follow-up, or if raw base64 persistence is sufficient for now.

Question Are all catalog/preset models actually multimodal?

resolveProvider marks every resolved model SupportsImages:true because pigo can't know per-id vision capability without a richer catalog. This means a text-only model reached through pigo will not get the clear error until the upstream rejects it. If precise per-model capability matters, the preset catalog would need a vision flag per entry — flagged as potential follow-up.
