Implementation Notes

Issue #121 — 会话树 id/parentId 结构与迁移 (US-005) — 2026-07-17

Design Decisions

Decision Entry 包裹器 + 独立 JSON 编解码，而非给每个 Message 加 id 字段

Ambiguity: Spec 只说“每条持久化 entry 增加 ID/ParentID”，没规定 id 存在消息内还是消息外。

Choice: 新增 Entry{ID, ParentID, Timestamp, Message} 包裹器，落盘形状为 {"id","parentId","timestamp","message":{…}}；内层 message 用 json.RawMessage 承载并复用 MessageList 的角色判别解码。

Rationale: 树元数据是持久化层的关注点，不该污染 agentcore.Message 这个跨层共享的密封接口。包裹器让 message 编解码逻辑保持零改动，与 pi 的 SessionEntryBase 分层一致。

Decision entry id 用 4 字节 crypto/rand（8 位十六进制），不复刻 uuidv7

Ambiguity: Spec 只要求“补 id”，未规定 id 生成算法。pi 用 uuidv7().slice(-8)。

Choice: newEntryID() 返回 4 随机字节的 8 位 hex；crypto/rand 极罕见失败时退化为时间戳派生 id，保证写入永不中断。

Rationale: entry 顺序由 ParentID 链给出，无需 uuidv7 的时间排序性；8 位宽度与 pi 对齐，单文件内碰撞概率可忽略。避免引入 uuid 依赖。

Decision Load 保持扁平返回值，新增 LoadEntries 暴露树

Ambiguity: 引入树后，既有 Load 调用方（resume 走 Load(opts.resumeID)）该不该变签名。

Choice: Load 内部改调 LoadEntries 再摊平成 MessageList，签名与行为完全不变；新增 LoadEntries(id) (SessionHeader, []Entry, error) 给需要 id/parentId 的场景。

Rationale: 不破坏现有 resume/REPL 调用方，同时满足验收条件 (c) 的树访问需求。是最小改动的正确边界。

Deviations

Deviation PathToLeaf 对断链/环“容错止步”，不像 pi 那样抛错

Spec: 验收条件 (c) 只要求“沿 parentId 回溯得线性路径”。pi 的 getPathToRoot 在父节点缺失时直接 throw。

Implemented: PathToLeaf 遇到缺失父节点或环时停在最后一个可解析祖先，返回已恢复的前缀（空/未知 leafID 返回 nil）。

Why: 部分损坏的会话文件仍应尽力恢复，符合 List“跳过损坏而非整体失败”的既有容错风格。抛错会让一条坏链毁掉整个 resume。

Tradeoffs

Tradeoff 加载时迁移 vs. 就地重写旧文件

Alternatives: (A) 读到 v1/v2 文件时合成 id/parentId 只存内存；(B) 加载后立即把文件重写为 v3。

Chosen: A（readSession 按 header.Version >= 3 判断是包裹 entry 还是裸 message，裸行现补 id 并链 parentId 到前一条）。

Why: 只读加载不产生意外写盘，旧文件保持原样直到下次显式 Save 才升到 v3，行为可预测、无副作用。代价是每次加载旧文件都要重新合成 id（旧文件 id 不稳定），但旧文件本就没有树语义，可接受。

Tradeoff 迁移时 Timestamp 用 header.UpdatedAt vs. 零值

Alternatives: (A) 旧行合成 entry 的 Timestamp 填 header.UpdatedAt；(B) 填零值。

Chosen: A。

Why: 旧格式无逐条时间戳，UpdatedAt 是文件级唯一可用的时间近似，比零值更有意义；且不影响 ParentID 链（顺序仍由行序决定）。

Open Questions

Question 是否需要暴露 fork/clone/getChildren 等完整树导航 API？

本 issue 只落地 id/parentId 结构、迁移与 PathToLeaf。pi 还有 getBranch/getChildren/fork。若后续 issue 要真正的分叉会话，需要在此基础上补这些 API 与写入分叉 entry 的路径。

Question Append 每次 load-modify-save 会给已有消息重新生成 id

当前 Append 复用 Save（整文件重写），因此每次 append 后全体 entry 的 id 都会变。线性会话下无碍（链仍自洽），但若将来有外部引用某条 entry 的稳定 id（如书签、分叉点），需改为真正的增量追加以保持 id 稳定。请确认当前无稳定 id 引用需求。
