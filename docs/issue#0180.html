Implementation Notes

Issue #180 — 支持 --system-prompt / --append-system-prompt (对标 pi) — 2026-07-20

Design Decisions

Decision --system-prompt maps directly onto the existing PromptConfig.BaseInstruction

Ambiguity: The issue asked to "use a custom system prompt instead of the default coding-assistant prompt" but the prompt assembler already had a BaseInstruction field with empty-string-means-default semantics.

Choice: Wire the flag straight into BaseInstruction — no new field, no new branch. The existing base := cfg.BaseInstruction; if base == "" { base = DefaultBaseInstruction } logic already does the override.

Rationale: Reuses proven code, keeps the environment block and AGENTS.md injection layered on top exactly as before. Minimal surface area.

Decision File-or-literal resolution lives in cmd/pigo, not in the pure assembler

Ambiguity: The spec says --append-system-prompt takes "text or file contents" but is silent on where the file read happens.

Choice: Added resolveAppendInstructions in cmd/pigo/run.go (the I/O layer) which stats each value and reads it if it's a regular file, else treats it as literal text. BuildSystemPrompt stays a pure string assembler receiving already-resolved AppendInstructions []string.

Rationale: Keeps the assembler deterministic and disk-free (its tests inject a fake ReadFile). Disk I/O for append values is an CLI concern, so it belongs at the CLI boundary.

Decision Empty append entries are trimmed and skipped

Ambiguity: The repeatable flag can receive blank or whitespace-only values; the spec doesn't say what to do with them.

Choice: strings.TrimSpace each entry in the append loop and continue on empty, so a stray --append-system-prompt "" doesn't inject dangling blank lines.

Rationale: Avoids polluting the assembled prompt with empty separators; matches the tolerant behavior tested in TestBuildSystemPromptAppendInstructions.

Deviations

None — implementation follows pi's semantics as described in the issue.

Tradeoffs

Tradeoff Asymmetric file resolution: --system-prompt is literal, --append-system-prompt is file-or-literal

Alternatives: (a) make both flags file-or-literal for consistency; (b) keep --system-prompt literal-only, matching pi.

Pros/cons: Symmetry would be tidier, but pi/Claude Code treat --system-prompt as literal text and only --append-system-prompt as file-capable. Diverging would break the "对标 pi" contract callers expect.

Why chosen: Fidelity to pi wins over internal symmetry — this is a compatibility feature.

Tradeoff Widening setupAgentEnv's positional signature vs. passing a config struct

Alternatives: (a) add two more positional params (current, now 6 args); (b) pass the whole cliOptions struct or a dedicated prompt-config struct.

Pros/cons: A struct would stop the signature widening as more flags land, but refactoring the call sites now is out of scope for this feature and would enlarge the diff/review surface.

Why chosen: Kept positional params to stay minimal; flagged the struct refactor as a possible future cleanup (see Open Questions).

Open Questions

Question Directory paths for --append-system-prompt are treated as literal text

Assumption: If a value stats as a directory, it falls through to literal-text append (not an error). A genuine typo pointing at a dir would be appended verbatim.

Verify: Is silent literal-append acceptable for a directory path, or should it error like an unreadable regular file does? Current behavior matches "only a regular file is read; everything else is literal."

Question Should setupAgentEnv take a config struct?

Assumption: The 6-positional-arg signature is fine for now.

Follow-up: If more prompt/config flags arrive, consider collapsing the params into a struct to avoid arg-order mistakes at the two call sites.
