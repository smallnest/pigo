Issue #25 — Provider 接口 + dual failure model（US-007） — 2026-07-10

Design Decisions

Decision StreamCompletion 返回 *AssistantMessageEventStream 而非裸 <-chan>

Ambiguity: issue 验收写 (<-chan StreamEvent, error)，但 #18/#20 已确立 EventStream 泛型作为流抽象。

Choice: 返回 (*AssistantMessageEventStream, error)——EventStream 内部即含 chan T（Events() 暴露 <-chan>）外加 Result/Error 语义。

Rationale: 与已落地的 StreamFn 契约完全一致，Provider 可经 StreamFnFromProvider 零成本适配进 loop，避免两套流类型。

Decision 新增 CompletionRequest 聚合入参

Ambiguity: 验收要 "输入 model + context + options"，未规定是散参还是聚合。

Choice: CompletionRequest{Model, Context LlmContext, Config StreamConfig} 聚合三者。

Rationale: 单一结构体便于未来扩展字段而不破坏接口签名；与 StreamFn 的散参经 adapter 桥接。

Deviations

Deviation 事件类型复用 AssistantMessageEvent，不新造 StreamEvent

Spec: 验收命名为 StreamEvent / StreamEventError。

Implemented: 复用 #20 已建的 AssistantMessageEvent（含 StreamErrorEvent 终态事件）。

Why: 同一套 delta 事件已在 provider.go 定义并被 loop 消费，另造一套会割裂类型系统；StreamErrorEvent 正是验收要的 "流内终态 error 事件"。

Tradeoffs

Tradeoff Model.ThinkingLevels 用 json:"-" 排除序列化

Alternatives: (A) 序列化 ThinkingLevelMap；(B) json:"-" 排除。

Pros/Cons: A 让 Model 完整可 JSON 往返但 map[level]*string 序列化语义含糊（nil 指针 vs 缺键）；B 保持 Model JSON 干净，thinking 映射作为运行时能力表由 provider 代码持有。

Winner: B——ThinkingLevelMap 的三态（nil/缺键/值）是内存语义，不适合走 JSON；Model 的 JSON 面向 UI 展示能力标记即可。
