Issue #94 — TUI: assistant 输出 Markdown 渲染 — 2026-07-10

Design Decisions

Decision 用 glamour 做 Markdown 渲染，仅对 assistant entry 生效

Ambiguity: AC 举例「如 glamour，或与主题配套的渲染」，未强制具体库。

Choice: 引入 github.com/charmbracelet/glamour，新增 markdown.go 的 mdRenderer；renderTranscriptEntry 只在 e.Kind == entryAssistant 时走 Markdown 路径，其余 entry 保持 #91/#93 的 wrapWidth+styleEntry 路径。

Rationale: glamour 是 charm 生态的成熟 Markdown 终端渲染器（goldmark + chroma 高亮），代码块高亮/边框开箱即用。只对 assistant 生效，避免把 user/tool/system 行误当 Markdown 解析（如 user 输入的 # 不该变成标题）。

Decision glamour 渲染宽度绑定 viewport，NO_COLOR 切 notty 样式

Ambiguity: AC 要求「渲染宽度与视口一致」「NO_COLOR 降级朴素渲染」，未规定实现方式。

Choice: NewTermRenderer(WithStandardStyle(style), WithWordWrap(width))，style 在 noColor() 时取 "notty" 否则 "dark"。renderer 按 (width, noColor) 缓存，仅在其变化（终端 resize / 测试切换 NO_COLOR）时重建。

Rationale: 宽度随 viewport 传入使 CJK 折行与视口对齐（US-003）；notty 样式是 glamour 官方的无色降级；缓存避免每帧重建 renderer（glamour 构造较重）。

Decision 保留 fenceBuffer：渲染前先补齐未闭合围栏

Ambiguity: AC 要求「保留原 fenceBuffer 语义，流式未闭合 ``` 优雅呈现」。

Choice: assistant 分支调用 m.md.render(fenceBuffer(e.Text), width, noColor())——先用既有 fenceBuffer 补一个收尾 ```，再交给 glamour。

Rationale: glamour 对未闭合代码块的处理不稳定；先补齐让它当作完整 fenced block 解析，代码内容不丢、布局不破，复用而非重写 #91 已验证的 fenceBuffer。

Deviations

None 实现遵循 spec

AC 五项（Markdown 渲染 + 代码块高亮、fenceBuffer 语义保留、渲染宽度随视口且中文不错位、NO_COLOR 降级纯文本、build+test 全绿）均满足。

Tradeoffs

Tradeoff glamour 折行之上再叠一层硬字符折行（reflow/wrap）

Alternatives: (a) 仅靠 glamour 的 WithWordWrap；(b) glamour 之后再用 reflow/wrap 按显示宽度硬折。

Chosen: (b)。

Why it won: glamour 的 word-wrap 按空格断行，一长串无空格的中文（CJK 无空格可断）不会被折，仍会溢出视口——测试 TestAssistantMarkdownRespectsWidth 实测 72 列超出 20 列限制。叠加 reflow/wrap.String 做 rune 边界的强制硬折（保留 ANSI 转义），CJK 长串才真正落在 width 内。代价是两层折行，但 word-wrap 仍优先处理英文的语义断行，硬折只作兜底。

Tradeoff 为接入 glamour 升级了若干依赖

Alternatives: (a) 保持依赖不动，手写轻量 Markdown 渲染；(b) 引入 glamour 并升级其要求的传递依赖。

Chosen: (b)。

Why it won: glamour v1.0.0 依赖较新的 charmbracelet/x/cellbuf（0.0.13→0.0.15）与 golang.org/x/text（0.14→0.30），旧 cellbuf 与新 ansi.Style API 冲突导致编译失败；升级后全仓库 go build/vet/test ./... 全绿，无回归。手写渲染器达不到 chroma 语法高亮的质量且维护成本高。

Open Questions

Question 流式逐 token 重渲染的性能

当前每次 refreshViewport 对每条 assistant entry 全量重渲染 Markdown。长回复 + 高频 token 事件下，glamour 全量渲染可能有开销。renderer 已按 (width,noColor) 缓存，但 Render(src) 本身每次都跑。若后续出现卡顿，可考虑对未变化的历史 entry 缓存渲染结果。

Question glamour 样式与 #89 主题的一致性

glamour 用自带的 "dark" 标准样式，与 #89 tuiTheme 的调色板是两套独立配色。是否需要把主题色注入 glamour 的 ansi.StyleConfig 以统一观感，留待样式体系收敛（#95 之后）确认。
