Implementation Notes

Issue #43 — [impl] 会话持久化（JSONL）（US-024） — 2026-07-10

Design Decisions

Decision JSONL layout: header line + one message per line

Ambiguity: The spec (issue #16, 会话格式 #5) fixes only "本地 JSONL，内部自洽，不与 pi 互读" — it does not prescribe the on-disk record layout.

Choice: Line 1 is a SessionHeader (version + metadata); every subsequent line is one message, encoded with the same "role"-discriminated shape as agent.MessageList.

Rationale: Listing reads only line 1 (cheap), and message decoding reuses the existing discriminated MessageList.UnmarshalJSON by wrapping each line in a one-element array […] — no second, drift-prone codec.

Decision Schema versioning with a hard forward-incompatibility guard

Ambiguity: The spec mentions a migration-chain design space but leaves the guard behavior unspecified.

Choice: SchemaVersion = 1 is written into every header; version == 0 (missing) and version > SchemaVersion (newer than the binary) are both rejected on read.

Rationale: A newer file must never be silently misread by an older binary; failing loudly preserves the migration hook for later versions.

Decision Atomic writes via temp file + rename

Ambiguity: Durability semantics were not specified.

Choice: Save writes to <id>.jsonl.tmp then os.Renames over the target.

Rationale: A concurrent or crashing reader never observes a half-written session; rename is atomic on the same filesystem.

Decision Time-ordered session IDs

Ambiguity: ID scheme unspecified.

Choice: NewID = UTC 20060102-150405 + a 6-digit microsecond suffix, so IDs sort lexicographically by creation time and disambiguate within a second.

Rationale: Human-readable, collision-resistant for interactive use, and naturally ordered without a separate index.

Decision Whole-session Save after each run rather than incremental Append in the CLI

Ambiguity: The spec requires persistence but not the write cadence.

Choice: The interactive driver keeps one growing AgentContext and calls Save with the full message slice when each run settles. Append exists and is unit-tested for callers that want incremental writes.

Rationale: The context grows in place across turns, so a single Save of the whole slice is simplest and always consistent; interactive session sizes make the rewrite cost negligible.

Deviations

Deviation CLI resume path bypasses Store.Resume's trailing-assistant guard

Spec: Resume "对接 agentLoopContinue", whose precondition rejects a context whose last message is an assistant message.

Implemented: Store.Resume still enforces that guard (for the headless/continue path and its tests). But the interactive CLI resume uses Store.Load directly and rebuilds the context, because the user always types a fresh prompt first — a new user message is appended before the loop runs.

Why: A normally-ended session (trailing assistant reply) is the common case a user wants to resume in the TUI; refusing it would make resume nearly useless interactively. The agentLoopContinue invariant is still upheld because the appended user prompt makes the last message a user message before any loop step.

Tradeoffs

Tradeoff Append does load-modify-save instead of true file append

Alternatives: (A) genuine O_APPEND of new message lines; (B) load whole session, append in memory, rewrite atomically.

Pros/cons: (A) is O(new) I/O but cannot update the header's UpdatedAt in place (the header is line 1), and risks a torn file on crash. (B) is O(total) I/O but keeps the header authoritative and reuses the atomic Save path.

Chosen: (B) — correctness and header freshness matter more than I/O at interactive session sizes.

Tradeoff Scanner buffer capped at 16 MB per line

Alternatives: default 64 KB bufio.Scanner cap; a streaming json.Decoder; a large fixed cap.

Pros/cons: 64 KB truncates long tool-result lines; a full streaming decoder complicates the line-oriented format; a fixed 16 MB cap is simple and covers realistic tool outputs.

Chosen: 16 MB cap — pragmatic headroom without unbounded memory.

Open Questions

Question Interactive resume + real provider not manually verified

Unit tests cover write/read/resume/replay and the CLI builds with --resume/--continue/--list-sessions, but a live end-to-end resume against a real model in a terminal has not been exercised. Worth a manual smoke test before relying on it.

Question Sessions never expire or get pruned

~/.pigo/sessions grows unbounded. Should there be a retention policy or a --rm/prune command? Not required by the acceptance criteria — confirm whether it's wanted as follow-up.

Question Concurrent writers to the same session id

Two pigo processes resuming the same id would last-writer-win via rename. Interactive use is single-writer, so this is assumed acceptable — confirm no multi-process resume scenario is expected.
