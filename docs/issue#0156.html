Implementation Notes

Issue #156 — 检测本机 npm 并从 npm 拉取包内容 — 2026-07-20

Design Decisions

Decision 用 npm pack 而非 npm install

Ambiguity: Issue 允许「npm pack 后解包，或 npm install 到临时目录」两种方式。

Choice: 选 npm pack --ignore-scripts：仅下载 .tgz 到临时目录并自行解包，不触发依赖安装或生命周期脚本。

Rationale: pack 是纯下载，安全（下载包不会执行其代码）、快、无副作用；运行 extension 是后续独立步骤。install 会拉整棵依赖树且可能跑脚本，超出「获取内容」所需。

Decision 自实现 tar.gz 解包并做路径穿越防护

Ambiguity: 未规定用何种解包方式。

Choice: 用标准库 archive/tar+compress/gzip 解包，safeJoin 拒绝逃逸 dest 的 ../ 条目，只处理常规文件与目录（跳过 symlink/设备）。

Rationale: 零外部依赖；对第三方 tarball 做路径穿越防护是安全底线。

Deviations

Deviation npm 二进制名做成变量便于测试

Spec: Issue 只说「检测 npm 是否在 PATH」。

Implemented: var npmExecutable = "npm"，测试可临时替换/配合 PATH 注入 fake npm 脚本，实现真正的端到端 fetch 测试而不触网。

Why: 满足验收里「有测试验证」的同时避免依赖真实 registry；生产行为不变。

Tradeoffs

Tradeoff tarball 定位：解析 npm stdout vs 扫描目录

Alternatives: (A) 只信 npm pack 打印的文件名；(B) 扫描 pack-destination 找唯一 .tgz。

Pros/Cons: stdout 精确但不同 npm 版本输出格式有差异（可能含 JSON/路径）；扫描目录健壮但假设目录干净。

Winner: 两者兼用——先试 stdout 文件名，失败则回退扫描。因每次用全新临时目录，扫描回退是安全的。
