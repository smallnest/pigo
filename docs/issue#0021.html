Implementation Notes

Issue #21 — 工具三段式 prepare→execute→finalize（US-004） — 2026-07-10

Design Decisions

Decision 三段拆为三个独立函数 + 短路结果指针

Ambiguity: 验收要求 prepare→execute→finalize，但未规定内部函数边界。

Choice: prepareToolCall 返回 (tool, args, *AgentToolResult, isError)——非 nil result 即短路（未找到/校验失败/block/abort），直接进 finalize 不 execute；runTool 负责 execute+panic 恢复；finalizeToolCall 负责 afterToolCall 覆盖 + 事件 + 组装消息。

Rationale: 让 "所有失败路径都走 finalize（因此都过 afterToolCall 钩子并发 end 事件）" 这一 pi 语义显式化，避免每个错误分支各自组装消息。

Decision block 默认文案 + 可选 Content/Details 覆盖

Ambiguity: 验收只说 "block 产出 error result"，未规定 result 内容。

Choice: BeforeToolCallDecision 带可选 *Content/*Details：不设则用默认 "blocked by beforeToolCall" 文案，设了则用调用方的（权限系统可给出拒绝原因）。

Rationale: 权限/沙箱场景（FR-26）需要向模型解释为何拒绝，硬编码文案不够。

Deviations

Deviation 所有失败编码为 error tool result，不返回 Go error

Spec: 验收写 "抛错时转为 error tool result 而非 panic"。

Implemented: executeToolCall 签名 (ToolResultMessage, bool)——完全不返回 error。未找到/校验/block/abort/tool error/panic 六种失败一律产 IsError=true 的消息。

Why: 契合 provider 层地基约定——失败回喂模型自我修正，loop 只有一条消息记录路径，无双路径错误处理。

Tradeoffs

Tradeoff afterToolCall 字段级覆盖用四个 if 而非反射/合并

Alternatives: (A) 逐字段 nil 判断覆盖；(B) 反射/深合并。

Pros/Cons: A 直白、与 pi "无深合并" 语义一致，每个指针字段独立三态；B 更 "通用" 但引入深合并语义，恰是 pi 明确拒绝的。

Winner: A——验收明确 "无深合并，与 pi 一致"，四个 if 正好表达字段级替换。
