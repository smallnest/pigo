Implementation Notes

Issue #36 — edit 工具（含 diff）（US-017）— 2026-07-10

Design Decisions

Decision 非唯一匹配默认报错，唯一性用 strings.Count 判定

Ambiguity: 验收要"不唯一时报错"，未规定唯一性判定粒度。

Choice: 用 strings.Count(original, old_string)：0 → not found，>1 且未开 replace_all → 报"not unique（N matches）"，恰好 1 → 替换。

Rationale: 与 pi 的 edit 语义一致——强制模型给足上下文以定位唯一编辑点，避免误改；错误信息带匹配数便于模型补上下文。

Decision 自实现基于 LCS 的行级 diff 供 UI 渲染

Ambiguity: 验收要"返回 diff 供 UI 渲染"，未规定 diff 格式与来源。

Choice: 用标准 LCS 动态规划生成行级 -/+/上下文 diff，带 --- a/ +++ b/ 头，放进结果文本与 Details.diff。

Rationale: 避免引入第三方 diff 依赖（保持零依赖）；LCS 行 diff 足够 UI 渲染，实现可测且确定。

Deviations

Deviation diff 为简化行级 diff，非完整 unified diff（无 hunk 合并/@@ 行号）

Spec: "返回 diff 供 UI 渲染"。

Implemented: 输出 --- / +++ 头 + 逐行 -/+/空格 标记，但不生成 @@ -l,s +l,s @@ hunk 头，也不做上下文裁剪。

Why: 编辑后 UI 只需展示改了什么；完整 unified diff 的 hunk 定位对单点替换是过度设计。如需可后续增强。

Tradeoffs

Tradeoff 自写 LCS diff vs 引入 diff 库

Alternatives: (A) 自实现 LCS 行 diff；(B) 引入 sergi/go-diff 等库。

Pros/Cons: A 零依赖、完全可控、O(n·m) 内存对工具级小文件足够；B 功能更全（字符级/patch）但增加依赖面。

Winner: A——项目坚持零外部依赖（同 transport/thinking），行级 diff 满足 UI 渲染需求。

Open Questions

Question 超大文件的 LCS diff 内存 O(n·m) 是否需要设上限

Assumption: edit 面向源码/文本文件（行数有限），LCS 表内存可接受。

Verify: 若允许编辑超大文件，O(n·m) 表可能过大；是否需要在行数超阈值时降级为"仅报告替换数、不生成 diff"？
