Issue #41 — [impl] 交互式 TUI（bubbletea v2）（US-022）— 2026-07-10

Design Decisions

Decision 纯 state machine 与 bubbletea Model 分层

Ambiguity: 验收要求「内部 state machine 单测（runID guard/steering 队列/两阶段 Ctrl+C），不做 golden-frame 快照」，但未说明如何让状态可测而不触碰终端。

Choice: 把所有交互状态与转移抽到无框架依赖的 uiState（state.go），bubbletea Model（model.go）只做键盘分发与渲染包装。单测直接调用 submit/pressCtrlC/applyEvent/finishRun 断言转移。

Rationale: state machine 不 import bubbletea、不需要伪终端即可用表驱动测试覆盖全部分支；渲染交给 bubbletea 的 View 差分，符合「不自研差分渲染」。

Decision runID stale-guard 用单调递增计数器

Ambiguity: 验收要求「runID stale-guard」，但未定义 runID 生成方式与判定点。

Choice: 每次 idle 状态下 submit 使 runID++；applyEvent(runID, ev) 与 finishRun(runID, err) 在 runID != s.runID 时丢弃。桥接 goroutine 把事件用启动时的 runID 打标后 Program.Send 回 Update。

Rationale: 中断旧 run 后立即开新 run 时，旧 run 的滞后事件必须不能污染新 run 的 transcript——计数器比较是最小且确定的判定。

Decision 运行中输入即 steering，经 GetSteeringMessages 注入

Ambiguity: 验收要「运行中输入 steering（pi per-turn 语义）」，但未说明队列如何对接 loop。

Choice: 运行中 submit 把文本入 steering 队列并回显；cmd/pigo 把 TUI 的 drainSteering 包装成 loop 的 GetSteeringMessages 钩子，loop 在每轮工具执行后拉取并在下一轮前注入。

Rationale: 复用既有 loop per-turn steering 机制（#23），无需 TUI 自造注入路径；drain 语义保证同一条 steering 不重复注入。

Deviations

Deviation 新增导出入口 agent.StartRun/ContinueRun

Spec: #41 只要求把 agent 回调桥接进 Update loop。

Implemented: 顺带在 loop.go 导出 StartRun（包 agentLoop）与 ContinueRun（包 agentLoopContinue），因为原入口是包内未导出的。

Why: TUI 在 internal/tui 包，需要跨包启动 run；保持 loop 内部实现不导出的同时，仅暴露两个薄封装。ContinueRun 也为 #43 会话续跑预留。

Tradeoffs

Tradeoff 桥接用 Program.Send vs tea.Cmd 阻塞读取

Alternatives: (A) 启动 goroutine 消费 EventStream，每事件 Program.Send 回 Update；(B) 用返回 tea.Cmd 每次阻塞读一个事件再回投。

Pros/Cons: B 与 bubbletea 命令模型贴合但每事件一次往返、流式增量延迟高，且难处理不定长事件流；A 让 unbuffered EventStream 被单 goroutine 完整 drain（无 producer 泄漏），所有状态变更仍集中在 Update goroutine。

Winner: A——流式渲染顺滑，且 -race 测试通过验证无数据竞争。
