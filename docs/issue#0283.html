Issue #283 — /btw: 在 /help 与命令补全中登记 — 2026-07-24

Design Decisions

Decision 单点登记同时驱动 /help 与斜杠补全

Ambiguity: Issue 分别列出"/help 显示"与"补全能补出 /btw"两条 AC，未说明是否需要两处独立改动。

Choice: 只在 registerLiveCommands（interactive.go:507）的"仅供 /help 列出"循环中登记 {"btw", ...}。/help 与行编辑器补全（line_editor.go 的 suggestions）都遍历同一个 reg.List()，故一处登记即满足两条 AC。

Rationale: 补全源就是 slash registry；无需在补全侧再维护一份命令名清单，避免两处清单漂移。

Deviations

Deviation 生产代码此前已在 #281 中落地，本 issue 仅补测试

Spec: AC 要求 /help 含 /btw、补全能补出 /btw。

Implemented: 登记条目在实现 #281 帮助文案时已随手加入（interactive.go:507），补全自然生效。本 issue 因此只新增 btw_help_test.go 两个测试来固化这一行为。

Why: 功能已满足全部 AC；缺的是防回归的测试。补测试而非重复改代码，符合"每个 AC 可验证"的要求，也防止未来有人删掉登记条目而无人察觉。

Tradeoffs

Tradeoff 测试直接调 registerLiveCommands 而非驱动整个 REPL

Alternatives: (A) 构造一个空 registry，调 registerLiveCommands，断言 reg.List() 含 btw 且 suggestions("/bt") 命中；(B) 跑完整 runREPL 输入 /help，正则匹配输出。

Chosen: A。Pros: 直接锚定登记这一真实契约，快、无 I/O、不受输出渲染/着色影响。Cons: 不覆盖 /help 输出的具体排版——但排版由通用 List 循环负责，已有其它路径覆盖。
