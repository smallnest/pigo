Implementation Notes

Issue #92 — TUI: 视口滚动浏览长对话 — 2026-07-10

Design Decisions

Decision 用 bubbles/v2/viewport 承载 transcript，而非自建 chatScrollOffset

Ambiguity: AC 写「引入 viewport（或等价的 chatScrollOffset 滚动状态）」，二选一。

Choice: 在 Model 里加 viewport.Model 字段，每次 transcript 变化调 refreshViewport() 把折行后的整段文本 SetContent 进视口，由视口负责裁剪与滚动。

Rationale: viewport 已实现 PgUp/PgDn/方向键、AtBottom、GotoBottom、resize 等全部所需能力，自己维护 offset 等于重写它且更易出 off-by-one。与 #90 textinput 一样，widget 放在 Model 层，uiState 保持 framework-free（FR-14）。

Decision follow-tail 状态放在 Model，靠 AtBottom() 推导

Ambiguity: AC 要求「在底部则自动跟随，已上滚则不打断」，未规定如何判定"在底部"。

Choice: Model 持一个 follow bool：新内容到达时若 follow 则 GotoBottom()；滚动键处理后用 m.viewport.AtBottom() 重算 follow。提交(steering)消息回显时强制 follow=true 以便用户看到自己的输入。

Rationale: 把"是否在底部"交给视口的权威判断，避免自己累计行数判断底部的漂移；用户上滚 → AtBottom() 为假 → 停止跟随；滚回底部 → 为真 → 自动重新跟随，恰好满足 AC 的双向语义。

Decision 视口高度 = 窗口高度 − 2 行 composer 预留

Ambiguity: AC 要求「resize 时视口尺寸更新，渲染不溢出」，未给出具体高度分配。

Choice: 常量 composerReservedRows = 2（一空行 + "> " 输入行），视口高 = height - 2（下限 1）。WindowSizeMsg 里同步 SetWidth/SetHeight。

Rationale: 固定预留保证输入行永不被高 transcript 挤出屏幕，且 resize 时随窗口重算，避免溢出。

Deviations

None 实现遵循 spec

AC 五项（引入 viewport、键盘上下滚动、底部自动跟随/上滚不打断、resize 更新视口、build+test 全绿）均按字面满足。

Tradeoffs

Tradeoff 首帧（WindowSizeMsg 未到）的降级渲染

Alternatives: (a) 视口未就绪时 View 回退为直接 dump 整段 transcript（#91 前行为）；(b) 用默认尺寸先建视口。

Chosen: (a)，以 vpReady 标志切换。

Why it won: 猜默认尺寸可能与真实终端不符，造成首帧错位；直接 dump 至少不破坏，首个 WindowSizeMsg 到达后自然进入视口模式。

Tradeoff 每次内容变化整体 SetContent 重渲

Alternatives: (a) 每次 transcript 变化重新拼接整段并 SetContent；(b) 增量只追加新行。

Chosen: (a)。

Why it won: 流式 assistant 消息是"就地增长"（upsertStreamingAssistant 覆盖末条），增量追加难以表达"改写最后一行"，整体重渲最简单且正确；transcript 规模对终端渲染而言开销可忽略。

Open Questions

Question 滚动键与 composer 编辑键的边界

目前把 up/down/pgup/pgdown/ctrl+u/ctrl+d 全部划给视口滚动，因此这些键不再进入 textinput。若后续希望方向键在输入行内移动光标（多行输入场景），需要区分"输入为空/多行"来决定归属。当前单行 composer 下此取舍无碍。

Question 鼠标滚轮未启用

viewport 支持 MouseWheelEnabled，但需在 bubbletea 程序层开启鼠标上报。本 issue 只做键盘滚动（AC 要求），鼠标滚轮留待后续按需接入。
