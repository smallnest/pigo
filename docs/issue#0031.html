Issue #31 — 模型注册表与 provider 目录（US-011）— 2026-07-10

Design Decisions

Decision 注册以 Provider 为单位，model id 全局唯一

Ambiguity: 验收要"按 id 查询 Model 与 Provider"，未规定同 id 跨 provider 冲突如何处理。

Choice: RegisterProvider 一次登记一个 provider 及其全部 Models；同一 model id 若已被别的 provider 占用则报错。

Rationale: model id 全局唯一保证 Resolve(id) 无歧义；单条 id→(Model, Provider) 是解析的核心契约。

Decision LoadCatalog 只补元数据，不登记 provider

Ambiguity: 验收要"模型清单可由数据文件生成"，但未规定数据文件与 provider 的关系。

Choice: LoadCatalog 解析 JSON []Model 合并进 id→Model 索引；不登记 provider，因此 catalog-only 的 model Resolve 会明确报"无 provider"错。

Rationale: 让能力元数据（UI 展示用）可独立于 provider 代码由数据文件供给；真正可调用仍需 provider 登记，两者解耦。

Deviations

None 无偏离

实现遵循验收：按 id 查 Model+Provider、内置 DefaultCatalog、数据文件经 LoadCatalog 生成、未知 id 经 Resolve/Model 明确报错、单测覆盖全部路径。

Tradeoffs

Tradeoff catalog 与 provider 的两级来源（数据 vs 代码）

Alternatives: (A) catalog 只补元数据、provider 才是权威可调用来源；(B) catalog 直接产生可用 provider。

Pros/Cons: A 职责清晰（元数据 vs 执行能力分离），但同一 model 信息可能来自两处需协调；B 一步到位但数据文件无法承载 provider 的执行逻辑（HTTP/decoder）。

Winner: A——provider 必然是代码（含 StreamCompletion），数据文件只能供元数据；分离后 catalog 可先于 provider 存在，UI 能提前展示能力标记。
