Implementation Notes

Issue #23 — 双层主循环 runLoop（US-006）— 2026-07-10

Design Decisions

Decision RunConfig 内嵌 LoopConfig，聚合四个循环级钩子

Ambiguity: 验收列出六个钩子，但 getSteeringMessages/getFollowUpMessages/prepareNextTurn/shouldStopAfterTurn 是循环级、beforeToolCall/afterToolCall 是工具级，配置如何分层未规定。

Choice: 新增 RunConfig：内嵌已有的 LoopConfig（每轮流式配置）+ BatchConfig（含 before/after 工具钩子）+ 四个循环级钩子字段 + EventBuffer。工具级两钩子复用既有 ToolExecutorConfig，不重复声明。

Rationale: 各层配置各归其位，runLoop 只消费 RunConfig，流式与批量执行的既有契约零改动。

Decision afterTurn 统一承载每个 turn_end 后的三段钩子

Ambiguity: steering 注入、prepareNextTurn、shouldStopAfterTurn 的相对顺序与触发条件在验收里分列，未给单一调用点。

Choice: 抽出 afterTurn(ctx, agentCtx, cfg, hadToolExecution)：先按 pi per-turn 语义拉 getSteeringMessages 并注入（仅当本轮有工具执行）、再 prepareNextTurn 应用 TurnUpdate、最后 shouldStopAfterTurn 决定是否退出。自然 turn end 与工具回灌 turn end 都经此点。

Rationale: 单一函数保证三个钩子的顺序与时机全局一致，避免在内层循环两处 turn_end 分支各写一遍导致漂移。

Decision TurnUpdate 用指针字段表达"仅替换已设置项"

Ambiguity: prepareNextTurn "可替换 context/model/thinkingLevel" 未说明部分替换语义。

Choice: TurnUpdate 全部字段为指针（*MessageList/*string/*[]AgentTool/*ThinkingLevel），applyTurnUpdate 只对非 nil 字段赋值；model/thinkingLevel 写回 cfg，context 三项写回 agentCtx。

Rationale: 沿用 pigo 既定的"三态用指针"映射，调用方可只换 model 而不动 context。

Deviations

Deviation 全部工具 terminate 时立即结束整个 run

Spec: 验收未显式描述 tool terminate 在双层循环中的传播。

Implemented: executeToolCalls 返回 allTerminate（批内每个结果都 terminate 才为真）后，runLoop 直接 finish()，不再进入 follow-up。

Why: 复刻 pi 语义——工具请求终止即终止整个 agent run；这是既有 batch_executor 契约的自然延伸，非偏离。

Tradeoffs

Tradeoff 生产者 goroutine + EventStream vs 同步回调

Alternatives: (A) 入口返回 LoopEventStream，runLoop 在 goroutine 中驱动、经 Emit 推事件、SetResult 交付新消息；(B) 入口接收一个 emit 回调同步跑完。

Pros/Cons: A 与 pi 的 async generator 语义一致、消费者可 range Events() 且能单独取 Result，天然背压；B 更简单但把并发/取消责任推给调用方，且与 US-003 流式层的 EventStream 风格不统一。

Winner: A——项目已确立 async generator→EventStream 映射，两入口 agentLoop/agentLoopContinue 均返回 *LoopEventStream。

Tradeoff run 结果 = 本次新增消息的快照拷贝

Alternatives: (A) 记录 startIdx，结束时 copy 出 agentCtx.Messages[startIdx:] 作为 AgentEndEvent.Messages 与 stream result；(B) 返回整个 agentCtx.Messages。

Pros/Cons: A 精确交付"本次 run 产出"、拷贝隔离后续变更；B 省一次拷贝但混入 run 前的历史消息，续跑场景语义错。

Winner: A——agentLoopContinue 从已有 context 续跑时，结果只应含本次新增。

Open Questions

Question emit 被取消（ctx cancel）时的收尾是否需要区分 aborted

Assumption: 当 Emit 因 ctx 取消返回错误，当前一律走 finish() 发 agent_end 并以已产出消息收尾。

Verify: 后续接入真实取消（US-013 中断）时，是否需要在结果里附带一个 aborted 标记或专门的终止消息，供会话恢复判断？当前留给上层。

Question steering 仅在"有工具执行"的 turn 后注入

Assumption: 依验收"每轮工具执行后拉取"，自然 turn end（无 tool call）不拉 steering。

Verify: 若产品希望用户在纯文本回复后也能插入 steering，需要放宽该条件——是否符合 pi per-turn 原意待确认。
