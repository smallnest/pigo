Implementation Notes

Issue #122 — /fork 与 /clone 命令 (US-006) — 2026-07-17

Design Decisions

Decision fork = 复制 root→leaf 路径到全新会话文件，而非在同一文件内分叉

Ambiguity: Spec 说“以选定消息为父创建新 leaf”“复制新分支不影响原分支”，但没规定分支是同文件多叶子还是独立文件。

Choice: Store.Fork(sourceID, leafID, now) 用 PathToLeaf 取出 root→leaf 的线性路径，SaveEntries 原样(保留 id/parentId)写入一个新 id 的会话文件，header 记 ParentSession=sourceID。

Rationale: 独立文件天然满足“互不污染”——向任一会话 Append 都不碰另一个。对齐 pi 的 createBranchedSession(也是复制路径到新文件 + parentSession 血缘)。多叶子单文件会让 resume/List 复杂化且违背既有“一个会话一个文件”的模型。

Decision /clone 与 /fork 共用一个 Fork 原语，只是 leafID 取法不同

Ambiguity: pi 里 clone 是 fork(leafId, {position:"at"})，fork 是 fork(entryId, {position:"before"})——两者是否该是两个函数。

Choice: 后端只有一个 Fork。/clone 传当前 leaf(最后一条 entry)的 id → 复制整段；/fork n 传第 n 条 user 消息的 ParentID → 复制到该消息之前。

Rationale: “position before/at”本质就是“选哪个 entry 当新 leaf”，无需两条代码路径。REPL 层负责把用户意图翻译成 leafID。

Decision /fork 交互拆成两步：无参列消息、/fork n 选择

Ambiguity: Spec 要求“列出历史 user 消息供选择”，但行式 REPL 无弹窗/方向键选择器(pi 有 showUserMessageSelector)。

Choice: /fork 无参时编号打印所有历史 user 消息(oneLine 截断)；/fork n 按编号选定。越界/非数字给出清晰错误。

Rationale: 与本仓库既有的纯行式 REPL 风格一致(#106 去掉了 bubbletea TUI)，无需引入交互式选择器依赖。

Deviations

Deviation /fork 与 /clone 在 REPL 循环里直接拦截，不走 SlashRegistry Action

Spec: 只要求命令存在并生效，未规定注册方式。

Implemented: 与 /compact//exit 一样在 runREPL 循环里拦截，因为它们要替换共享的 deps.header 和 deps.agentCtx——而 Action 闭包签名是纯 func(string) string，够不到这些 per-run 状态。仍在 registerLiveCommands 注册占位以便 /help 能列出。

Why: 沿用仓库已确立的“需要改运行时状态的命令在循环里拦截”约定，避免给 Action 闭包硬塞可变状态。

Tradeoffs

Tradeoff fork 前先整文件 Save，再 LoadEntries 复制

Alternatives: (A) fork 前把当前内存 context 落盘、再从磁盘读 entry 树复制；(B) 直接用内存里的 message 列表重建 entry 复制。

Chosen: A。

Why: 磁盘上的 entry 树是 id/parentId 的权威来源(内存里只有扁平 MessageList，没有 entry id)。多一次读盘换来 fork 的 id 保真与逻辑简单，对交互式命令的延迟无感。

Tradeoff 抽出 atomicWrite 复用给 Save/SaveEntries

Alternatives: (A) SaveEntries 复制粘贴 Save 的临时文件+rename 逻辑；(B) 提取 atomicWrite(id, write func(io.Writer) error) 共用。

Chosen: B。

Why: 消除重复的原子写样板，两个写入者共享同一份“临时文件→flush→close→rename”正确性保证。代价是多一层闭包间接，可接受。

Open Questions

Question fork/clone 后是否需要自动切换 resume 目标或提示 session id？

当前实现打印新 session id，用户下次 --resume <id> 可走到该 leaf。是否需要一个 /session 展示当前活动 id(将在 #125 落地)或 fork 后写入 “last session” 指针，待确认。

Question 空 fork(在首条消息之前)产出空会话是否符合预期？

/fork 1 会 fork 到第一条 user 消息之前 → 新会话为空(仅 header)。实现按“合法的空分支”处理，resume 后从零开始但保留 systemPrompt/model。若期望禁止此操作请确认。
