Implementation Notes

Issue #117 — 上下文 token 会计与压缩触发判定 — 2026-07-17

Design Decisions

Decision 用 type switch 而非 pi 的 role 字符串分派

Ambiguity: pi 的 estimateTokens 用 message.role 字符串 switch；pigo 的消息是密封接口（UserMessage/AssistantMessage/ToolResultMessage）。

Choice: 用 Go type switch 分派到具体类型，未知类型返回 0。

Rationale: 与 agentcore 既有的 discriminated-union 惯例一致（content.go / message.go 都用 type switch），编译期类型安全，无需运行时字符串比较。

Decision calculateContextTokens 只求和 input+output

Ambiguity: pi 的 calculateContextTokens 折叠了 cacheRead/cacheWrite/totalTokens，而 pigo 的 agentcore.Usage 只有 InputTokens/OutputTokens。

Choice: 返回 InputTokens + OutputTokens。

Rationale: pigo 目前不追踪缓存 token；两家 provider（anthropic/openai）解码时也只填 input/output。如果将来 Usage 扩展缓存字段，此函数是唯一的改点。

Decision ShouldCompact 额外守卫 contextWindow <= 0

Ambiguity: pi 只检查 enabled，未处理 contextWindow 未知（0）的情况。

Choice: contextWindow <= 0 时直接返回 false。

Rationale: provider.ModelInfo.ContextWindow 显式定义 0 表示未知；若不守卫，未知窗口会退化成 contextTokens > -reserveTokens 恒为真，导致每轮都误触发压缩。

Deviations

Deviation LastUsageIndex 用 -1 而非 null 表示"无"

Spec said: pi 的 ContextUsageEstimate.lastUsageIndex 是 number | null。

Implemented: Go 用 int 字段，无 usage 时取 -1。

Why: Go 无 nullable int；-1 哨兵是惯用做法，且切片索引本就非负，语义无歧义。

Tradeoffs

Tradeoff toolCall 参数按 RawMessage 字节长度计

Alternatives: (A) 直接用 len(c.Arguments)（已是紧凑 JSON 字节）；(B) 反序列化后再 json.Marshal 规范化再计长度。

Chosen: A —— 非空时直接取字节长度，仅在空 RawMessage 时 marshal 出 "null" 补 4 字符。

Why: 这是保守估算而非精确计费，规范化不值得额外分配；pi 用 safeJsonStringify(arguments)，字符数量级一致。
