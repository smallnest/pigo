Implementation Notes

Issue #120 — /compact 命令 + loop 自动压缩接入 — 2026-07-17

Design Decisions

Decision 自动压缩放在 runLoop 内部的 afterTurn，而非 PrepareNextTurn 钩子

Ambiguity: Spec 只要求“每轮结束调用 ShouldCompact 并在超阈值时压缩”，未指定接入点。

Choice: 在 afterTurn 中新增 maybeAutoCompact，并给 afterTurn 注入一个 emit 闭包。

Rationale: 现有钩子(PrepareNextTurn/ShouldStopAfterTurn)签名只能读写 agentCtx，无法发出事件。CompactionEvent 必须进入事件流(供 headless/stream-json 观察)，因此压缩逻辑只能住在能拿到 stream.Emit 的 runLoop 内。

Decision /compact 在 REPL 循环中直接拦截，不走 SlashCommand.Action

Ambiguity: 其它斜杠命令都用 Action 闭包实现。

Choice: 像 /exit//quit 一样在 runREPL 循环里拦截 /compact，调用 runManualCompact；仅在注册表登记以便 /help 列出。

Rationale: Action 的签名是 func(string) string，既不能跑 summarization stream 也不能改写共享的 agentCtx.Messages。手动压缩两者都要做。

Decision ContextWindow 经 LoopConfig / liveRunConfig 下沉，默认 128000

Ambiguity: runLoop 原本拿不到模型的 context window。

Choice: 给 LoopConfig 加 ContextWindow int，REPL 的 liveRunConfig 用 defaultContextWindow=128000 兜底。<=0 时禁用自动压缩。

Rationale: 窗口未知时宁可不压缩也不误伤；128000 足够大，只有真正的长会话才触发。

Deviations

Deviation 压缩流复用主轮的 API key 解析

Spec: 未提及 summarization 请求的鉴权。

Implemented: runCompaction 与 runManualCompact 都按主轮方式解析 key(动态 GetAPIKey 优先、静态 APIKey 兜底)后再构造 StreamConfig。

Why: 否则对需要鉴权的 provider，Compact 会以 “missing API key” 失败，自动/手动压缩形同虚设。此为 review-it 阶段发现并修复的真实缺陷。

Tradeoffs

Tradeoff 压缩失败非致命 vs. 中断会话

Alternatives: (A) 失败即中断 run；(B) 失败保留原上下文并发出带 ErrorMessage 的 CompactionEvent。

Chosen: B。

Why: US-004 明确要求压缩失败不打断会话；保留原始上下文让用户可继续对话，失败通过事件可观察而非静默。代价是超长上下文可能继续增长，但这比丢会话可接受。

Open Questions

Question defaultContextWindow 是否应随模型真实窗口动态设置？

当前 REPL 固定 128000 兜底。若 provider 的 preset 已知真实 ContextWindow，理想情况下应读取 preset 覆盖默认值，让阈值更精确。可作为后续 issue 跟进。

Question CompactionEvent.Reason 目前自动路径固定 "threshold"

结构预留了 "manual"/"overflow" 语义，但手动 /compact 当前直接打印而不发事件(REPL 未接入事件流)。若将来手动压缩也要走 stream-json，需要补 "manual" reason 的发射。
