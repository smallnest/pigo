Implementation Notes

Issue #162 — pigo install 端到端命令 — 2026-07-20

Design Decisions

Decision 子命令在 pflag 解析之前从 argv 剥离

Ambiguity: pigo 现有 CLI 是 flag 驱动的 agent 模式（-p/-m 等），未规定包管理子命令如何接入。

Choice: main() 在调 flag.Parse() 前先看 os.Args[1] 是否是 install|list|uninstall|update，是则走 runPackageCommand 并退出。

Rationale: 包管理子命令是位置参数语义，agent 的 flag 对它们无意义；提前剥离避免 flag 集污染，且让 pigo install npm:xxx 的 npm:xxx 不被误当 flag。

Decision pkgmgr.Install 编排，CLI 只做壳

Ambiguity: 编排逻辑放 CLI 还是 pkgmgr。

Choice: 端到端流程（parse ref → fetch → classify → 按类型分发 → 写 lockfile）落在 pkgmgr.Install；CLI pkgcmd.go 只解析 argv、解析 lockfile 路径、打印结果。

Rationale: 业务逻辑可单测（用 fake npm 端到端跑通），CLI 保持极薄，符合仓库既有分层。

Deviations

Deviation 多包安装：一个失败不中止其余

Spec: US 聚焦单包 pigo install npm:<name>。

Implemented: runInstall 接受多个 ref，逐个安装，某个失败打印错误并继续，最终退出码在有任一失败时为 1。

Why: 低成本增量，符合包管理器惯例；单包场景行为不变。

Tradeoffs

Tradeoff 分发中途失败的清理策略

Alternatives: (A) 失败即返回错误，已写文件留待重装覆盖；(B) 事务式回滚，失败时删除本次所有已写文件。

Pros/Cons: B 更干净但复杂，且每个分发器已"先清理自身旧目标"故重装幂等；A 简单，重跑 install 即可自愈。

Winner: A（本期）。分发幂等 + 重装覆盖已能收敛；真正的部分失败清理留给 #163 uninstall 的文件清单能力。
