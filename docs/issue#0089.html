Implementation Notes

Issue #89 — TUI: 引入 lipgloss 主题层 — 2026-07-10

Design Decisions

Decision palette（原始颜色）与 tuiTheme（解析后的 style）分离两层

Ambiguity: AC 只要求「定义 tuiTheme 持有各元素的 lipgloss.Style」和「buildTheme(mode) 将调色板解析为样式」，没规定调色板本身怎么存。

Choice: 用一个纯字符串的 palette struct 存 dark/light 两套原始 ANSI 颜色码，buildTheme 里再用一个共享的 fg() 闭包把 palette 塑造成 style。

Rationale: 颜色与样式塑形分离后，dark/light 两个模式复用同一段 style-shaping 逻辑（加粗、前景色），新增模式只需再加一个 palette 常量，不必重复写 style 构造。这也贴合 zero 的 buildTheme「resolve a palette into styles」的思路。

Decision 用 ANSI 基础色号（"12"、"15" 等）而非十六进制真彩色

Ambiguity: AC 未指定调色板用哪种色彩空间。

Choice: dark/light 都用 0–15 的 ANSI 基础色号（如 dark 的 user="12" 亮蓝、assistant="15" 近白）。

Rationale: 基础色号会自动适配用户终端的主题配色，在低色彩能力终端也能降级，比钉死的十六进制真彩色更稳。观感对标（US-007/#94）阶段若需要更精细的配色可再升级为十六进制。

Deviations

None 实现遵循 spec

AC 四项（新增 theme.go + tuiTheme + buildTheme；dark/light 两套调色板；遵守 NO_COLOR；build/vet/test/gofmt 全绿）全部按字面满足，无偏离。

Tradeoffs

Tradeoff 测试断言「含 \x1b[ ANSI 转义」而非断言具体颜色码

Alternatives: (a) 断言渲染结果精确等于某段带具体色码的字符串；(b) 只断言输出里包含 ANSI 转义序列前缀 \x1b[。

Chosen: (b)。

Why it won: lipgloss v2 的 Style.Render 在设了前景色时直接产出 ANSI，具体字节随库版本/色彩 profile 可能变动；断言精确色码会让测试脆弱。断言「有转义 vs. 无转义」恰好覆盖了本 issue 的语义核心——彩色时着色、NO_COLOR 时纯文本——且对库实现细节不敏感。

Tradeoff lipgloss v2 无需全局 color profile 即产出 ANSI

Alternatives: 最初测试里想调 lipgloss.SetColorProfile(lipgloss.ANSI) 强制着色（v1 惯用法），但 v2 没有这些导出符号（触发 undefined 编译错误）。

Chosen: 移除该调用，直接依赖 v2 的 Style.Render——只要设了前景色就输出 ANSI。

Why it won: 读了 style.go 的 Render 源码确认：设了 fg 即 te.ForegroundColor(fg) 直接产出转义，不依赖全局 profile。顺应库的实际行为比硬套 v1 API 更正确，也消除了编译错误。

Open Questions

Question 是否需要 themeAuto（探测终端背景色自动选 dark/light）？

本次只提供显式 themeDark/themeLight，默认 dark。zero 有 themeAuto 会请求终端背景色来自动决定明暗。PRD 已将 auto 列为后续增强——确认本阶段不做 auto、默认 dark 是否可接受。

Question 主题目前尚未接入渲染路径

#89 只建立主题层本身（tuiTheme/buildTheme），renderEntry/View 尚未使用它——那是 #94（角色样式化渲染）与 #92（spinner/卡片）的工作。因此本 issue 合入后 TUI 观感暂无变化，属预期。确认这一分层交付节奏 OK。

Question 配色可读性需实机确认

dark/light 的具体色号（如 light 下 assistant="0" 黑、system="3" 黄）是按常规对比度选的，尚未在真实明/暗终端里目视核对。接入渲染（#94）后建议在两种终端各扫一眼确认对比度足够。
