Issue #37 — bash 工具（US-018）— 2026-07-10

Design Decisions

Decision 非零退出码返回 Go error 以触发 isError

Ambiguity: 验收要"退出码非 0 标记 isError 并携带输出"，但 executor 只在工具返回 Go error 时才置 isError=true；普通 errorResult 结果不会。

Choice: 非零退出时返回 (AgentToolResult{output, Details.exitCode}, fmt.Errorf(...))——走 Go-error 路径，让 runTool 自动标记 isError；错误信息内嵌退出码与完整输出。

Rationale: 与既有 executor 契约一致，无需改动 tool_executor.go；输出同时进入 result content（供流式展示）与 error（供 isError 语义）。

Decision stdout/stderr 合并到单一缓冲，经并发安全 streamWriter 流式回传

Ambiguity: 验收要"流式返回 stdout/stderr"，未规定二者是否分流。

Choice: 定义 streamWriter{mu, buf, onUpdate}，stdout 与 stderr 共享同一互斥锁与 buffer；每次写入用累积快照调用 onUpdate，映射为 tool_execution_update 的增长型 partial。

Rationale: 终端交互中 stdout/stderr 交错是自然预期；共享锁保证两 goroutine 写入不竞态；累积快照契合 pi 的 partial 语义（每次更新是完整当前输出，而非增量片段）。

Decision 默认 2 分钟超时，上限 10 分钟；exec.CommandContext 负责杀进程

Ambiguity: 验收要"支持超时与取消（context 取消杀子进程）"，未给具体时长。

Choice: bashDefaultTimeout=2min、bashMaxTimeout=10min；用 context.WithTimeout 包裹传入 ctx，交给 exec.CommandContext，超时/取消时由标准库杀子进程。区分 DeadlineExceeded（超时）与 Canceled（取消）给出不同错误信息。

Rationale: 沿用 pi 的默认量级；CommandContext 是标准且可靠的取消杀进程机制，无需手写信号处理。

Deviations

Deviation 取消时杀单个进程而非整个进程组

Spec: "context 取消杀子进程"。

Implemented: 依赖 exec.CommandContext 默认行为，仅杀直接子进程，未设置 Setpgid 后 kill 负 PID 杀整个进程组。

Why: 标准库默认行为满足绝大多数场景，且跨平台可移植；进程组管理需 SysProcAttr 平台特化代码，对 MVP 属过度设计。若发现命令派生的孙进程残留，可后续增强（见 Open Questions）。

Tradeoffs

Tradeoff 合并 stdout/stderr vs 分流

Alternatives: (A) stdout/stderr 合并到一个 buffer；(B) 分别缓冲，result 中带两个字段。

Pros/Cons: A 保留时序交错、贴近终端体验、实现简单；B 可区分来源便于程序化处理，但丢失交错时序且 partial 结构更复杂。

Winner: A——bash 工具面向 agent 观察命令输出，交错时序比来源区分更有价值；与 pi 的合并输出一致。

Open Questions

Question 是否需要进程组级 kill 以清理孙进程

Assumption: 大多数命令取消时 CommandContext 杀直接子进程即可。

Verify: 若命令形如 foo & bar & 派生后台孙进程，取消后可能残留；是否需要 SysProcAttr{Setpgid:true} + 杀进程组？

Question 输出是否需要大小上限

Assumption: 命令输出可全量缓冲进内存。

Verify: 高产出命令（如 cat 大文件）会无界占用内存；是否需要按字节数截断并加省略提示（类比 read 工具的 2000 行上限）？
