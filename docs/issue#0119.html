Design Decisions

Decision CompactionMessage as a first-class message role, not a separate session-entry type

Ambiguity: The spec says "结果作为一条 CompactionEntry 写入 session JSONL" but pigo has no entry-type abstraction — it uses a flat []agentcore.Message, whereas pi uses a tree of SessionTreeEntry.

Choice: Added RoleCompaction = "compaction" and a CompactionMessage struct that implements the sealed agentcore.Message interface, wired into decodeMessage.

Rationale: A compaction checkpoint then flows unchanged through the existing MessageList decode, session JSONL append/load, and provider encoders. No parallel persistence path or entry-type switch is needed.

Decision Store CompactionDetails as json.RawMessage in agentcore

Ambiguity: The details (readFiles/modifiedFiles) shape is owned by the compaction package, but the message lives in agentcore.

Choice: CompactionMessage.Details is a json.RawMessage; the compaction package marshals/unmarshals its own CompactionDetails struct.

Rationale: Keeps agentcore free of any dependency on compaction, avoiding an import cycle while still round-tripping the metadata through JSONL.

Decision Index-based FirstKeptIndex instead of pi's firstKeptEntryId

Ambiguity: pi identifies the cut boundary by tree entry id; pigo has no ids.

Choice: CompactionResult.FirstKeptIndex is a slice index into the flat message list; RebuildContext returns [checkpoint] + msgs[FirstKeptIndex:].

Rationale: Matches pigo's existing flat-list model; the cut point is naturally an index from FindCutPoint.

Deviations

Deviation Usage tokens summed as input+output only

Spec/pi: pi folds cache-read and cache-write tokens into the context-token total.

Implemented: calculateContextTokens sums only InputTokens + OutputTokens.

Why: pigo's Usage struct does not track cache tokens, so there is nothing to fold in. Behavior is faithful to the data pigo actually has; if cache accounting is added later, this is the one place to extend.

Tradeoffs

Tradeoff maxTokens cap via cfg.Extra["max_tokens"] rather than a typed field

Alternatives: (a) add a dedicated MaxTokens int to StreamConfig; (b) pass through the existing Extra map[string]any.

Chosen (b): providers already read Extra["max_tokens"] via maxOutputTokensFor, so summarization reuses that path with no signature change across every provider.

Cost: loses compile-time typing on the key, but keeps the provider interface stable and avoids touching Bedrock/OpenAI/Ollama constructors.

Tradeoff Deterministic tool-call serialization (sorted keys)

Alternatives: serialize tool-call args in map iteration order, or sort keys.

Chosen: formatToolCall sorts keys before emitting name(key=value, ...).

Why: stable output makes serializeConversation testable and reproducible; the small sort cost is irrelevant at compaction scale.
