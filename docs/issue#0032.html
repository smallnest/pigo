Issue #32 — OAuth 与 API key 解析（US-012）— 2026-07-10

Design Decisions

Decision 三层凭证解析顺序：OAuth > 环境变量 > 配置文件

Ambiguity: 验收要求"从环境变量与配置文件读 API key"+"OAuth 流程"，但未规定三者同时存在时的优先级。

Choice: CredentialStore.GetAPIKey 按 OAuth token（自动刷新）→ env → config 依次解析，返回首个非空值。

Rationale: OAuth 是显式配置的强凭证，env 是运行时注入（CI/容器），config 是持久默认；越"临时/显式"的越优先，符合运维直觉。

Decision TokenSource 用 Leeway 提前刷新，签名匹配 LoopConfig.GetAPIKey

Ambiguity: "token 过期时刷新返回新 token"未说明如何避免在过期边界处竞争。

Choice: TokenSource 提供 30s 默认 Leeway，在真实过期前刷新；GetAPIKey(ctx, provider) string 与 LoopConfig.GetAPIKey 形状一致，可直接赋值接入 loop。

Rationale: 复用现有 streamAssistantResponse 的动态 key 解析钩子（#25/US-003），无需改 loop；Leeway 规避边界抖动。

Deviations

Deviation OAuth 提供通用 TokenSource 而非绑定具体 provider 的完整授权码流程

Spec: "支持 OAuth 流程（至少覆盖一个 OAuth provider 作为样例）"。

Implemented: 通用可注入 Refresh 的 TokenSource + OAuthToken，覆盖"短时 token 过期刷新"这一核心运行时契约；具体 provider 的浏览器授权码换取由上层装配 Refresh 闭包完成。

Why: 授权码交换涉及浏览器重定向/回调，属 CLI 装配层职责；本 issue 聚焦 loop 侧的凭证解析与刷新，抽象出可注入点比硬编码单一 provider 更通用、更可测。

Tradeoffs

Tradeoff OAuth 刷新失败时回退 env/config，而非直接报错

Alternatives: (A) 刷新失败即返回空/错误；(B) 刷新失败静默回退到 env/config 静态 key。

Pros/Cons: A 语义清晰但一次 OAuth 抖动就中断可用的静态凭证；B 提升可用性但掩盖了 OAuth 故障。

Winner: B——GetAPIKey 契约是"尽力返回可用 key"，与 loop 的静态 fallback 语义一致；OAuth 故障可由 TokenSource.Token 的显式错误单独观测。
