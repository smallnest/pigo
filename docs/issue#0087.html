Design Decisions

Decision Import pflag under the alias flag

Ambiguity: The AC mandates the switch to pflag but doesn't say whether to import it as pflag and rewrite every call site, or alias it as flag.

Choice: flag "github.com/spf13/pflag" — keep the local identifier flag, so every call site (flag.StringVar, flag.Parse, flag.Args) stays textually unchanged except the one StringVarP line.

Rationale: pflag is intentionally a drop-in for the stdlib API; aliasing minimizes the diff to exactly the semantic changes (the -p/--print merge), making the migration reviewable at a glance and leaving no stray pflag. qualifiers to reconcile.

Decision Collapse the two -p/--print bindings into one StringVarP

Ambiguity: stdlib registered -p and --print as two independent flags both writing &prompt. pflag models short+long as a single flag, so the AC explicitly asked for StringVarP.

Choice: flag.StringVarP(&prompt, "print", "p", "", "...") — print is the long name, p the short.

Rationale: This is pflag's canonical idiom and yields the correct combined usage line -p, --print string. Two separate StringVar calls would either collide or produce a duplicate-flag panic under pflag.

Tradeoffs

Tradeoff Ran go mod tidy to promote pflag from indirect to direct

Alternatives: (a) leave go get's initial // indirect marker on pflag; (b) run go mod tidy to reclassify it as a direct dependency.

Chosen: (b).

Why it won: pflag is now imported directly by cmd/pigo/main.go, so the // indirect comment go get left behind was factually wrong. go mod tidy moved it into the direct-require block, keeping go.mod honest and preventing a future tidy from producing a spurious diff.

Question pflag error-exit behavior differs subtly from stdlib

pflag and stdlib flag both print usage on an unknown flag, but exit-code and error-message wording differ slightly. The command's own os.Exit(1) paths (no prompt in a pipe, etc.) are unaffected since they don't depend on the flag library.
