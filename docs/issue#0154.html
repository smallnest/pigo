Issue #154 — 定义包清单 lockfile 与安装目录约定 — 2026-07-20

Design Decisions

Decision 新建 internal/pkgmgr package，镜像 internal/trust 的约定

Ambiguity: PRD/Issue 未指定 lockfile 模块放哪、用什么加载语义。

Choice: 新建 internal/pkgmgr，lockfile 的 load/save 语义完全对齐既有的 internal/trust.Manager：文件缺失 = 空清单（非错误），文件存在但损坏 = 硬错误。

Rationale: 项目已有一套成熟的「$PIGO_HOME + 缺失非错/损坏报错」范式，复用它让读者零学习成本，也满足验收条件里的「不存在时视为空清单」「损坏文件容错」。

Decision lockfile 按包名 map 存储，List/Save 排序输出

Ambiguity: Issue 只要求「记录每个已安装包」，未规定内部结构与顺序。

Choice: Packages map[string]InstalledPackage（按名查找 O(1)，卸载/更新友好）；Save 用 MarshalIndent 且 map 天然由 encoding/json 按 key 排序，List() 显式按 name 排序。

Rationale: uninstall/update 都按包名操作，map 是最自然的主键结构；排序输出保证 lockfile diff 稳定、list 输出确定。

Decision skills 目录不挂在 $PIGO_HOME 下

Ambiguity: PRD 映射表把 skill 落到「skills 目录」，但 pigo 的 skill 加载器读的是 ~/.agents/skills（PIGO_SKILLS_DIR 可覆盖），而非 $PIGO_HOME/skills。

Choice: SkillsDir() 遵循现有 skill 加载器的路径约定，是唯一不挂在 $PIGO_HOME 下的类型；plugins/commands/themes 都在 $PIGO_HOME 下。

Rationale: 分发的唯一目的是让文件被现有发现机制加载。若装到 $PIGO_HOME/skills，skill 加载器根本不会去读，安装等于无效。

Deviations

Deviation lockfile 字段用 types[] 而非 PRD 文字里的 type[]

Spec: PRD 正文写「type[]」。

Implemented: JSON 字段名用 types（Go 字段 Types []PackageType）。

Why: 复数名更符合 Go/JSON 惯例（切片用复数），语义完全一致。这是纯命名，不影响契约。若后续 Issue 依赖精确字段名，以本实现为准。

Tradeoffs

Tradeoff lockfile 记录绝对文件路径

Alternatives: (A) 记录相对 $PIGO_HOME 的相对路径；(B) 记录落地文件的绝对路径。

Pros/Cons: 相对路径更可移植（$PIGO_HOME 搬家不失效），但 uninstall 需要重新拼接根路径、且 skills 不在 $PIGO_HOME 下、相对基准不统一；绝对路径 uninstall 直接删、逻辑最简，代价是 $PIGO_HOME 迁移后失效。

Winner: 绝对路径。安装是本机行为、$PIGO_HOME 迁移极少见，而 skills/commands/plugins 分属不同根，统一绝对路径避免「相对谁」的歧义，uninstall 实现最直接。
