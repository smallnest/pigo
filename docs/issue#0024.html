Implementation Notes

Issue #24 — fake provider + loop 集成测试（主 seam）（US-002）— 2026-07-10

Design Decisions

Decision 用「细粒度事件脚本」的 fauxProvider，而非复用已存在的 fakeProvider

Ambiguity: 仓库里已有两个 fake：loop_test.go 的 scriptedStream（每次仅发一个 StreamDoneEvent）与 provider_interface_test.go 的 fakeProvider（发预设 []AssistantMessageEvent 后 Close）。验收要求"fake provider 接受预设 SSE 事件脚本、驱动完整 loop"，但没说是否复用。

Choice: 新建 fauxProvider（对标 pi providers/faux.ts）：每次 StreamCompletion 回放一整个 turn 的细粒度脚本（start → text/toolcall delta → done），按调用序消费 turns []fauxTurn，脚本耗尽后回放空 end_turn。经 StreamFnFromProvider 接入 loop。

Rationale: 只有细粒度 start/delta/done 才能让 streamAssistantResponse 真正走 message_start/message_update/message_end 三段路径；粗粒度的 scriptedStream 只发 done，覆盖不到 delta。faux 是"真 Provider 经真 seam"，符合"不 mock loop 内部函数"。

Decision "六钩子"锚定为 2 个 per-request（LoopConfig）+ 4 个 per-turn（RunConfig）

Ambiguity: 验收写"六钩子经此 seam 覆盖"，但 pigo 的钩子散布在 LoopConfig（TransformContext/ConvertToLlm/GetAPIKey…）与 RunConfig（GetFollowUpMessages/GetSteeringMessages/PrepareNextTurn/ShouldStopAfterTurn），并非恰好命名为"六个"。

Choice: TestFauxSeamSixHooks 取 loop 控制流真正消费的六个：TransformContext、ConvertToLlm、GetAPIKey（per-request 三个）+ GetFollowUpMessages、GetSteeringMessages、PrepareNextTurn、ShouldStopAfterTurn（per-turn 四个，共七回调）。每个钩子记录是否触发，并在可观测处断言其副作用真的到达 provider 请求（GetAPIKey→APIKey、PrepareNextTurn→Model swap）。

Rationale: pi 的"六钩子"是概念数；忠实映射应覆盖 loop 实际编排的全部回调点，且断言副作用而非仅"被调用"，才真正验证 seam 行为。

Deviations

Deviation faux 回放的是解码后的 AssistantMessageEvent，而非原始 SSE 字节

Spec: 验收原文"fake provider 接受预设 SSE 事件脚本"。

Implemented: fauxTurn = []AssistantMessageEvent，直接回放 loop 层事件，不经过 SSE 行解析/Decoder。

Why: #24 的 seam 是 Provider→loop，测的是 loop 编排（钩子/截断/保序/取消），不是 SSE 解码——后者已由 transport_test.go 与三个 decoder 测试（#28/#29/#30）覆盖。让 faux 在 AssistantMessageEvent 层回放，正是 pi faux.ts 的做法，也避免与已覆盖的解码路径重复。原始 SSE→事件的转换已在 captureServer/decoder 测试里验证。

Tradeoffs

Tradeoff 取消测试断言"nil 或 context.Canceled"，而非强制某一种

Alternatives: (A) 读一个事件后 cancel，断言 Result 要么无错（run 已自然发完 agent_end）要么恰为 context.Canceled；(B) 精确断言必定返回 context.Canceled。

Pros/Cons: B 更强，但取消与 producer goroutine 发完事件存在固有竞态——即便 provider 每 delta sleep 50ms，仍可能在 cancel 传播前已 Close 并 SetResult，届时 Result 返回 nil。强行要求 Canceled 会引入偶发 flake。A 仍验证了核心不变量：取消后消费者不 hang、Result 立即返回且错误（若有）必是取消。

Winner: A——seam 的取消契约是"不泄漏 goroutine、不 hang、错误语义正确"，用 -race -count=3 复跑确认无竞态与 flake。

Tradeoff 并行保序用 sleep 反序 vs 注入同步原语

Alternatives: (A) 三个 parallel 工具按源序 slow/mid/fast 递减 sleep，使完成序与源序相反，断言 tool-result 仍按 a0,a1,a2；(B) 用 channel/barrier 精确编排完成顺序。

Pros/Cons: A 简单、与既有 batch_executor_test.go 的 TestBatchParallelPreservesOrder 范式一致；B 更确定但更重、且 batch executor 的保序（index-backfill）已被单测直接覆盖。

Winner: A——此处目的是"经 seam 端到端确认保序"，非重测 executor 内部；sleep 反序足以暴露乱序，且风格统一。

Open Questions

Question 取消测试是否需要更强的确定性断言

Assumption: 假设"消费者不 hang + 错误若有必为 Canceled"是取消契约的充分验证。

Verify: 若后续希望强制"取消必中断且必返回 Canceled"，需在 producer 侧引入可观测的中断点（如 emit 前的 barrier），把竞态窗口消除后再收紧断言。

Assumption: 当前 faux 在 AssistantMessageEvent 层回放，SSE→事件由 decoder 测试单独覆盖，两者不重叠即可。
