Implementation Notes

Issue #33 — Bedrock/OpenRouter/Ollama provider（US-013）— 2026-07-10

Design Decisions

Decision 两个共享 driver 覆盖三个 provider，而非各写一份

Ambiguity: 验收要求"复用共享 driver / OpenRouter 作为通用 OpenAI 兼容层样例"，但未规定 driver 的边界如何切分。

Choice: 抽出 openAICompatDriver（POST {baseURL}/chat/completions + OpenAIDecoder）与 anthropicCompatDriver（POST Messages wire + AnthropicDecoder）。OpenRouter/Ollama 是前者的两个实例（仅 baseURL/auth 不同），Bedrock 走后者。

Rationale: Chat Completions 与 Messages 是两种 wire 契约；按 wire 而非按厂商切分，让"新增一个 OpenAI 兼容网关"退化为一次构造调用（Groq/together/本地 server 皆可），真正兑现"通用兼容层"。

Decision 这是仓库中第一批具体 Provider——此前只有 Decoder 和测试 fakeProvider

Ambiguity: 验收写"各实现 Decoder"，但 anthropic/openai/gemini 的 Decoder 早已存在（#28/#29/#30），从未有 Provider 把 Decoder 经 StreamRequest 串起来。

Choice: 把验收理解为"实现首批 Provider"：StreamCompletion 构建 *http.Request（base URL + auth + JSON body）后交给 StreamRequest(TransportConfig{NewRequest, Decoder})，复用既有 Decoder 而非重写。

Rationale: Decoder 已完备且测试充分；#33 的真正缺口是"把 Decoder 接到真实 HTTP/SSE 上的 Provider"。复用 Decoder 即满足"各实现 Decoder"的意图。

Decision 缺 API key 是唯一的早返回错误，其余失败一律走流内终态

Ambiguity: 验收未提失败模型，但 FR-13 双失败模型是全局约束。

Choice: requiresAuth 的 provider（OpenRouter/Bedrock）在 key 为空时早返回 "<provider>: missing API key"；Ollama 本地无需 key。请求构建后所有运行时失败由 StreamRequest 落到终态 StreamErrorEvent。

Rationale: 与 transport/provider_interface 的双失败模型对齐；错误仅按 provider 名引用，绝不泄漏 key 值（US-012/US-026）。

Deviations

Deviation Bedrock 走标准 Anthropic Messages 端点，未实现 AWS SigV4 / invoke-with-response-stream

Spec: 验收要求"Bedrock 实现 Decoder，复用共享 driver"与"每个 provider 最小流式单测"。

Implemented: Bedrock 复用 AnthropicDecoder，以 Authorization: Bearer（Bedrock API key）POST Messages wire。真实 Bedrock 的 /model/{id}/invoke-with-response-stream 路径与 SigV4 签名以 pathFor 钩子和"由调用方 HTTP client 分层签名"的方式预留，未在本 issue 内落地。

Why: 验收明确是"最小流式单测 + 复用共享 driver"，Anthropic-on-Bedrock 本就说 Messages 协议，Decoder 可原样复用；SigV4 与区域端点属部署/凭证细节，超出本 issue 范围，留待接入真实 Bedrock 流量时补齐。

Tradeoffs

Tradeoff 内容降维为纯文本 vs 保留完整多模态块

Alternatives: (A) contentToText 把 content list 压平为文本（tool_call/thinking 走各自字段）；(B) 完整映射 image/thinking 等所有块到各 wire。

Pros/Cons: A 是所有 OpenAI 兼容网关都接受的最小公分母、实现简单；B 更完备但各网关对多模态支持参差，易在边缘网关上出错。Anthropic 侧已保留 text/tool_use/tool_result 结构化块。

Winner: A（OpenAI 侧）——先保证跨网关可用；多模态透传待具体 provider 需要时按需扩展。

Tradeoff captureServer 断言 wire 形状 vs 仅断言事件流

Alternatives: (A) 新增 captureServer 记录 path/headers/body，断言 driver 真正 POST 出的报文形状；(B) 仅复用 sseServer 断言解码后的事件。

Pros/Cons: A 能验证 auth header、system 顶层字段、stream flag、max_tokens 等 driver 独有职责；B 只覆盖 Decoder（已被 #29/#30 测过），对 Provider 层几乎零增量。

Winner: A——#33 的新增职责是"构建请求"，测试必须落在请求形状上才有意义。
