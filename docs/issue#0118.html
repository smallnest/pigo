Issue #118 — 会话切点计算 FindCutPoint — 2026-07-17

Design Decisions

Decision 适配到扁平 message 列表而非 pi 的 SessionTreeEntry

Ambiguity: pi 的 findCutPoint 操作 SessionTreeEntry[]（含 branch_summary/model_change/compaction 等非消息条目），并带 startIndex/endIndex 参数；pigo 当前是扁平 []agentcore.Message。

Choice: 签名简化为 FindCutPoint(msgs, keepRecentTokens)，去掉 start/end 与非消息条目分支。

Rationale: pigo 尚无会话树（#121 才引入 id/parentId），当前只有线性消息。保留 pi 的核心算法（反向累加 + 最近合法切点 + split-turn 判定），省去暂不存在的条目类型；等 #121 落地后如需可再引入范围参数。

Decision 合法切点 = user | assistant，其余（含 toolResult）不可切

Ambiguity: pi 的 findValidCutPoints 把 bashExecution/custom/branchSummary 等也算合法切点。

Choice: pigo 只有 user/assistant/toolResult 三种消息角色，切点判定为 user||assistant，toolResult 明确排除。

Rationale: 严格保证 toolCall 与其 toolResult 不被拆开——这是 AC 的硬要求。其他角色 pigo 目前不存在，default 分支自然排除。

Deviations

Deviation 省略 pi 的 cutIndex 回退循环

Spec said: pi 在选定 cutIndex 后有一个 while (cutIndex > startIndex) 循环，越过前面的非 message / 非 compaction 条目把 cutIndex 往前挪。

Implemented: pigo 未实现该循环。

Why: 该循环的作用是跳过 SessionTreeEntry 中夹杂的元数据条目（model_change 等）。pigo 扁平列表中每个元素都是消息，循环恒为 no-op，实现它反而引入无意义代码。

Tradeoffs

Tradeoff 无合法切点时保留全部而非报错

Alternatives: (A) 返回 FirstKeptIndex=0 表示"整段保留"；(B) 返回错误要求调用方处理。

Chosen: A，与 pi 一致（firstKeptEntryIndex: startIndex）。

Why: 只有 toolResult 的消息序列在正常会话中几乎不会出现；保守地"不切"比让上层处理错误更简单，且压缩本就是尽力而为。
