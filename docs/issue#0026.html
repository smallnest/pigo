Implementation Notes

Issue #26 — 共享 transport driver（SSE+retry+双看门狗）— 2026-07-10

Design Decisions

Decision Decoder 接口用 []StreamEvent，StreamEvent 别名 AssistantMessageEvent

Ambiguity: 验收写 Decode(payload []byte) ([]StreamEvent, error)，但仓库中 StreamEvent 未独立定义。

Choice: type StreamEvent = AssistantMessageEvent（类型别名），Decoder 返回 []StreamEvent。

Rationale: 延续 #25 决策——复用 AssistantMessageEvent，不造平行事件类型；别名让验收签名字面成立又零成本融入现有流。

Decision 连接前置于 goroutine 外，dual failure 边界落在首次 connect

Ambiguity: "cannot build the stream" 的边界在哪一步？

Choice: StreamRequest 同步完成首次 connect（含 retry），失败即 return error；成功后才起 pump goroutine，此后所有失败走终态 StreamErrorEvent。

Rationale: 与 FR-13/#25 双失败模型精确对齐——"建流前"失败是唯一的返回错误，一旦流开始则错误骑在流上。

Decision content-stall 看门狗 = idle×1.2

Ambiguity: 验收给了 idle 默认 5m 与可调环境变量，但 stall 只写 "idle×1.2"。

Choice: stall = idle × 1.2，任意字节 reset idle，成功 flush 一个事件 reset stall。

Rationale: idle 防"完全无字节"，stall 防"有 keep-alive 但无实质内容进展"；1.2 倍留出慢速但推进的流的余量。

Deviations

Deviation connect-only retry，不做流内重连重放

Spec: "connect-only 重连绝不重放已消费流"。

Implemented: retry 仅发生在 connect 阶段（首个响应头到达前）；一旦 pump 开始消费流，读错误直接终态化，不尝试重连。

Why: "绝不重放已消费流"的最强保证就是流开始后完全不重连；每次 connect 用全新 NewRequest 构造，天然不重放 POST body。这比"重连但跳过已消费部分"更安全且更简单。

Tradeoffs

Tradeoff 看门狗用 timer + reader goroutine + done channel，而非 SetReadDeadline

Alternatives: (A) 独立 reader goroutine + select 双 timer + done 退出信号；(B) net.Conn.SetReadDeadline 直接在 body 上设超时。

Pros/Cons: A 与传输层无关（对任意 io.Reader 生效，httptest 可测），但需 done channel 防 goroutine 泄漏；B 更省 goroutine 但要拿到底层 conn、且无法区分 idle vs stall 两种超时。

Winner: A——双看门狗语义要求两个独立计时器，SetReadDeadline 只能表达单一超时；done channel 已在 pump 返回时 close，reader 不会泄漏（race 测试通过）。
