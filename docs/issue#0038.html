Implementation Notes

Issue #38 — grep/find/ls 搜索工具（US-019）— 2026-07-10

Design Decisions

Decision 自实现最小化 .gitignore 匹配器

Ambiguity: 验收要"尊重 .gitignore"，未规定支持哪些语法子集。

Choice: 实现常见子集——注释/空行、前导 / 锚定、尾随 / 仅目录、! 反选、按 base name 或路径 filepath.Match；非锚定 pattern 匹配任意路径段（目录被忽略则其下全部隐藏）。后规则覆盖前规则。

Rationale: 覆盖绝大多数真实 .gitignore；坚持零外部依赖（不引入 go-gitignore 库）；显式记录不支持 ** 跨段与嵌套 .gitignore。

Decision 三工具共享 resolveWithin 边界策略，均为 parallel

Ambiguity: 验收未提路径边界与并发模式。

Choice: 复用与 ReadTool 一致的 Root + filepath.Rel 越界拒绝；grep/find/ls 只读 → ToolExecutionParallel。

Rationale: 与既有工具（read/write/edit）一致的边界与并发语义，减少认知负担；只读工具并发安全可提升批量效率。

Decision 结果上限 searchMaxResults=1000

Ambiguity: 验收未提结果规模上限。

Choice: grep 匹配、find 命中均在达到 1000 时经 filepath.SkipAll 提前终止，grep 附 [truncated] 提示。

Rationale: 防止宽泛查询淹没模型上下文；与 read 工具的 2000 行上限同理。

Deviations

Deviation 仅读取 Root 级单个 .gitignore，不支持嵌套/全局 gitignore

Spec: "尊重 .gitignore"。

Implemented: 只加载 root/.gitignore；不递归读取子目录 .gitignore，不读全局 core.excludesfile，不解析 ** 跨段。始终额外硬跳过 .git/。

Why: 单文件覆盖工具级搜索的主要场景；完整 gitignore 语义（嵌套、全局、**）复杂度高，对 MVP 属过度设计，可后续增强。

Tradeoffs

Tradeoff 三个独立工具 vs 单一多模式工具

Alternatives: (A) grep/find/ls 各自独立工具；(B) 单 search 工具带 mode 参数。

Pros/Cons: A schema 清晰、模型选型直观、与 pi 工具粒度一致；B 工具数少但参数条件耦合、schema 复杂。

Winner: A——独立工具边界清晰，共享 helper（resolveWithin/gitignore）已消除重复。

Open Questions

Question 是否需要嵌套/全局 .gitignore 支持

Assumption: Root 级单一 .gitignore 足够。

Verify: 若仓库大量使用子目录 .gitignore 或全局 excludesfile，搜索结果可能包含本应忽略的文件；是否需要升级匹配器？

Question grep 是否需要二进制文件检测

Assumption: 逐行扫描所有非忽略文件。

Verify: 二进制文件（图片/可执行）会被逐行扫描且可能产生乱码匹配；是否需要 NUL 字节嗅探跳过二进制？
