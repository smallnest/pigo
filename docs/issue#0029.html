Implementation Notes

Issue #29 — OpenAI 兼容 provider decoder（US-009）— 2026-07-10

Design Decisions

Decision 自定义 base URL 归为 transport 关注点，decoder 完全 base-URL 无关

Ambiguity: 验收要求"支持自定义 base URL"，但未规定该配置落在 decoder 还是 transport。

Choice: OpenAIDecoder 只负责解析 wire 格式，不含任何 URL 概念；base URL 由 transport 的 NewRequest（构建 *http.Request）决定。同一个 decoder 因此可复用于所有 OpenAI 兼容网关（OpenRouter、Groq、together、本地 server）。

Rationale: Chat Completions 的 SSE chunk 格式与端点无关；把 URL 选择留在 transport 层符合既有 Decoder 契约的职责边界，也让"兼容 provider"天然成立而无需 decoder 改动。

Decision tool_calls 按 delta index 归集，text 累积到单一 builder

Ambiguity: 验收只要求"解析 tool_calls 增量与文本增量"，未规定跨 chunk 的累积组织方式。

Choice: 文本用单个 strings.Builder（OpenAI 单条 choice 的 content 顺序到达）；tool_calls 用 map[int]*openaiToolCall 按 delta index 归集，id/name 首片到达即记，arguments 逐片拼接；单独记录 toolOrder 保证首见顺序。

Rationale: 复刻 anthropic.go 的 index 归集范式，天然处理并行 tool_calls 的交错分片；terminal 消息里 text 块在前、tool_call 块按 index 升序，顺序稳定可预期。

Decision finish_reason 映射表：length→length、tool_calls/function_call→tool_use、其余→end_turn

Ambiguity: 验收要求"stopReason 映射正确"，但 OpenAI 还有 content_filter、legacy function_call 等值未列出。

Choice: length→StopReasonLength、tool_calls 与 legacy function_call→StopReasonToolUse、stop→StopReasonEndTurn，未知值（含 content_filter）兜底 end_turn。

Rationale: 与 mapAnthropicStopReason 对齐；legacy function_call 语义等同工具调用；未知值兜底为非 error 的自然停止，避免把新 wire 值误判为失败。

Deviations

Deviation feedSSE 测试辅助新增 [DONE] 跳过，而非改 decoder

Spec: OpenAI 流以 data: [DONE] 作为终止哨兵；dual failure model 要求 decoder 不因畸形 payload panic。

Implemented: [DONE] 是 transport 层终止符（transport 的 SSE 解析已将其丢弃，不会喂给 Decode）。测试辅助 feedSSE 原先手工切分不识别 [DONE]，故在其中加一行跳过，与 transport 行为对齐；OpenAIDecoder 本身不需要、也不含对 [DONE] 的特判。

Why: [DONE] 归属 transport 关注点；若在 decoder 里特判会把 transport 职责泄漏进 decoder。fixture 里保留 [DONE] 行是为了让录制流结构忠实于真实 wire，测试辅助对齐 transport 的丢弃行为即可。

Tradeoffs

Tradeoff 内联 error 对象表达为 Decode 返回 error

Alternatives: (A) 部分网关会在 SSE chunk 里内联 {"error":{...}}，decoder 检测到即返回 Go error，由 transport flush() 转成终态 StreamErrorEvent；(B) 忽略内联 error，仅依赖 HTTP 状态码/transport 层错误。

Pros/Cons: A 覆盖 OpenRouter/Groq 等网关在 200 流中途报错的真实场景，与 anthropic.go 的 error 事件处理一致；B 更简单但会漏掉流内错误，让消费方拿到静默截断的空响应。

Winner: A——兼容 provider 的错误常以流内 error 对象出现；统一走 transport 的单一失败落点（terminal StreamErrorEvent），语义与 Anthropic decoder 对齐。

Tradeoff 复用 feedSSE/sseServer 录制 fixture vs 新建独立测试骨架

Alternatives: (A) 复用 anthropic_test.go 的 feedSSE/eventKinds 与 transport_test.go 的 sseServer/newReqFn，一份 fixture 同时跑纯 decoder 单测与 transport 全链路；(B) 为 OpenAI 单独写一套测试驱动。

Pros/Cons: A 零重复、与 Anthropic 测试范式一致、天然验证 [DONE] 经 transport 的处理；B 隔离性略好但重复造轮子、且要另建 SSE server。

Winner: A——两个 decoder 共享 transport 契约，共享测试骨架保证行为一致性，只需为 [DONE] 在共享辅助里加一行跳过。

Open Questions

Question OpenAI usage 是否需要 stream_options:{include_usage:true} 才下发

Assumption: 假设请求已带 stream_options.include_usage=true，故流末会出现 usage-only chunk（choices:[] + usage）；decoder 直接覆盖式记录 prompt/completion tokens。

Verify: 该 flag 属请求构建（transport/provider 层）职责，需在 provider 实现里确保开启；否则某些网关流不含 usage，最终 Usage 为 nil。接入真实流量后核对 token 数。

Question 是否需要保留 OpenAI o系列的 reasoning/thinking 增量

Assumption: 当前 decoder 只解析 content 与 tool_calls，不处理 o1/o3 等模型可能下发的 reasoning 摘要字段。

Verify: 若后续需支持 OpenAI 兼容端点的思考内容展示，需扩展 openaiChunk 与 delta 解析承载 reasoning 字段并映射到 ThinkingContent。
