Implementation Notes

Issue #178 — 补齐 pi 的 --approve / --no-skills 命令行参数 — 2026-07-20

Design Decisions

Decision Map --approve onto session trust, not a persisted decision

Ambiguity: pi's --approve/-a means "trust project-local files for this run". The trust store offers two grants: SetSessionTrust (in-memory, per-run) and SetDecision (persisted to trust.json).

Choice: --approve calls mgr.SetSessionTrust(cwd), granting trust only for the current process.

Rationale: "for this run" is per-invocation. Persisting would silently record a permanent trust decision the user never explicitly confirmed via the menu, changing behavior of future launches. A verifying test (TestEstablishTrustApprove) asserts trust does NOT survive a store reload.

Decision Introduce establishTrust as the single trust-entry seam

Ambiguity: The --approve branch could have been inlined into runInteractive as an if/else around ensureTrustPrompt.

Choice: Extracted establishTrust(out, in, mgr, cwd, approve) in trust.go, which either grants session trust up front or defers to ensureTrustPrompt.

Rationale: Keeps the trust-establishment policy beside the other trust helpers (testable in isolation), and keeps runInteractive a straight wiring function. Two focused unit tests cover both branches without spinning up a REPL.

Decision Gate skill discovery via a noSkills parameter on buildSlashRegistry

Ambiguity: pi's --no-skills disables skill discovery but still loads explicit --skill paths. pigo has no --skill path flag yet, only directory discovery under ~/.agents/skills.

Choice: Threaded a noSkills bool into buildSlashRegistry that skips only the loadSkillCommands block; user command templates under ~/.pigo/commands still load.

Rationale: Matches pi's semantics as closely as the current feature set allows — discovery off, everything else unchanged. When an explicit --skill path flag is added later, it slots in beside this gate untouched.

Deviations

Deviation Flags are REPL-only; silently inert in headless (-p) mode

Spec: pi applies --approve/--no-skills as global CLI flags.

Implemented: Both flags parse in all modes but only affect the interactive REPL path. The headless -p path has no trust gating hook and does no slash/skill resolution, so they are no-ops there.

Why: Trust gating and skill slash-commands only exist in the REPL. Headless already runs side-effect tools ungated and has no /skill-name concept, so there is nothing for the flags to change. Wiring them into headless would require inventing behavior that does not exist. Documented as a known limitation (see Open Questions).

Tradeoffs

Tradeoff Silently ignore vs. warn on the two flags in headless mode

Alternatives: (a) silently accept the flags in headless mode; (b) print a "flag has no effect in headless mode" warning to stderr; (c) error out.

Chosen: (a) silent acceptance.

Pros/cons: (c) would break scripts that pass the flag defensively across modes. (b) is the safest but adds output that could contaminate stream-json/CI pipelines. (a) keeps the CLI surface uniform and pipeline-clean; the cost is a user could believe the flag took effect in -p mode when it did not. Review flagged this as the single low-severity finding; deferred to a docs/warning follow-up rather than blocking this change.
