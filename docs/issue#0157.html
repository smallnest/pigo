Implementation Notes

Issue #157 — 识别 pi 包类型 (classify pi package types) — 2026-07-20

Design Decisions

Decision 显式 pi 元数据 + 结构化信号 双通道识别

Ambiguity: Issue/PRD 给了映射表（extension via bin/pi.extension；skill via SKILL.md/pi.skill；prompt via pi.prompt/commands/*.md；theme via pi.theme），但未规定当二者冲突或包"少声明"时如何取舍。

Choice: 两路并集：先读 package.json 的 pi 块（支持单 type、types 列表、以及 per-capability 键），再叠加结构化回退（bin→extension、SKILL.md→skill、commands/→prompt）。结果是一个去重后的类型集合。

Rationale: pi 生态里存在 combined "extensionskill" 包，一个包可同时是多类型；并集能覆盖"声明齐全"和"结构清晰但少声明"两种真实情况，稳健且不误判。

Decision 无法识别时报错而非猜测

Ambiguity: 遇到普通 npm 包（如 lodash）应如何处置。

Choice: 空集合直接返回 unrecognized pi package: no known pi metadata。

Rationale: pigo install 需要对非 pi 包快速失败并给出明确信息，避免把任意 npm 包错误地分发到 pigo 目录。

Deviations

Deviation Classify 同时返回 name/version

Spec: US-004 只要求"返回类型集合"。

Implemented: 签名为 Classify(pkgDir) (name, version string, types []PackageType, err error)，顺带回传从同一份 package.json 读到的名字与版本。

Why: 后续 issue（#158-#162 分发与写 lockfile）都需要 name/version，且它们和类型来自同一次解析，一并返回避免重复读取 package.json。生产行为不变。

Tradeoffs

Tradeoff 未知 pi.type 字符串：忽略 vs 报错

Alternatives: (A) 遇到无法识别的 type 字符串（如 "widget"）直接报错；(B) 忽略该字符串，仅当最终集合为空才报错。

Pros/Cons: A 更严格但对 pi 生态未来新增类型不向前兼容；B 宽容、向前兼容，但可能"静默丢弃"一个真实类型。

Winner: 选 B。若包同时有可识别信号则照常分类；若只有未知 type 则仍因空集合报错，安全兜底。未来 pigo 支持新类型时只需扩展 normalizeType。

Open Questions

Question pi 元数据字段名的确切约定

Assumption: 顶层 pi 对象含 type(string)/types([]string) 以及 per-capability 键 extension/skill/prompt/theme。这是依据 pi 包惯例的推断。

Verify: 若 pi.dev 官方约定的字段名不同，需调整 piMeta 结构；结构化回退（bin/SKILL.md/commands）不受影响，可作为兜底。
