Implementation Notes

Issue #234 — Add model-name → provider inference table — 2026-07-23

Design Decisions

Decision Ordered prefix table instead of a map

Ambiguity: The issue lists name→provider pairs but not the data structure.

Choice: A top-to-bottom ordered slice modelPrefixProvider, first-match-wins.

Rationale: Prefix matching is inherently order-sensitive (a longer prefix must be tried before a shorter overlapping one). A slice makes that ordering explicit and reviewable; a map would hide it.

Decision Routed provider/model ids are not inferred

Ambiguity: The issue only names bare model families; it is silent on OpenRouter-style routed ids like openai/gpt-4o or ollama/llama3.3.

Choice: Any id containing / returns ok=false.

Rationale: Those ids are already handled by the preset catalog and the ollama/ / nvidia/ prefix paths in resolveProvider. Inferring from the leading segment would wrongly route openai/gpt-4o to the direct OpenAI provider instead of OpenRouter, silently changing behavior.

Decision qwen-* → dashscope, glm-* → zai

Ambiguity: Qwen and GLM are served by multiple built-ins (dashscope/together/openrouter; zai/zai-coding-cn/cerebras).

Choice: Map to the model family's first-party direct provider (Alibaba DashScope, Z.AI) per the issue table.

Rationale: The issue explicitly names these targets; the first-party provider is the least surprising default for a bare family name.

Deviations

Deviation Added o4-* to the OpenAI patterns

Spec: Listed gpt-* / o1-* / o3-* → openai.

Implemented: Also mapped o4-* → openai.

Why: The o-series naming is sequential; omitting o4 would be an obvious gap and matches the issue's clear intent. Low risk — the prefix is unambiguously OpenAI.

Tradeoffs

Tradeoff Conservative prefix matching over broad heuristics

Alternatives: (a) substring/regex matching to catch more names; (b) strict prefix-only for names that map to exactly one provider.

Chosen: (b). Ambiguous families (llama-*, qwq-*, gemma-*, mixtral-*) deliberately return ok=false.

Why it won: A false inference silently sends traffic to the wrong gateway with the wrong API key — worse than falling through to the existing OpenRouter default. Precision beats recall here.
