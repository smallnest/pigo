Implementation Notes

Issue #116 — 抽取 internal/jsonrpc stdio JSON-RPC 2.0 层 — 2026-07-17

Design Decisions

Decision 面向 client 而非通用双向 peer

Ambiguity: Issue 只要求"JSON-RPC 2.0 request/response/notification 编解码"与"子进程 stdio transport client"，未规定是否要处理 server→client 的反向请求。

Choice: 只实现 client 侧：发 request/notification、收 response。readLoop 遇到无法解析为 response 的行（可能是 server 发起的 request/notification）直接跳过。

Rationale: 三个下游消费者（MCP #130/#131、插件 #132/#133、进程隔离 #135）当前都只需 client 语义。等 MCP 需要 sampling 等反向调用时再扩展，避免过早引入双向派发的复杂度。

Decision id 用 n:/s: 前缀区分作 map key

Ambiguity: 规范允许 id 为 number 或 string，但 pending map 需要一个可比较的键，且数字 id 与字符串 id 不能碰撞。

Choice: ID.String() 对数字返回 n:123、对字符串返回 s:abc。本 client 只生成数字 id，但对端回显字符串 id 时仍能正确关联。

Rationale: 前缀消除跨类型碰撞（数字 1 与字符串 "1" 不会撞键），实现简单且零依赖。

Decision 读循环上限 16MB

Ambiguity: Issue 未规定单条消息大小上限，而 bufio.Scanner 默认行上限仅 64KB。

Choice: scanner.Buffer(64KB, 16MB)。

Rationale: MCP tools/list 返回的工具 schema 可能很大，64KB 会导致 bufio.ErrTooLong 静默截断。16MB 给足余量又能防御失控输入。

Deviations

None — 实现满足 Issue 全部验收条件（编解码、子进程 stdio client、超时、优雅关闭、并发 id 关联、测试通过）。

Tradeoffs

Tradeoff 换行分隔 JSON vs LSP 式 Content-Length 帧

Alternatives: (A) 每条消息一行、以 \n 分隔；(B) LSP/DAP 的 Content-Length 头 + body 帧协议。

Chosen: A（换行分隔）。

Why: MCP 的 stdio transport 官方即用换行分隔，插件与进程隔离子 agent 也由本项目自定义、可约定单行输出。方案 A 用 bufio.Scanner 一行搞定，比手写 Content-Length 帧解析简单得多。代价是消息体内不能含裸换行——JSON 编码器默认不产生，已满足。

Tradeoff 测试用 TestMain 自举 mock server vs 独立 helper 二进制

Alternatives: (A) 用 TestMain + 环境变量让测试二进制自身充当 echo/silent server；(B) 单独编译一个 mock server 可执行文件。

Chosen: A。

Why: 无需额外构建步骤或 go build 临时二进制，os.Executable() 直接复用测试进程，是 Go 测试子进程 transport 的标准手法。缺点是 TestMain 里多了模式分支，可读性略降，但集中在测试文件内可接受。

Tradeoff Close 的 5s grace 定时器（review 追加）

Alternatives: (A) 仅等 cmd.Wait() 或 reader done；(B) 额外加 grace 定时器兜底。

Chosen: B（review 阶段补强）。

Why: 原实现若子进程既忽略 stdin EOF 又保持 stdout 打开（挂死的插件/server），Close 会永久阻塞。加 closeGrace = 5s 定时器后超时即 Process.Kill()，保证 Close 有界返回。代价是引入一个常量与 time 依赖，值得。
