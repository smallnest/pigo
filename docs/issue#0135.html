Implementation Notes

Issue #135 — 子 agent process 隔离模式 (US-019) — 2026-07-19

Design Decisions

Decision Isolation field on SubAgentSpec, zero-value = goroutine

Ambiguity: The spec said add Isolation: goroutine | process but did not say how the default preserves existing behavior.

Choice: A new SubAgentIsolation enum (SubAgentIsolationGoroutine = iota default, SubAgentIsolationProcess). Execute dispatches on it; the goroutine branch is the original code verbatim.

Rationale: The zero value is goroutine, so every existing SubAgentSpec (built by Skill.SubAgentSpec and tests) keeps running in-process with no behavior change - satisfying "默认 goroutine 模式无行为变化".

Decision Subprocess = the pigo binary itself over stdio JSON-RPC, reusing internal/jsonrpc

Ambiguity: The spec said "pigo 以子进程启动自身 headless/rpc...复用 internal/jsonrpc" without specifying the wire shape.

Choice: The parent (defaultProcessCall) launches os.Executable() with --subagent-rpc via jsonrpc.NewClient, sends one subagent/run request, and decodes the {text} response. The subprocess side is a new pigo --subagent-rpc mode that reads JSON-RPC requests, resolves the provider, runs RunSubAgentOnce, and writes responses.

Rationale: Spawning pigo itself (not a separate helper) matches "启动自身"; internal/jsonrpc.Client already handles exec, line-delimited framing, id correlation, EOF/crash detection, and Close-with-grace-kill, so the parent transport is reuse, not reinvention.

Decision Serializable provider config (Model/BaseURL/Protocol) + env-inherited credentials

Ambiguity: Goroutine mode passes a RunConfig with in-process Stream/GetAPIKey funcs; the spec did not say how a subprocess gets its provider.

Choice: SubAgentProcessConfig carries Model/BaseURL/Protocol (strings). The subprocess calls resolveProvider (the same resolution the CLI uses) and inherits the parent environment, so API keys resolve via the same env vars.

Rationale: Funcs cannot cross a process boundary. Forwarding the model id + endpoint overrides lets the subprocess reconstruct the identical provider; env inheritance is the natural credential channel and needs no key marshaling over the wire.

Decision Crash and failed-run both surface as a tool error

Ambiguity: The spec required "子进程崩溃被父捕获为工具错误" but did not define failed-run handling.

Choice: A subprocess crash (exit without responding) is caught by jsonrpc.Client's readLoop EOF → failAll → Call returns an error → executeProcess wraps it as a tool error. A failed child run (StopReasonError/Aborted) is returned by RunSubAgentOnce as an error → the subprocess emits it as a JSON-RPC error → same parent path. The parent loop is never affected.

Rationale: Treating both as tool errors matches goroutine mode's "failed run → IsError" contract and gives the parent model a clear signal the delegation failed, whether the child crashed or merely errored.

Decision Injectable processCall seam for testing

Ambiguity: The spec's test gate is go test ./internal/runtime/..., but the real transport needs a subprocess + provider.

Choice: SubAgentTool has an unexported processCall field; when nil, Execute uses defaultProcessCall (real transport). Tests in package runtime inject a fake to verify params shaping and crash-as-error without a binary. The real transport is covered separately by compiling a tiny JSON-RPC helper binary (mirroring the plugin tests).

Rationale: Fast, hermetic unit tests for the process-mode logic plus a real exec+pipe+JSON-RPC round-trip (and crash) test - without needing a live provider. The seam is set only at construction (safe by convention).

Deviations

Deviation Process mode does not stream partial updates

Spec: Goroutine mode forwards streamed child text deltas via onUpdate.

Implemented: Process mode returns only the child's final text; onUpdate is intentionally not forwarded (documented at the dispatch site).

Why: The JSON-RPC protocol is a single request/response carrying the final result; streaming deltas would require server->client notifications, which is out of scope for #135. The parent still receives the complete result when the child settles.

Deviation Process-mode tools are builtins-only, rebuilt by name

Spec: SubAgentSpec.Tools is an arbitrary []agentcore.AgentTool.

Implemented: Process mode forwards only the tool names; the subprocess rebuilds its builtin set (builtinTools) filtered by those names. Custom/plugin/MCP tools in spec.Tools are silently dropped.

Why: In-process tool objects (closures, registries) cannot be serialized across a process boundary. Builtins are reconstructable; non-builtins are not. This preserves the common "read-only researcher" pattern (filtering builtins by name) while documenting the limitation for custom tools.

Deviation Removed a Provider name field after review

Spec: The spec did not mandate a provider-name field.

Implemented: An initial draft forwarded a Provider name hint, but resolveProvider derives the provider from model/baseURL/protocol and never read it - so a developer setting Provider: "anthropic" on a non-preset model would silently get OpenRouter. The field was removed from SubAgentProcessConfig and the wire payload.

Why: A dead, misleading field is worse than none. Forcing a wire format is done via Protocol ("anthropic"/"openai"); preset ids encode their provider. (Caught in code review, M1.)

Tradeoffs

Tradeoff Per-call subprocess spawn vs. a long-lived worker pool

Alternatives: (a) spawn a fresh pigo per sub-agent call (chosen); (b) keep a pool of long-lived subprocesses reused across calls.

Pros/cons: (a) is simplest and gives strongest isolation (each child is fresh; a crash corrupts nothing shared), at the cost of an exec + JSON-RPC handshake per call. (b) amortizes spawn cost but adds pool lifecycle, worker health checks, and state-leak risk.

Chose: Per-call spawn. The AC accepts subprocess execution and prioritizes isolation; spawn cost is acceptable for an "optional enhancement, low priority" feature. A pool is a possible follow-up if delegation becomes frequent.

Tradeoff Duplicate drain+classify logic in executeGoroutine and RunSubAgentOnce

Alternatives: (a) extract a shared "drain + classify failed run" core used by both (chosen NOT to); (b) leave them duplicated.

Pros/cons: (a) removes duplication but touches the goroutine path, risking the "no behavior change" regression guarantee. (b) keeps the goroutine path byte-identical to master (verified by git show master in review) at the cost of two ~similar blocks.

Chose: Duplication (b). The two differ deliberately anyway (goroutine forwards onUpdate, uses a placeholder for empty text; RunSubAgentOnce falls back to ErrorMessage and returns the raw text). A future change to one must be mirrored in the other - noted as an open item.

Tradeoff Cancellation via stdin close rather than a context-aware reader

Alternatives: (a) make runSubAgentRPC's stdin scanner select on ctx.Done(); (b) rely on the parent's Client.Close() closing the child's stdin to unblock the scanner.

Pros/cons: (a) is more obviously correct but needs a custom reader/goroutine. (b) is simpler and already correct: the parent owns the child's lifetime, and closing stdin on cancel/Close makes scanner.Scan return false.

Chose: (b). The ctx still flows into RunSubAgentOnce so an in-flight child run is cancelled; the scanner unblocks via stdin EOF. Documented as a known minor asymmetry.

Tradeoff Child stderr discarded by default

Alternatives: (a) SubAgentProcessConfig.Stderr defaults to nil (discarded) - chosen; (b) always capture and surface child stderr.

Pros/cons: (a) avoids noisy output when a caller does not care; diagnostics are instead surfaced through the RPC error path (RunSubAgentOnce falls back to ErrorMessage). (b) gives full child logs but at the cost of wiring/buffering.

Chose: (a) with opt-in Stderr. The key failure diagnostics (the cause of an error stop) reach the parent as the RPC error message, so stderr is not needed for correctness - only for deep debugging, which a caller can opt into.

Open Questions

Question Should process mode stream partial child updates?

Assumption: Process mode returns only the final result; goroutine mode streams text deltas via onUpdate. This is an intentional parity gap.

Verify/Follow-up: Is streaming parity wanted? It would require server->client JSON-RPC notifications (the current jsonrpc.Client drops notifications), so it is a non-trivial protocol extension deferred from #135.

Question Is per-call spawn acceptable, or do we need a subprocess pool?

Assumption: Each sub-agent call execs a fresh pigo. Fine for occasional delegation; costly for frequent delegation.

Verify: If sub-agents become a hot path, revisit a worker pool with health checks. Not needed for #135's "optional enhancement" scope.

Question Are builtins-only tools acceptable, or should the subprocess support a tool protocol?

Assumption: A process-isolated child runs with builtins (filtered by name); custom/plugin/MCP tools are silently dropped.

Verify: If a process-isolated sub-agent needs plugin/MCP tools, the subprocess would need a tool-registration RPC or to load the same plugins. Currently out of scope; confirm this limitation is acceptable.

Question Should --subagent-rpc be documented or stay hidden?

Assumption: The flag is internal (an implementation detail of process-isolated sub-agents) and not advertised in --help-style usage beyond its flag description.

Verify: Keep it hidden, or surface it for power users who want to drive the sub-agent protocol directly?

Question Sub-agents are library-level only - how should isolation be chosen in the REPL?

Assumption: SubAgentSpec is built by Skill.SubAgentSpec but no CLI/REPL path currently spawns sub-agents, so process mode is exercised by tests only. The default is goroutine.

Verify/Follow-up: When sub-agents are wired into the REPL (e.g. via skills), should there be a per-skill or global switch for process isolation, or should it be automatic for certain tool sets? This is a product decision deferred until sub-agents ship in the CLI.
