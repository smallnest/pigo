Implementation Notes

Issue #108 — REPL 会话持久化与恢复 — 2026-07-13

Design Decisions

Decision printSessions 改为写入注入的 io.Writer

Ambiguity: 验收要求 --list-sessions 保持现有输出格式，但原实现把结果直接 fmt.Println 到进程 stdout，忽略了 dispatch 已经持有的 out writer。

Choice: 把 printSessions() 改为 printSessions(out io.Writer)，dispatch 传入自己的 out。

Rationale: dispatch 是 run-assembly seam，其"输出经参数注入"的契约应贯穿到底；否则 --list-sessions 无法在测试中断言输出，也无法重定向。这是修复一处可测性缺陷，行为对终端用户不变。

Decision 恢复会话在启动时按角色回显

Ambiguity: "回显既有对话"的呈现形式未规定。

Choice: replayTranscript 遍历消息，user 前缀 > 、assistant 直出文本 + → tool: 行、toolResult 折叠为 ← result: 行——与 REPL 运行时的实时呈现保持一致。

Rationale: 回显与实时输出用同一套单行约定，用户在恢复与新对话间看到的格式统一。

Deviations

None — 每轮持久化到 ~/.pigo/sessions（或 $PIGO_HOME/sessions）、--list-sessions/--resume/--continue、模型随 SessionHeader 持久化、旧会话向后兼容（SchemaVersion 守卫）均在 #106 落地时已实现。本 Issue 补齐 US-006 验收测试并顺带修复 printSessions 可测性缺陷。

Tradeoffs

Tradeoff t.Setenv(PIGO_HOME) 隔离 vs 注入 Store

Alternatives: (A) 测试用 t.Setenv("PIGO_HOME", t.TempDir()) 让 --list-sessions/--continue 打到隔离目录；(B) 把 *session.Store 作为参数注入 dispatch。

Pros/Cons: (A) 零改动生产签名、覆盖真实的 sessionStore 解析路径；(B) 更纯粹但需要改 dispatch 签名、扩大改动面。

Why chosen: 选 (A)——sessionStore 已支持 PIGO_HOME 覆盖，用它隔离最小侵入且验证的是真实路径。
