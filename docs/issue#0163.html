Implementation Notes

Issue #163 — pigo list + pigo uninstall 命令 — 2026-07-20

Design Decisions

Decision uninstall 删除顺序：路径长度降序 + RemoveAll

Ambiguity: lockfile 的 Files 是文件与目录混合（含 plugins/<name>.pkg 目录项），删除顺序未定义（这正是 #162 的 Open Question）。

Choice: 先按路径长度降序排序再逐个 os.RemoveAll，使嵌套文件项先于其父目录项被删；RemoveAll 同时兼容普通文件与整棵 payload 目录。

Rationale: 长度降序保证子路径在父路径前处理；RemoveAll 对"已被父目录删除"的子项是幂等 no-op，二者叠加使删除既完整又健壮。

Decision list/uninstall 仍是纯 lockfile 操作，CLI 只做壳

Ambiguity: 逻辑放 CLI 还是 pkgmgr。

Choice: pkgmgr.ListInstalled / pkgmgr.Uninstall 承载逻辑并可单测；pkgcmd.go 的 runList/runUninstall 只做打印与退出码。

Rationale: 与 #162 的分层保持一致，业务逻辑可脱离 CLI 测试。

Deviations

Deviation uninstall 支持多包，一个失败不中止其余

Spec: AC 写的是 pigo uninstall <name>（单包）。

Implemented: runUninstall 接受多个 name，逐个卸载，失败打印错误并继续，任一失败退出码为 1。

Why: 与 #162 install 的多 ref 行为对称；单包场景行为不变。

Tradeoffs

Tradeoff 缺失文件：跳过 vs 报错

Alternatives: (A) 某记录文件已不存在则报错中止；(B) 视缺失为已达目标态，跳过并继续，最终仍移除 lockfile 条目。

Pros/Cons: A 更严格但会因外部手动删除或部分失败而卡住，无法收敛；B 幂等，重跑/半清理场景都能把记录清干净。

Winner: B。os.RemoveAll 对不存在路径返回 nil，天然契合"目标是 gone"的语义。
