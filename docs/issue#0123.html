Implementation Notes

Issue #0123 — /tree 分支导航 (US-007) — 2026-07-17

Design Decisions

Decision 在文件内引入分支存储（AppendBranch），而非重写为线性链

Ambiguity: Spec 要求 “/tree 选择某节点作为新活动 leaf 并从该节点延续”，但 pigo 会话文件此前始终是线性的（每次 Save 重写整个文件）。线性写入无法保留切换后产生的兄弟分支。

Choice: 新增 Store.AppendBranch(header, parentLeafID, messages)，保留所有既有 entry，只把新消息链接到指定父 leaf 之后，返回新 leaf id。REPL 用 curLeaf/persisted 跟踪活动 leaf 与已落盘条数，每轮以分支形式追加。

Rationale: 这是唯一能让 “切换 leaf 后继续” 在磁盘上真正生成兄弟分支、而不截断历史的方式。与 schema v3 的 entry 树（#121）自然衔接。

Decision 纯文本 ├─/└─ 树渲染，放弃移植 pi 的 TUI

Ambiguity: pi 的 tree-selector.ts 是全屏光标交互 TUI；spec 只要求 “纯行式 REPL 下可读（无需 TUI）”。

Choice: 在 session 包实现 RenderTreeLines(entries, leafID) []TreeLine：pre-order DFS，├─/└─ 连线，活动 leaf 标 “← current”，每行返回对应 Entry 以便按序号选择。

Rationale: pigo 已用行式 REPL 替代 bubbletea TUI（#106），文本渲染与该架构一致，且可被单测直接断言输出。

Decision /tree 在 REPL 循环中拦截，而非走 SlashRegistry Action

Ambiguity: 多数 slash 命令是纯 string→string 的 Action；/tree 需要改写共享 agentCtx.Messages 与 curLeaf。

Choice: 与 /compact /fork /clone 一致，在 runREPL 内直接拦截；同时在 registerLiveCommands 注册为 no-op builtin，仅为让 /help 列出。

Rationale: Action 闭包无法触及每轮可变状态；沿用既有拦截模式保持一致性。

Deviations

Deviation resume 路径改用 LoadEntries 重建 curLeaf

Spec: 未规定 resume 时如何设定活动 leaf。

Implemented: runInteractive 恢复会话时用 LoadEntries（而非 Load）读取 entry 树，并把 curLeaf 设为最后一个 entry，使 resume 后 /tree 能立即正确定位。

Why: 若不设 curLeaf，resume 后首个 /tree 会误判活动分支，破坏 “从当前 leaf 延续” 的语义。属于为满足验收条件而做的必要补齐，非偏离本意。

Tradeoffs

Tradeoff 空 tail 时 persistTurn 直接 return，不重写文件

Alternatives: (A) 每次都 Save 刷新 header 的 UpdatedAt；(B) 无新消息时直接 return。

Pros/Cons: A 会让 Save 重新生成 entry id 并把树拍平，直接使 curLeaf 失效、丢弃兄弟分支（正是首次测试失败的根因）；B 保留磁盘树完好，仅牺牲空轮时 UpdatedAt 的即时刷新。

Winner: B。分支完整性远重于空轮刷新时间戳；实际有新消息的轮次仍会 bump UpdatedAt。
