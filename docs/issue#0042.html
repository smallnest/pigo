Implementation Notes

Issue #42 — [impl] 分层配置系统（US-023）— 2026-07-10

Design Decisions

Decision 用指针字段区分「未设置」与「显式零值」

Ambiguity: 验收要求「较高层覆盖较低层」，但未说明如何区分「本层没设这个字段」与「本层显式设成空串」。

Choice: ConfigLayer 的可选字段全部用指针（*string）。nil = 未设置（透传下层），非 nil = 本层显式值（覆盖下层）。

Rationale: 这是 pi/zero「三态可选」惯用法的 Go 移植；只有这样才能实现字段级替换而非整层替换——较高层只覆盖它真正设置的字段。

Decision 凭据按 provider 合并，而非整体替换

Ambiguity: 验收把 credentials 列为配置项之一，但未说明多层间 credentials 如何合并。

Choice: Credentials 逐 provider 合并——较高层的某 provider key 覆盖较低层同名 key，但仅存在于较低层的其它 provider key 保留。

Rationale: 全局层可放公共 provider key，项目层只覆盖需改的那一个，无需重复列全部；与其它字段的「整体替换」区分开更贴合实际使用。

Decision 缺失文件不报错，非法内容才报错

Ambiguity: 验收要求「缺失/非法明确报错」，但「缺失」指配置文件缺失还是必填项缺失存在歧义。

Choice: LoadConfigLayer 对缺失文件返回 (nil, nil)（缺层不是错误），对存在但格式错误的文件返回硬错误；validate() 对非法字段值（空 model、未知 mode/level）返回硬错误。

Rationale: 全局/项目配置文件本就常缺失（合理默认即可运行），缺文件视为错误会破坏开箱即用；但文件存在却损坏、或字段值非法，是必须暴露的真实故障。

Deviations

Deviation 环境层刻意不采集凭据环境变量

Spec: credentials 是配置项之一，环境层是「环境/命令行」这一最高文件无关层。

Implemented: EnvConfigLayer 只采集 PIGO_MODEL / PIGO_PROVIDER / PIGO_TOOL_EXECUTION_MODE / PIGO_THINKING_LEVEL，不把凭据环境变量并入 ConfigLayer；凭据由 CredentialStore 惰性解析。

Why: 遵守 US-012「敏感值不写入日志」约束——不把 API key 塞进一个可能被打印/序列化的结构体，从源头杜绝泄露风险。

Tradeoffs

Tradeoff 变参 ResolveConfig(...*ConfigLayer) 且跳过 nil 层

Alternatives: (A) 变参传层、内部 if layer == nil { continue }；(B) 固定四参数（default/global/project/env）显式命名各层。

Pros/Cons: B 参数语义清晰但僵化，层数变动要改签名，且强迫调用方为缺层构造空结构；A 可直接把 LoadConfigLayer 的输出（可能为 nil）串进去，层数灵活。

Winner: A——调用方 ResolveConfig(&def, global, project, env) 一行到位，缺层天然透传。
