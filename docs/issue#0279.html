Issue #279 — /btw: 主循环拦截 + 侧线程上下文隔离骨架 — 2026-07-24

Design Decisions

Decision 侧线程用主消息切片的浅拷贝隔离

Ambiguity: PRD (FR-2) 要求"只读拷贝"作为背景，但未规定深拷贝还是浅拷贝。

Choice: newSideContext 用 make + copy 复制切片头，元素（agentcore.Message 接口值）共享。

Rationale: 消息一旦入列即不可变（全项目按追加式使用，从不原地改元素），因此复制切片头就足以保证向侧线程 append 永不触及 deps.agentCtx.Messages。深拷贝每条消息是无谓开销。TestNewSideContextIsolated 锁定了这一隔离保证。

Decision 完整复用 streamRun/drainGoalStream 的渲染管线

Ambiguity: PRD 设计考量提到"行式简化实现"，未指定复用哪段渲染代码。

Choice: drainSideStream 直接照搬 drainGoalStream 的 Markdown 缓冲 + 工具活动渲染，仅把目标 AgentContext 指向侧线程副本，且不做任何持久化。

Rationale: 保持侧线程输出风格与主对话/goal 一致；渲染 bug 只有一处来源。

Deviations

Deviation 本 issue 未实现 btw · side thread 之外的滚动 UI

Spec: PRD 描述了 pi-btw 风格的 PgUp/PgDn 滚动、固定头部 overlay。

Implemented: 仅打印一次头部横幅 + 行式流式输出。

Why: #279 范围明确限定为"拦截 + 隔离骨架"；多轮追问/退出交互属 #280，滚动浏览为 PRD Open Question 中的可选增强。pigo 当前 REPL 是逐行流式（非全屏 TUI），行式实现与现有架构一致。

Tradeoffs

Tradeoff 独立 runBtw 函数 vs 复用 streamRun 加参数

Alternatives: (A) 给 streamRun 加 persist bool/目标 ctx 参数复用；(B) 新写独立的 runBtw/askSide/drainSideStream。

Chosen: B。Pros: streamRun 的持久化逻辑（persistTurn、curLeaf 推进、header 更新）与"零污染/零落盘"契约根本对立，加开关会让 streamRun 变得布满 if persist 分支，易误用；独立函数让隔离契约在代码层面显式且不可能意外落盘。Cons: 渲染代码有一份复制（与 goal 已有的复制同源）。与 /goal 的既有模式一致，团队已接受这种"每个特殊模式一个 drain 函数"的结构。
