Issue #164 — pigo update 更新包到最新版本 — 2026-07-20

Design Decisions

Decision fetch+classify 先行，确认有新版本后才删旧文件

Ambiguity: AC 要求"更新失败时保留原安装（不留下损坏状态）"，但未规定失败点与顺序。

Choice: Update 先 Fetch+Classify 拿到新版本号；只有当新版本 ≠ 已装版本时，才 Uninstall 旧文件再 installFetched 新版本。

Rationale: 网络/解析失败发生在任何删除之前，旧安装原封不动，天然满足"不留损坏状态"。

Decision 抽出 installFetched 供 install 与 update 共用

Ambiguity: update 的"重新分发+写 lockfile"与 install 后半段重复。

Choice: 把 Install 的 classify→distribute→lockfile 抽成 installFetched(pkgDir, ref, ...)，Install 与 Update 都走它。

Rationale: 单一路径保证 install/update 分发与记账行为一致，减少漂移。

Decision update 以 source 去掉 pinned 版本拉最新

Choice: 从 lockfile 的 Source 解析出 ref 后清空 Version，让 npm pack 取最新已发布版本。

Rationale: "update to latest"的语义即忽略当初固定的版本。

Deviations

Deviation 更新旧文件用「Uninstall 后再装」而非原地覆盖

Spec: AC 描述为"卸载旧文件 → 重新拉取 → 重新分发"。

Implemented: 精确按此：复用 Uninstall 删除 lockfile 记录的旧文件，再 installFetched。这保证跨版本删掉的文件不会残留（原地覆盖会漏删旧版本独有文件）。

Why: 与 spec 一致，且比覆盖更干净。

Tradeoffs

Tradeoff update（不带名）遇错：中止 vs 继续

Alternatives: (A) 任一包失败即整体中止；(B) 逐包更新，失败收集进 joined error，最终退出码非零。

Pros/Cons: A 简单但一个坏包卡住全部；B 尽力而为，与 install/uninstall 的多目标行为对称。

Winner: B（UpdateAll）。CLI runUpdate 无参走 UpdateAll，有参逐个 Update。

Open Questions

Question "最新版本"由本机 npm 决定，无版本比较

Assumption: 判定"是否需要更新"仅比较 npm pack 返回的版本字符串是否等于已装版本；不做 semver 大小比较。

Verify: 若 registry 回退到更旧的 latest（罕见），当前逻辑会"更新"到那个版本。是否需要"仅当更高才更新"的 semver 守卫，留待确认。
