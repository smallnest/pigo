Implementation Notes

Issue #19 — 工具注册表 + JSON Schema 校验（US-014） — 2026-07-10

Design Decisions

Decision 注册时预编译 Schema，坏 Schema 早失败

Ambiguity: 验收只要求 "每工具声明 JSON Schema 并校验"，未规定编译时机。

Choice: Register 时即 Compile 并缓存 *jsonschema.Schema；非法 Schema 在注册即报错。

Rationale: 每次调用重编译浪费且把配置错误推迟到运行时；预编译让 Schema 错误在启动期暴露，运行时校验零编译开销。

Decision 空/缺失参数按空对象 {} 校验

Ambiguity: 模型可能对无参工具发 "" 或省略 arguments，Schema 校验对 nil 行为未定义。

Choice: nonEmptyJSON 把空参数归一为 {} 再校验。

Rationale: 使 "只含可选属性" 的 Schema 通过，同时让 required 缺失仍被正确报错，行为可预测。

Deviations

Deviation 校验失败返回 tool result 而非 Go error

Spec: 验收写 "校验失败产出字段级 error tool result"。

Implemented: Validate 返回 []FieldError，ValidationErrorResult 组装成 AgentToolResult（Details 携带结构化 []FieldError，Terminate 留 nil）。

Why: 契合 provider 层 "运行时失败编码进结果而非抛错" 的地基约定——校验失败应回喂给模型自我修正，不是中断循环。

Tradeoffs

Tradeoff 只收集叶子 cause 做字段级错误

Alternatives: (A) 只返回顶层 error.Error() 字符串；(B) 递归 ValidationError.Causes 取最具体的叶子。

Pros/Cons: A 简单但信息笼统（"validation failed"）；B 需遍历树，但能给出 /age: got string, want integer 这类可定位反馈。

Winner: B——验收明确要 "字段级"，叶子 cause 的 InstanceLocation 正好映射为 JSON pointer。

Tradeoff 每工具用独立 mem:// URL 编译

Alternatives: 共享一个 compiler/URL vs 每工具独立 resource URL。

Pros/Cons: 共享省对象但 $id/$ref 可能冲突；独立 URL（mem:///{name}.json）隔离彻底。

Winner: 独立 URL，避免跨工具 Schema 命名冲突，代价仅是每工具一个 compiler。
