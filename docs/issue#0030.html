Implementation Notes

Issue #30 — Gemini provider decoder（US-010）— 2026-07-10

Design Decisions

Decision functionCall 无 id，分配稳定合成 id call_<序号>

Ambiguity: 验收要求"解析 functionCall"，但 Gemini 的 functionCall part 不携带调用 id，而 pigo 的 ToolCallContent 及后续 tool-result 关联依赖非空 id。

Choice: 按到达顺序为每个 functionCall 分配 call_0、call_1… 的合成 id。

Rationale: 下游 executeToolCalls 用 id 关联 tool-result；合成 id 稳定、唯一、可预期，且 Gemini 回传 tool-result 时按 name/顺序匹配而非 id，故合成 id 不影响与 Gemini 的往返。

Decision functionCall 的 args 作为完整 JSON 对象一次性承载，非增量拼接

Ambiguity: 其它 provider（OpenAI/Anthropic）的工具参数是跨分片累积的字符串，验收未说明 Gemini 是否相同。

Choice: Gemini 在单个 part 里给出完整的 args 对象（json.RawMessage），直接透传，不做增量 builder 累积。

Rationale: Gemini wire 格式本就单片交付完整 args；强行套用累积模型反而引入无谓状态。终态仍做 {} 兜底以防空 args。

Decision thought=true 的 part 归为 thinking 内容

Ambiguity: Gemini 2.x 的 thinking 以带 thought:true 标记的 text part 表达，验收只提"解析文本与 functionCall"未明确 thinking。

Choice: part 若 thought=true 则累积进 thinking builder 并发 StreamThinkingEvent；否则普通 text。terminal 消息中 thinking 块在 text 块之前。

Rationale: 与 Anthropic decoder 的 thinking 处理对齐；提前正确归类避免把推理内容混入正文。

Deviations

Deviation finishReason=STOP 在存在 functionCall 时映射为 tool_use 而非 end_turn

Spec: 验收要求"stopReason 映射正确（length/tool_use/end_turn）"。

Implemented: Gemini 即便产生 functionCall 也报 finishReason=STOP（不像 OpenAI 有专门的 tool_calls）。因此 mapGeminiFinishReason 与 finishDone 均优先判断"是否出现过 functionCall"：出现即映射 tool_use，与 loop 的工具执行分支对齐。

Why: 若照字面把 STOP 映射为 end_turn，loop 会误判为自然终止而不执行工具调用，导致 Gemini 的工具能力失效。以"是否有 tool call"为准更贴近语义与 pi 行为。

Tradeoffs

Tradeoff usageMetadata 覆盖式记录 vs 累加

Alternatives: (A) 每次出现 usageMetadata 覆盖式写入 input/output tokens；(B) 累加各 chunk 的增量。

Pros/Cons: A 简单，符合 Gemini "usageMetadata 携累积总量、通常仅在末尾 chunk 出现"的行为；B 在 Gemini 报增量时才需要，但 Gemini 实际报累积值，累加会翻倍。

Winner: A——与 Anthropic decoder 的 usage 处理一致；Gemini 的 usageMetadata 是累积快照，覆盖即正确。

Tradeoff 复用 feedSSE/sseServer 录制 fixture vs 独立测试骨架

Alternatives: (A) 复用共享的 feedSSE/eventKinds/sseServer/newReqFn，一份 Gemini fixture 同时跑纯 decoder 单测与 transport 全链路；(B) 为 Gemini 单建骨架。

Pros/Cons: A 与 Anthropic/OpenAI 测试范式统一、零重复；B 隔离性略好但重复。

Winner: A——三个 provider decoder 共享 transport 契约与测试骨架，保证行为一致。

Assumption: 假设 Gemini 的 functionResponse 按函数 name 匹配，不依赖调用方回传 id，故合成 call_N 安全。

Verify: 接入真实多轮工具调用后核对：向 Gemini 回传 tool-result 时是否需要原样带回某个 id；若需要，则合成方案需改为透传 Gemini 侧标识。
