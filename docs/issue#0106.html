Implementation Notes

Issue #106 — 实现行式 REPL 并替换 TUI 交互入口 — 2026-07-13

Design Decisions

Decision 用单行工具活动提示替代全屏渲染

Ambiguity: Spec 要求 REPL 展示工具调用/结果，但未规定展示形式。

Choice: 在助手回复流式输出后，用 → tool: read 与 ← result: … 两行紧凑提示呈现工具活动，多行结果折叠成单行并截断到 120 字符（见 oneLine）。

Rationale: 行式 REPL 的核心目标是"保持代码简洁"。全屏 viewport 渲染正是被删除的 bubbletea TUI 的复杂度来源；单行提示信息足够、无状态、无需重绘。

Decision 跨轮次共享同一个不断增长的 AgentContext

Ambiguity: Spec 要求"多轮对话共享历史"，但未规定上下文的生命周期管理方式。

Choice: runInteractive 构造单个 *agentcore.AgentContext，streamRun 每轮把用户消息 append 进去并原地增长，运行结束后整体持久化到 JSONL session。

Rationale: 单一可变上下文比每轮重建更简单，且天然满足"下一个 prompt 延续对话"的要求；session 持久化复用已有的 Store.Save 全量写。

Decision SIGINT 只取消当前运行，空闲时才退出

Ambiguity: Spec 要求 EOF//exit//quit 干净退出（code 0），但未明确运行中按 Ctrl+C 的语义。

Choice: 为整个 REPL 安装一个 SIGINT handler，通过互斥保护的 runCancel 只取消当前 run 的 context，打印 ^C interrupted 后回到提示符；空闲时读到 EOF 才退出。

Rationale: 对标常见 REPL 交互习惯——中断一次长运行不应杀死整个会话。

Deviations

None — 实现遵循了 Issue #106 的九条验收标准（行式 REPL、共享上下文、忽略空行、EOF//exit//quit code 0、非终端且无 prompt 无 resume 报错、经 runtime.StartRun/LoopEventStream 增量输出助手文本、单行工具调用/结果提示、cmd/pigo 中无 internal/tui 引用、build+vet 通过）。

Tradeoffs

Tradeoff 行式 REPL vs 保留 bubbletea 全屏 TUI

Alternatives: (A) 保留 bubbletea TUI 并做增量优化；(B) 完全删除 TUI，改为行式 read→run→print 循环。

Pros/Cons: TUI 提供富交互（菜单、viewport、主题），但带来大量状态机、渲染与测试复杂度（15 个已删除文件）；行式 REPL 功能更朴素，但代码量极小、无隐藏状态、易测试。

Why chosen: 用户的总目标即"tui 代码逻辑太复杂了，删除 tui 相关，保持代码简洁"——简洁性压倒富交互。

Tradeoff 每轮全量 Save vs 增量 Append 持久化

Alternatives: (A) 每轮 Store.Save 全量重写 session 文件；(B) 用 Store.Append 只追加新消息。

Pros/Cons: 全量写每轮重写整个文件（会话越长开销越大），但逻辑最简单且天然更新 header 的 UpdatedAt/Model；Append 更省 I/O，但因 header 在首行，仍需 load-modify-save。

Why chosen: pigo 产生的会话规模下全量写完全够用，且 REPL 需要每轮更新 header 的 model/provider（可能被 /model 中途切换），全量写路径最直接。
