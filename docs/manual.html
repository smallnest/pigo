pigo 使用手册

pigo 是一个面向命令行的 AI Agent，可在脚本与 CI 中无头运行，并支持安装 pi 生态的扩展、技能、命令与主题包。

简介

pigo 提供两类能力：

  • Agent 模式——以 flag 驱动，针对单条 prompt 运行 Agent 循环，适合脚本化与流水线。
  • 包管理——通过 install / list / uninstall / update 子命令，从 npm 安装 pi 生态的扩展、技能、命令模板与主题。
提示　包管理子命令是位置参数，独立于 Agent 模式的 flag；pigo 会在解析 flag 之前先把它们剥离出来。

安装与构建

pigo 使用 Go 编写。从源码构建：

# 克隆并构建
git clone https://github.com/smallnest/pigo.git
cd pigo
go build -o pigo ./cmd/pigo

# 或直接安装到 $GOBIN
go install github.com/smallnest/pigo/cmd/pigo@latest
前置依赖　包管理功能（install/update）依赖本机已安装的 npm，pigo 通过 npm pack --ignore-scripts 安全下载包，不会执行包的生命周期脚本。

快速开始

# 1) 用 Agent 跑一条 prompt
pigo -p "读取 README 并用中文总结"

# 2) 安装一个 pi 扩展包
pigo install npm:pi-mcp-adapter

# 3) 查看已安装的包
pigo list

# 4) 更新到最新版本
pigo update pi-mcp-adapter

运行 Agent

Agent 模式对单条 prompt 运行一次 Agent 循环，退出码反映成功（0）或失败（1），便于在管道中组合。

# 打印模式：仅输出最终文本
pigo -p "解释这段代码" -m openrouter/free

# 流式 JSON 事件（行分隔），适合程序消费
pigo -p "..." --output-format stream-json

# prompt 也可作为位置参数传入
pigo "帮我写一个 Go 的快排"

常用 flag

Flag简写说明
--print-p要运行的 prompt（无头打印模式）
--model-m模型 id，默认 openrouter/free
--base-url-u覆盖 Provider 的 base URL（如本地 Ollama）
--api-key-kProvider 的 API Key（覆盖环境变量/配置）
--protocol-P强制线协议：openai | anthropic
--output-format-o输出格式：text | stream-json
--no-tools-n禁用内置的文件/Shell 工具
--continue-c恢复最近一次交互会话
--resume-r按 id 恢复指定的交互会话
--list-sessions-l列出已存储的交互会话并退出

模型与 Provider

pigo 从 --model 解析对应的内置 Provider，API Key 从环境变量取用（<PROVIDER>_API_KEY）。解析规则如下：

  1. 若显式指定 --protocol（openai/anthropic），优先于一切启发式；openai 需配合 --base-url。
  2. 预置目录中的模型 id 使用其自带 Provider（OpenRouter / NVIDIA / Ollama）。
  3. ollama/ 前缀或指向 11434 端口的 base URL → 本地 Ollama。
  4. nvidia/ 前缀 → NVIDIA NIM。
  5. 其余一律走 OpenRouter（参考 OpenAI 兼容网关）。
# 本地 Ollama
pigo -p "你好" -m ollama/qwen2.5

# 自建 OpenAI 兼容端点
pigo -p "..." -P openai -u https://my-endpoint/v1 -m my-model

包管理概览

pigo 可安装 pi 生态发布到 npm 的四类包，并把内容分发到 pigo 现有发现机制能识别的目录，无需额外配置：

extension 扩展

可执行扩展，常见为 MCP adapter。落到 plugins/，由 pigo 插件系统发现。

skill 技能

含 SKILL.md 的技能包，落到技能目录（~/.agents/skills）。

prompt 命令

命令/prompt 模板（*.md），落到 commands/。

theme 主题

主题包，存储到 themes/（暂无运行时消费者）。

安装信息记录在 lockfile（$PIGO_HOME/packages.json）中，作为 list/uninstall/update 的事实来源。

来源前缀　当前仅支持 npm:，包括作用域包 npm:@scope/name，可选带版本 npm:name@1.2.3。

install

安装一个或多个 pi 包。流程为：解析引用 → npm pack 拉取并解压 → 分类 → 按类型分发 → 写 lockfile。

# 安装最新版
pigo install npm:pi-mcp-adapter

# 指定版本 / 作用域包
pigo install npm:pi-mcp-adapter@1.2.0
pigo install npm:@myscope/pi-theme

# 一次安装多个（某个失败不影响其余）
pigo install npm:pi-mcp-adapter npm:pi-some-skill

安装成功后打印 Installed <name>@<ver> (<types>) — N file(s)。多包安装时，若任一失败退出码为 1，但其余包照常安装。

幂等　每个分发器会先清理自身旧目标再写入，因此重复安装可覆盖收敛；安装非 pi 包会失败且不写 lockfile。

list

列出全部已安装的包，每行以制表符分隔 名称　版本　类型　来源，便于用 cut -f 等工具消费。

pigo list
# pi-mcp-adapter    1.2.0    extension    npm:pi-mcp-adapter
# pi-demo-skill     0.3.1    skill        npm:pi-demo-skill

# 未安装任何包时
pigo list
# no packages installed

uninstall

卸载包：删除 lockfile 中记录的全部落地文件，再移除该条目。支持一次卸载多个。

pigo uninstall pi-mcp-adapter
pigo uninstall pi-mcp-adapter pi-demo-skill
  • 按路径长度降序删除，并对目录使用 RemoveAll，确保 payload 目录被整体清理。
  • 某个记录文件若已不存在，跳过并继续，最终仍会移除 lockfile 条目（收敛到"已卸载"状态）。
  • 卸载未安装的包会报错 package not installed: <name> 并以非零码退出。

update

把已安装的包更新到 npm 上的最新版本。

# 更新指定包（可多个）
pigo update pi-mcp-adapter

# 不带包名：遍历更新全部已安装包
pigo update
  • 先拉取并分类最新版本，仅当版本与已装不同才「卸旧文件 → 重新分发 → 更新 lockfile 版本」。
  • 已是最新版时打印 <name> is up to date，不做无谓改动。
  • 失败安全：网络或分类失败发生在任何删除之前，原安装保持不变，不会留下损坏状态。
  • 无参 update 逐包尽力而为，任一失败则退出码非零。

目录与环境变量

路径 / 变量说明
$PIGO_HOMEpigo 主目录，默认 ~/.pigo
$PIGO_HOME/packages.jsonlockfile，记录已安装包
$PIGO_HOME/plugins/扩展（extension）落地目录
$PIGO_HOME/commands/命令/prompt 模板落地目录
$PIGO_HOME/themes/主题存储目录
技能目录默认 ~/.agents/skills，可用 PIGO_SKILLS_DIR 覆盖
<PROVIDER>_API_KEY各 Provider 的 API Key，如 OPENROUTER_API_KEY
注意　技能是 $PIGO_HOME 之外的唯一例外：pigo 从 ~/.agents/skills（或 PIGO_SKILLS_DIR）加载技能，而非嵌套在主目录下。

常见问题

提示找不到 npm？

install/update 依赖本机 npm。请先安装 Node.js（自带 npm），并确认 npm 在 PATH 中。

安装报"unrecognized pi package"？

该包不含任何可识别的 pi 元数据或结构特征（如 bin、SKILL.md、commands/），因此不是 pi 包，pigo 拒绝安装且不写 lockfile。

只支持 npm 来源吗？

本版本仅支持 npm:（含作用域包）。github:/file: 等来源不在当前范围内。

update 会做版本大小比较吗？

当前仅比较 npm pack 返回的版本字符串是否与已装版本相同，不做 semver 大小判断——即"更新到 latest"。
