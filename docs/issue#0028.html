Implementation Notes

Issue #28 — Anthropic provider decoder（US-008）— 2026-07-10

Design Decisions

Decision AnthropicDecoder 作为有状态 Decoder，按 content-block index 累积 partial

Ambiguity: 验收只要求"正确解析 text/thinking/tool_use 增量"，未规定 decoder 如何组织跨多个 SSE 事件的累积状态。

Choice: 用 map[int]*anthropicBlock 按 Anthropic 的 content-block index 归集每个块的增量（text/thinking 各自 strings.Builder，tool_use 累积 partial JSON 字符串），并单独记录 order 保证首见顺序；每次 delta 后 partial() 物化出完整 AssistantMessage 随事件外发。

Rationale: 复刻既有 transport Decoder 契约（Decode+Finish），与 jsonDecoder 测试范式一致；index 归集天然处理 Anthropic 交错到达的 block 事件，partial 物化让消费方每步都拿到可用快照。

Decision message_delta 不产出独立事件，仅累积 stop_reason 与 usage

Ambiguity: AssistantMessageEvent 集合里没有"usage/stop 更新"这种 kind，验收也没要求为其单独发事件。

Choice: onMessageDelta 只把 stop_reason（经 mapAnthropicStopReason 映射）与 output/input token 写入 decoder 状态，返回 nil（零事件）；这些值在终态 StreamDoneEvent.Message 里一并交付。

Rationale: 事件集合是固定的 6 种（start/text/thinking/toolcall/done/error），不为 Anthropic 特有的中间态硬造事件；终态消息已携带 Usage 与 StopReason，消费方无需中途感知。

Decision stop_reason 映射表：max_tokens→length、tool_use→tool_use、其余→end_turn

Ambiguity: 验收要求"正确映射 stopReason（length/tool_use/end_turn）"，但 Anthropic 还有 stop_sequence 等值未列出。

Choice: max_tokens→StopReasonLength、tool_use→StopReasonToolUse、end_turn/stop_sequence→StopReasonEndTurn，未知值兜底 end_turn。

Rationale: stop_sequence 语义上是自然停止，归入 end_turn 最贴近 pi；未知值兜底为非 error 的自然停止，避免把新出现的 wire 值误判为失败。

Deviations

Deviation Anthropic error 事件表达为 Decode 返回 error，而非自行发 StreamErrorEvent

Spec: 验收要求"网络/限流/鉴权失败以流内终态 error 消息表达不 panic"。

Implemented: decoder 遇到 error 事件或畸形 JSON 时返回 Go error；由 transport 的 flush() 统一转成终态 StreamErrorEvent（stopReason=error）。decoder 自身从不构造终态 error 事件、从不 panic。

Why: transport 已拥有 dual failure model 的唯一落点（fail()），decoder 只需把失败上报为 error，职责单一；网络/限流/鉴权失败本就发生在 transport 层（连接/读循环），decoder 层的 error 事件与之走同一条终态路径，语义统一。

Tradeoffs

Tradeoff tool_use 参数在 partial 中透传原始累积串，终态兜底 {}

Alternatives: (A) partial 阶段直接透传 toolJSON 累积串（可能是不完整 JSON），终态若为空则兜底 {}；(B) 每次 delta 都尝试解析完整 JSON，未闭合则丢弃。

Pros/Cons: A 简单、零解析开销、流式 partial 可展示"正在输入的参数"，代价是 partial 期 Arguments 可能非法 JSON（消费方本就应只在 done 后解析）；B 保证每个 partial 的 Arguments 恒合法，但要反复试解析、且丢弃中间态失去流式可见性。

Winner: A——input_json_delta 是逐段拼接设计，闭合只在最后一段发生；下游（loop/executeToolCalls）只消费终态 done 的 Arguments，partial 的合法性非必需。

Tradeoff 录制 fixture SSE 单测 vs mock decoder 单元测试

Alternatives: (A) 内联一段结构忠实的录制 Anthropic SSE 常量，既做纯 decoder 单测（feedSSE 手工切分）又串 transport（sseServer）跑全链路；(B) 只测 Decode 逐事件返回。

Pros/Cons: A 覆盖真实事件序列（message_start→block×3→message_delta→message_stop）与 transport 集成，贴合验收"录制 fixture 的 SSE 解析单测"；B 更细粒度但漏掉 transport 拼装/flush 路径。

Winner: A——复用 transport_test.go 已有的 sseServer/newReqFn 范式，一份 fixture 同时验证 decoder 与端到端。
