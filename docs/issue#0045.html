Implementation Notes

Issue #45 — sub-agent 编排 + skills + slash-commands（US-027/028/029）— 2026-07-10

Design Decisions

Decision Sub-agent is just another AgentTool

Ambiguity: The spec said "父 agent 通过普通的 tool call 触发子 agent" without prescribing the integration seam.

Choice: SubAgentTool implements the existing AgentTool interface; Execute spins up a child StartRun over an independent AgentContext and returns the child's final assistant text as the tool result.

Rationale: Reuses the batch executor, tool-result plumbing, and streaming machinery unchanged. Concurrency comes for free — the batch executor already runs parallel tool calls in separate goroutines, so multiple sub-agents run concurrently with no new orchestration code.

Decision Drain events, then read Result

Ambiguity: How to consume the child's LoopEventStream (buffer 0, synchronous back-pressure) without deadlocking.

Choice: Execute ranges stream.Events() to channel close (forwarding streamed text via onUpdate when a sink is set), then calls stream.Result(ctx).

Rationale: runLoop.finish() calls SetResult before Close, and Result resolves via its own channel independent of event draining — so draining to close then reading the result is deadlock-free. Draining also unblocks the buffer-0 producer.

Decision Skills reuse the sub-agent abstraction

Ambiguity: How a declarative SKILL.md becomes an executable capability.

Choice: A Skill materializes into a SubAgentSpec — the markdown body becomes the child's system prompt, allowed-tools filters the tool set, and description is surfaced to the parent.

Rationale: A skill is conceptually a pre-configured sub-agent; layering it on SubAgentTool avoids a second execution path.

Decision Built-in slash-commands win over user commands

Ambiguity: Conflict resolution when a user ~/.pigo/commands/*.md template shares a name with a compile-time built-in.

Choice: Built-in wins; the shadowed user command is recorded and reported on stderr. Built-ins register from init() via RegisterBuiltin.

Rationale: Matches the spec's "冲突时 built-in 优先" and keeps a fork's guarantees stable regardless of user files on disk.

Deviations

Deviation No standalone plugin mechanism

Spec: The PRD's capability list mentions a plugin surface alongside skills and commands.

Implemented: No separate plugin registry. Fork authors extend via init() registration (built-in commands/tools); external/third-party extension is deferred to MCP.

Why: A bespoke in-process plugin loader would duplicate what init()-time registration and (future) MCP already cover, adding a maintenance surface with no distinct use case in v1.

Deviation Failed child surfaced as a tool error (added in review)

Spec: Silent on how a child's failed run should appear to the parent.

Implemented: When the child's final turn stops on error/aborted, Execute returns a non-nil Go error (tool executor marks it IsError), carrying the child's diagnostic text.

Why: The codebase convention is "failures are terminal messages, not Go errors," which meant a failed delegation reached the parent as a successful tool result. Surfacing it as an error gives the parent model a signal to react to. Covered by TestSubAgentFailedChildErrors.

Tradeoffs

Tradeoff Single-process goroutines vs cross-process RPC

Alternatives: (a) sub-agents as goroutines sharing the parent process; (b) spawn child agents as separate OS processes over RPC.

Chosen: (a). Pros: zero serialization, shared credential store, trivial cancellation via the parent ctx, no transport code. Cons: no hard memory/crash isolation between parent and child.

Why it won: The spec explicitly decided "单进程 goroutine"; isolation is not a v1 requirement and RPC would add a large surface for no current benefit.

Tradeoff Lockless builtinCommands global vs guarded map

Alternatives: (a) plain package-level map with an init-only contract; (b) a sync.RWMutex-guarded registry.

Chosen: (a), with a comment reaffirming the init-only discipline (added in review).

Why it won: Registration happens only from init() (single-threaded, before main) and reads happen only afterward, so no concurrent access occurs. A mutex would add ceremony without preventing any real race.

Open Questions

Question Should sub-agent nesting depth be bounded?

A sub-agent's tool set could itself contain sub-agent tools, allowing recursive delegation. There is currently no depth guard. Verify whether a max-depth or cycle limit is wanted before enabling nested sub-agent tool sets in practice.

Question Streaming sub-agent text into the transcript

Execute forwards child MessageUpdateEvent text via onUpdate as tool-execution updates, but the TUI currently renders tool results as a one-line gist. Confirm whether live sub-agent progress should be shown more prominently in the interactive transcript.
