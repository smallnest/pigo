Issue #18 — EventStream 事件流机制（US-002） — 2026-07-10

Design Decisions

Decision Result 与 error 用同一个 resolvedCh + sync.Once 兑现

Ambiguity: 研究文档只说 "用 sync.Once + closed channel 表示就绪"，未定义失败路径如何表达。

Choice: SetResult 与 SetError 共用一个 resultOnce，谁先调用谁生效，都 close 同一个 resultCh；Result(ctx) 返回 (R, error)。

Rationale: agent loop 的终态可能是正常结束（agent_end→messages）或中断（aborted/error）。用统一的 once 保证结果只兑现一次，且 Result 的调用方一次拿到成功值或失败原因，语义清晰。

Decision 保留 IsComplete/ExtractResult 回调，但允许显式 SetResult

Ambiguity: 研究文档提出两种等价路径（回调自动捕获 vs loop 内显式设置），未强制其一。

Choice: 两者都支持——回调设为可选字段，Emit 时若命中 IsComplete 则自动 ExtractResult；provider 层可用回调，agent loop 可显式 SetResult。

Rationale: 验收条件明确要求 "保留 isComplete/extractResult 回调字段"，同时显式 SetResult 更符合 Go 习惯，两条路径共存不冲突（都走 once）。

Deviations

Deviation Close 无结果时兜底为 ErrStreamIncomplete

Spec: pi 的 end(result?) 允许无结果关闭，async iterator 自然终止。

Implemented: Close() 若结果未兑现，用 ErrStreamIncomplete 兜底再关闭 channel。

Why: Go 的 Result(ctx) 是阻塞调用，若生产者忘记 SetResult 就 Close，消费者会永久阻塞。兜底把 "异常结束" 显式化为可观测的 error，避免死锁。

Tradeoffs

Tradeoff 结果不走事件 channel，单独 resultCh 兑现

Alternatives: (A) 把最终结果作为最后一个事件推入 channel；(B) 独立 resultCh + sync.Once。

Pros/Cons: A 实现简单，但消费者若提前 break 就永远拿不到结果，且类型 T≠R 无法共用 channel；B 多一个 channel，但消费与取结果解耦。

Winner: B——研究文档明确 "不要把 result 也塞进事件 channel"，且 Result 可在不读完事件的情况下获取。

Tradeoff 默认 buffer=0 提供同步背压

Alternatives: 无缓冲（同步） vs 带缓冲 channel。

Pros/Cons: 无缓冲精确对应 pi 的顺序 await emit，消费者慢则生产者阻塞（想要的背压）；缓冲会让生产者跑在消费者前面，事件与状态可能失步。

Winner: 默认 0，但 NewEventStream(buffer) 保留调参余地（测试与 provider 层可按需加缓冲）。
