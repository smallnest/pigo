Issue #35 — write 工具（US-016）— 2026-07-10

Design Decisions

Decision 覆盖行为：写前 stat 检测，结果里报告 Created/Overwrote

Ambiguity: 验收要"覆盖已存在文件的行为与 pi 一致（记录/提示）"，未规定如何提示。

Choice: 写入前 os.Stat 判断目标是否已存在；结果文本用 "Created" / "Overwrote" 区分，并在 Details 里带 overwrote 布尔。

Rationale: pi 的 write 会告知是否覆盖；显式提示让模型/用户对破坏性覆盖有感知，Details 结构化字段便于上层 UI 高亮。

Decision write 为 sequential 执行，复用 read 的 Root 边界策略

Ambiguity: 工具的并发执行模式验收未涉及。

Choice: ExecutionMode 返回 Sequential；resolvePath 与 ReadTool 完全一致的 Root 越界判定。

Rationale: 写会改文件系统，串行避免同批次并发写竞争；共享一份边界策略保证 read/write 安全语义一致。

Deviations

None 无偏离

实现覆盖验收：写指定路径、按需创建父目录（MkdirAll）、覆盖有提示、写失败明确报错、t.TempDir 单测、go test ./... 通过。

Tradeoffs

Tradeoff 直接 WriteFile 覆盖 vs 写临时文件后原子 rename

Alternatives: (A) os.WriteFile 直接覆盖；(B) 写 tmp + os.Rename 原子替换。

Pros/Cons: A 简单，与 pi 直写一致；B 崩溃时不会留半截文件，但多一次 tmp 写与 rename，且跨设备 rename 失败。

Winner: A——对齐 pi 的直写语义，工具单次写体量小；原子写可在需要时作为增强项（见 Open Questions）。
