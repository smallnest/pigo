Issue #20 — streamAssistantResponse 流式回填（US-003） — 2026-07-10

Design Decisions

Decision 用 sealed interface + EventKind() 建模 provider delta 事件

Ambiguity: pi 用 TS discriminated union 表达流式 delta；Go 无原生等价物。

Choice: AssistantMessageEvent 密封接口（未导出 isAssistantMessageEvent()）+ EventKind() 判别串，6 种事件 start/text/thinking/toolcall/done/error。

Rationale: 密封接口锁死实现集合，loop 用 type switch 穷尽分发；EventKind 供 MessageUpdateEvent 透传给消费者，无需反射。

Decision "push placeholder + replace last" 回填部分消息

Ambiguity: 验收要 "partial 实时回填到 context"，但未定义首个 delta 前 context 尚无占位消息时的处理。

Choice: addedPartial 布尔：首个 delta 时 append 占位，之后替换末位；done/error 时 finalizeMessage 同样兼顾 "没有 start 直接 done" 的边界。

Rationale: 保证 context 里始终只有一条 assistant 消息随流增长，最终被终态消息原位替换，不会重复 append。

Deviations

Deviation 终态用 AssistantMessage{} 零值而非 nil

Spec: pi 中 message 可为 null/undefined，失败路径返回空。

Implemented: AssistantMessage 是具体 struct（非接口），作为 EventStream[_, AssistantMessage] 的 R 类型不可为 nil。取消/失败路径返回 AssistantMessage{}, err；空判用 final.RoleField == "" 而非 == nil。

Why: Go 类型系统约束——具体 struct 无 nil 态，零值 + 字段判空是等价且类型安全的表达。

Tradeoffs

Tradeoff 早期 build 失败合成终态消息 vs 返回 error

Alternatives: (A) StreamFn 早期失败时 streamAssistantResponse 直接返回 Go error；(B) 合成 stopReason=error 的终态 assistant 消息，返回 nil error。

Pros/Cons: A 让 loop 需双路径处理错误；B 统一——所有失败（早期/运行时）都表现为一条终态 assistant 消息，loop 只有一条记录路径。

Winner: B（newErrorAssistantMessage），契合 FR-13 provider 契约 "运行时失败编码进消息而非抛错"。
