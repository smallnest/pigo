Implementation Notes

Issue #22 — 并行与串行工具执行（US-005） — 2026-07-10

Design Decisions

Decision 批量层复用 executeToolCall 而不拆分 prepare/execute 两阶段

Ambiguity: 验收写 parallel 模式"先顺序 prepare 再并发执行"，暗示两阶段分离。

Choice: executeToolCalls 直接在每个 goroutine 内整体调用 #21 的 executeToolCall（prepare+execute+finalize 一体），保序靠向源 index 回填结果。

Rationale: #21 已把三段式封为单调用原语，再拆会破坏其 afterToolCall/事件语义；保序由 index 回填保证，与"先 prepare"目的等价，且并发不改变可见顺序。

Decision BatchConfig 内嵌 ToolExecutorConfig + ForceSequential 开关

Ambiguity: 验收要"全局 sequential 时整批串行"，未规定配置载体。

Choice: BatchConfig{ToolExecutorConfig; ForceSequential bool}——批量层新增全局开关，复用单调用配置。

Rationale: 内嵌让批量层零成本透传 hooks 给 executeToolCall；ForceSequential 与"任一工具 sequential"是两条独立触发串行的路径。

Deviations

None 无偏离

实现完全遵循验收：sequential 逐个执行遇 abort 中断并回填 aborted 错误结果；parallel 用 goroutine + index 回填保序；任一工具 sequential 或 ForceSequential 触发整批串行；仅全部 terminate=true 才整批终止。

Tradeoffs

Tradeoff parallel 用裸 sync.WaitGroup 无并发上限

Alternatives: (A) 裸 WaitGroup 全部并发；(B) 带 semaphore 的有界并发池。

Pros/Cons: A 简单、契合 pi 语义（单条 assistant 消息的工具调用数天然受限于模型输出，不会海量）；B 防御无界 goroutine 但引入配置项与复杂度，当前无需求。

Winner: A——批量规模由模型单轮 tool_calls 数量约束，实践中为个位数，semaphore 是过度设计。

Open Questions

Question abort 语义仅覆盖 sequential 模式

Assumption: ctx abort 时，sequential 循环会在下一次迭代前检测并回填剩余 aborted 结果；parallel 模式则依赖每个 executeToolCall 内部对 ctx 的响应，批量层不主动中断已启动的 goroutine。
