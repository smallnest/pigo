Implementation Notes

Issue #280 — /btw: 侧线程多轮追问与退出交互 — 2026-07-24

Design Decisions

Decision 追问循环复用现有 replLineEditor.readLine

Ambiguity: Issue 要求"直接输入文本回车即追问"，但未指定用哪套输入读取。

Choice: btwFollowUpLoop 用主 REPL 的同一个 editor（透传进 runBtw），以 btw> 提示符读行。

Rationale: 复用同一 editor 得到一致的行编辑/历史/补全行为，且共享同一 buffered reader，输入不会在主循环与侧线程间被切割。

Decision 侧线程消息在同一 side *AgentContext 上累积

Ambiguity: "追问共享侧线程历史"未规定存储方式。

Choice: 每次 askSide 追加到同一个 side.Messages，后续追问自然带上前序 Q&A；该 ctx 是主上下文的拷贝，退出即丢弃。

Rationale: 最小状态，天然满足 FR-4，且隔离契约不变。

Deviations

Deviation 头部横幅只在进入时打印一次

Spec: Issue 写"头部持续显示固定标识"。

Implemented: 行式 REPL 无固定 header 区，进入侧线程打印一次横幅；每轮追问用 btw> 提示符持续标识当前处于侧线程。

Why: pigo 是逐行流式输出而非全屏 TUI，无法钉住一行 header；btw> 提示符已足以持续区分侧线程与主对话。

Tradeoffs

Tradeoff 两段式 Ctrl+C 的实现方式

Alternatives: (A) 在 askSide 内计数连续 Ctrl+C；(B) 分工：运行中的 Ctrl+C 由 SIGINT→cancel 取消该答，空闲时 readLine 返回 errLineInterrupted 即退出线程。

Chosen: B。Pros: 复用现成的 SIGINT cancel 管线与 readLine 的中断返回，无需额外计数状态，语义天然分层（运行中取消 / 空闲退出）。Cons: "取消答"与"退出线程"是两次独立 Ctrl+C 而非一次内的状态机，但这正是 Issue 描述的期望。

Open Questions
