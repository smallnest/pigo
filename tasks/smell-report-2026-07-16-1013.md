# Architecture Smell Report

**Project:** pigo (github.com/smallnest/pigo)
**Scope:** Entire project (全项目全量) — 37 non-test `.go` files, ~7,288 lines
**Date:** 2026-07-16
**Analyzer:** smell skill (Ducc)

---

## Executive Summary

pigo is a headless / REPL CLI AI agent written in Go, organized as a clean layered (onion) architecture. The overall architectural health is **strong**. There are **no circular dependencies, no layer-boundary violations, and no God Objects**. `agentcore` is a proper dependency sink (a leaf package importing zero internal packages, targeted by 23 of 24 internal import edges), and every other package depends inward toward it. Naming is excellent, there is zero commented-out code, and there is zero TODO/FIXME accumulation.

The findings are therefore almost entirely **code-level and micro-architectural**, not structural. The three issues most worth acting on are: (1) a genuine algorithmic hot path — `.gitignore` matching runs a full linear rule scan **per file** during grep/find directory walks; (2) duplicated code — the workspace-root boundary check is copy-pasted across three (nearly four) file tools, six provider constructors are near-identical, and scanner-buffer sizing is repeated with three inconsistent magic values; and (3) a handful of long, deeply-nested functions (`runLoop`, `transport.pump`) that mix abstraction levels.

Nothing here blocks correctness or ships a bug. This is a healthy codebase with a short, well-bounded refactoring backlog.

## Detailed Findings

### Finding #1: Per-file linear `.gitignore` matching in directory walks
- **Category:** Complexity
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Repeated Linear Scan / Render-Path Recompute (hot-path variant)
- **Location:** `internal/agenttool/search_tool.go:203` (grep `WalkDir`), `search_tool.go:322` (find `WalkDir`) → `internal/runtime/config.go:107-138` (`gitignore.ignored`)
- **Principle Violated:** Avoid repeated work on a static input
- **Description:** Both the grep and find tools call `gi.ignored(rel, ...)` for **every** file and directory visited during `WalkDir`. Inside `ignored()` the code loops over all rules (`config.go:114`), and for each non-anchored rule additionally does `strings.Split(relPath, "/")` and a nested `filepath.Match` (`config.go:126`). Combined cost is `O(files × rules × pathSegments)`. On a large repo with a sizable `.gitignore`, this is the dominant cost of the agent's core search tools.
- **Evidence:** The rule set is static for the lifetime of a walk, yet is re-interpreted per file.
- **Recommendation:** Precompile rules once (separate anchored vs. non-anchored buckets; precompute split segments / compiled matchers). Consider short-circuiting directory-level ignores so entire subtrees are skipped without per-descendant checks. This is the single highest-value performance change.

### Finding #2: Duplicated workspace-root boundary check
- **Category:** Code (Duplication)
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Duplicated Code
- **Location:** `read_tool.go:74-99`, `write_tool.go:63-87`, `edit_tool.go:64-88`, partial `search_tool.go:46`
- **Principle Violated:** DRY
- **Description:** The `resolvePath` logic (resolve against Root, `filepath.Abs`, `filepath.Rel`, `..` boundary check) is copy-pasted across the file tools. This is **security-relevant** code (workspace escape prevention) — duplication means a fix or hardening must be applied in 3-4 places, and they can drift.
- **Evidence:** `read_tool.go:93-97` boundary check is byte-for-byte repeated in write/edit tools.
- **Recommendation:** Extract a shared `resolveWithinRoot(root, p string) (string, error)` helper (in `agenttool`, or a small internal path util). All tools call the one implementation; the security invariant lives in one place.

### Finding #3: Six near-identical provider constructors
- **Category:** Code (Duplication)
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Duplicated Code
- **Location:** `internal/provider/providers.go:346-447` (`NewOpenRouterProvider`, `NewOllamaProvider`, `NewNvidiaProvider`, `NewOpenAICompatibleProvider`, `NewAnthropicProvider`, `NewBedrockProvider`)
- **Principle Violated:** DRY
- **Description:** The OpenAI-compatible constructors differ only in `name`, default `baseURL`, `requiresAuth`, and `extraHeaders`. They repeat the same `if baseURL == "" { baseURL = ... }` + struct-literal shape.
- **Recommendation:** Introduce a small preset table (`{name, defaultURL, requiresAuth, extraHeaders}`) and a single `newOpenAICompat(preset, baseURL, models)` factory. Anthropic/Bedrock similarly share `anthropicCompatDriver`. This shrinks the surface and makes "add a gateway" a one-line table entry.

### Finding #4: Inconsistent scanner-buffer magic values
- **Category:** Code (Magic Numbers)
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Magic Numbers / Inconsistent Configuration
- **Location:** `repl.go:51` (4MB), `search_tool.go:237` (1MB), `read_tool.go:153`, `session.go:156`, `session.go:230` (16MB)
- **Description:** `bufio.Scanner` max-buffer sizes are set in 5 places with **three different** values and no shared named constant. A line longer than the smallest limit (1MB in grep) silently truncates or errors differently than elsewhere — inconsistent behavior for the same conceptual "max line" concern.
- **Recommendation:** Define named constants (e.g. `maxScanLine`) with a documented rationale per use, or a single shared default. At minimum, name each value so the intent (why 1MB for grep but 16MB for session replay) is explicit.

### Finding #5: Long, deeply-nested functions
- **Category:** Code (Long Method)
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Long Method / Deep Nesting
- **Location:** `internal/runtime/loop.go:93-204` `runLoop` (112 lines, **5 levels** of nesting at :143/147/162/166 — most severe), `internal/provider/transport.go:155-269` `pump` (115 lines), `streamAssistantResponse` (94), `search_tool.go:177-267` `GrepTool.Execute` (91), `run.go:94-183` `dispatch` (90), `runREPL` (83), `runInteractive` (80)
- **Principle Violated:** Single Level of Abstraction; readability
- **Description:** These functions mix control flow, error handling, and business logic across many nesting levels. `runLoop`'s 5-level nesting is the hardest to follow. Not a correctness problem, but a maintainability and testability cost.
- **Recommendation:** Extract inner blocks into named helpers at a single abstraction level (e.g. in `runLoop`, pull the tool-dispatch and turn-finalization branches into helpers; in `pump`, separate the frame-decode loop from the retry/stall logic). Prefer early-return guards over nested `if`.

### Finding #6: Scattered magic numbers (transport & file perms)
- **Category:** Code (Magic Numbers)
- **Severity:** 🔵 Suggestion
- **Location:** `transport.go:130` (529), `:142` (400), `:143` (4096), `:92` (maxRetries=2), `:162` (1.2 stall factor), `:335` (backoff base); `auth.go:133` (30s leeway); file perms `0o755`/`0o644` scattered
- **Description:** HTTP status codes, retry/backoff tuning, and token/permission literals appear inline.
- **Recommendation:** Promote to named constants (`httpStatusTooManyRequestsUpstream`, `maxRetries`, `stallFactor`, `dirPerm`/`filePerm`). Improves grep-ability and makes tuning intentional.

### Finding #7: Repeated tool-argument decode+error pattern
- **Category:** Code (Duplication)
- **Severity:** 🔵 Suggestion
- **Location:** ~8 sites across the tool `Execute` implementations
- **Description:** Each tool repeats the "unmarshal args into params struct, on error return tool-error result" boilerplate.
- **Recommendation:** A small generic helper `decodeArgs[T](raw) (T, *toolError)` (Go generics) removes the boilerplate and standardizes the error message shape.

### Finding #8: Thin direct unit tests on `internal/agentcore`
- **Category:** Testing
- **Severity:** 🔵 Suggestion
- **Anti-Pattern:** Weak coverage of core types
- **Location:** `internal/agentcore` — `hooks.go`, `event.go`, `message.go`, `content.go`, `helpers.go`, `tool.go` have no same-name `_test.go` (only `event_stream.go` and `types_test.go` are directly tested)
- **Description:** The core domain types (message/content/tool/hooks) are only covered indirectly through other packages. Every package has *some* tests and no package is a total blank — but the most-depended-on layer has the least direct testing.
- **Recommendation:** Add focused unit tests for the content/message constructors and helpers (`ContentToText`, message encoding round-trips). These are cheap, fast, and protect the layer everything else builds on.

### Finding #9: `session` package layering
- **Category:** Architecture (minor)
- **Severity:** 🔵 Suggestion
- **Location:** `internal/session` imported only by `cmd/pigo`
- **Description:** `runtime` orchestrates runs but session persistence lives outside it, wired only at the `cmd` layer. This is a legitimate design choice (keeps persistence a CLI concern), but worth a conscious note: if programmatic/embedded use ever needs persistence, it currently can't reach it without going through `cmd`.
- **Recommendation:** No action now. Revisit only if a non-CLI entry point needs session persistence.

## Refactoring Roadmap

### Immediate Actions (This Sprint)
1. Extract the shared `resolveWithinRoot` path-boundary helper (Finding #2) — security-relevant duplication, cheap to fix.
2. Precompile/bucket `.gitignore` rules to kill the per-file linear scan (Finding #1) — highest performance value.
3. Name the scanner-buffer sizes with shared constants and reconcile the 1MB/4MB/16MB inconsistency (Finding #4).

### Short-Term (1-3 Months)
1. Collapse the six provider constructors into a preset table + factory (Finding #3).
2. Decompose `runLoop` and `transport.pump` into single-abstraction-level helpers (Finding #5).
3. Add direct unit tests for `internal/agentcore` core types (Finding #8).

### Long-Term (3-12 Months)
1. Introduce a generic `decodeArgs[T]` tool-argument helper and sweep the ~8 call sites (Finding #7).
2. Promote scattered transport/permission magic numbers to named constants (Finding #6).
3. Reassess `session` layering only if a non-CLI entry point emerges (Finding #9).
